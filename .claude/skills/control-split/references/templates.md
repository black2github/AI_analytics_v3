# Шаблоны производных документов

**Имя файла, `title` и `doc_id` строятся из одной основы — не расходятся.** Основа =
исходный заголовок документа `[<префикс>] …` с вставленным маркером `BE` / `FE Контроли`
(правила имени файла — references/formatting.md). Отличаются только оформлением:

- имя файла — дефисы вместо пробелов, `.md`, без спецсимволов (URL-safe);
- `title` — та же строка с пробелами и обычной пунктуацией;
- `doc_id` — `{{<service>: <текст title без внешних кавычек>}}`.

**Не используй для `title` отдельный формат «Проверки сущности — …».** Имя файла,
`title`, `doc_id` и ссылки (`related`, смарт-ссылки) обязаны указывать на одну и ту же
сущность, иначе ломаются навигация и редактирование (имена файлов критичны: по ним идут
ссылки между документами).

Примеры:

- BE: файлу `[КК_ВК]-BE-Контроли-заявки-на-выпуск-карты-и-открытие-счета.md`
  соответствуют `title: '[КК_ВК] BE Контроли заявки на выпуск карты и открытие счета'`
  и `doc_id: '{{CC: [КК_ВК] BE Контроли заявки на выпуск карты и открытие счета}}'`.
- FE: файлу `[КК_ВК]-FE-Контроли-ЭФ-Клиента-страница-Анкеты.md`
  соответствуют `title: '[КК_ВК] FE Контроли ЭФ Клиента: страница "Анкеты"'`
  и `doc_id: '{{CC: [КК_ВК] FE Контроли ЭФ Клиента страница Анкеты}}'`.

Frontmatter наследует поля исходного документа (`service_code`, `source`,
`confluence_page_id`, `is_platform` и пр.); меняются `doc_id`, `title`,
`description`, `control_kind`, `related`, `tags`.

**Поле `related` — строка со списком смарт-ссылок через запятую** (в проекте это
comma-separated-поле, как `jira_ids`), а не YAML-массив. При типичной конфигурации
**один BE ↔ несколько FE**:

- в **BE** `related` перечисляет **все** парные FE-документы через `, `:
  `related: '{{<service>:[<префикс>] FE Контроли <страница 1>}}, {{<service>:[<префикс>] FE Контроли <страница 2>}}'`;
- в **каждом FE** `related` указывает на **один** BE-документ.

Каждая ссылка — `{{<service>:<текст doc_id>}}` соответствующего документа (та же
основа, что имя файла и `title`, см. выше). Связь двусторонняя: на что ссылается BE,
то должно ссылаться на BE в ответ.

## Документ экранной формы (screen-form)

```markdown
---
doc_id: '{{<service>: [<префикс>] FE Контроли <форма/страница>}}'
title: '[<префикс>] FE Контроли <форма: страница "...">'
description: 'Контроли, оперирующие полями экранной формы: обязательность полей вкладок, формат и подтверждение полей при вводе. В OpenAPI не проецируются.'
doc_type: requirement
requirement_type: control
control_kind: screen-form
service_code: <из исходника>
source: <из исходника>
confluence_page_id: '<из исходника>'
status: draft
version: 1.0.0
related: '{{<service>:[<префикс>] BE Контроли <исходный заголовок>}}'
tags: [control, screen-form, frontend, ux]
---

> **Зона 3.** Эти проверки оперируют **полями экранной формы** {{<service>:[<префикс>] ЭФ ...}}.
> Значения полей формы не обязаны совпадать с атрибутами сущности ни по составу,
> ни по реализации проверки. В OpenAPI не проецируются — гарантия целостности
> лежит на проверках сущности ({{<service>:[<префикс>] BE Контроли <исходный заголовок>}}).
> Часть полей проверяется и здесь, и в сущности — это осознанное дублирование.

## Назначение

Контроли полей формы заявки {{<service>:[<префикс>] <сущность>}}, выполняемые на
клиенте при работе с формой — {{<service>:[<префикс>] Функция создания заявки}} и
{{<service>:[<префикс>] Функция редактирования заявки}}.

## Группы триггеров

Группа — событие формы, при котором запускается подмножество проверок. «V» на
пересечении проверки и группы означает выполнение при наступлении события группы.

- **Группа №1** — <описание перехода/события>.
- **Группа №2** — <описание перехода/события>.

## Контроли заявки

`ID` — стабильный сквозной идентификатор; `Ориг.` — исходный № Confluence-страницы.

| ID | Ориг. | Поле / объект формы | Условие (кратко) | Сообщение об ошибке | Гр.1 | Гр.2 |
|----|-------|---------------------|------------------|---------------------|:----:|:----:|
| SF-01 | ... | ... | ... | ... | V | |
```

## Документ сущности (data-model)

```markdown
---
doc_id: '{{<service>: [<префикс>] BE Контроли <исходный заголовок>}}'
title: '[<префикс>] BE Контроли <исходный заголовок>'
description: 'Контроли, оперирующие атрибутами логической сущности: обязательность, формат, справочники, межатрибутные и межсистемные правила. Источник для проверок, разделяемых между событиями.'
doc_type: requirement
requirement_type: control
control_kind: data-model
service_code: <из исходника>
source: <из исходника>
confluence_page_id: '<из исходника>'
status: draft
version: 1.0.0
related: '{{<service>:[<префикс>] FE Контроли <страница 1>}}, {{<service>:[<префикс>] FE Контроли <страница 2>}}'
tags: [control, data-model, backend]
---

> **Зона 1 (публичный контракт).** Эти проверки оперируют **атрибутами сущности**
> {{<service>:[<префикс>] <сущность>}} и выполняются сервисом при изменении статуса
> документа в жизненном цикле. Парный документ
> {{<service>:[<префикс>] FE Контроли <форма/страница>}} описывает проверки
> полей формы; одно и то же значение может проверяться и там, и здесь — это
> осознанное дублирование (форма — ранняя реакция, сервис — гарантия).

## Назначение

Контроли заявки {{<service>:[<префикс>] <сущность>}}, оперирующие атрибутами и
выполняемые при изменении статуса документа. Применяются: <точки применения —
процессы и функции со смарт-ссылками>.

## Группы триггеров

Группа — статус сущности в жизненном цикле, при переводе в который запускается
подмножество проверок; операция (`operationId`) — только если у группы нет
своего персистентного статуса (номер группы — сквозной с парным FE-документом;
словесное описание — своё, в терминах BE, а не скопированное из FE-документа —
см. SKILL.md, «Правило раздельного описания групп»). Выбор статус/операция —
**один на весь документ**, не смешивай по группам. Формулировка — только сам
триггер, без механики вычисления и без обработки непройденных проверок (это в
процессе жизненного цикла сущности, сюда — самое большее одна ссылка на него).

- **Группа №1** — <целевой статус сущности, напр. DRAFT>.
- **Группа №2** — <целевой статус сущности, напр. NEW>.
- **Группа №3** — <целевой статус сущности, напр. SIGNED; операция `sign`, только если у этого перехода нет своего статуса>.
- **Группа №4** — <целевой статус сущности либо операция, если статуса нет, напр. приём данных из ЕСК после проверки полномочий>.

## Контроли заявки

`ID` — стабильный сквозной идентификатор; `Ориг.` — исходный № Confluence-страницы;
`Тип` — `field` (автономное ограничение значения атрибута) или `rule` (нужен контекст:
межатрибутное / справочник / другая сущность / внешняя система / полномочия). Критерий
и пограничный случай «соответствие справочнику — это `rule`» — см. formatting.md.

| ID | Ориг. | Тип | Атрибут / объект | Условие (кратко) | Сообщение об ошибке | Гр.1 | Гр.2 | Гр.3 | Гр.4 |
|----|-------|-----|------------------|------------------|---------------------|:----:|:----:|:----:|:----:|
| DM-01 | ... | rule | ... | ... | ... | V | | V | |

## Связь с OpenAPI

Операция статусного перехода ссылается на этот документ через `x-controls` с
указанием группы — без перечисления каждого `422`. Стабильна структура
`ControlViolation` (`code` + `message`), не наполнение.

`source` указывает на **этот** BE-документ по его `doc_id`. Если над агрегатом
несколько операций-триггеров — у каждой свой блок с тем же `source` и своей `group`
(см. references/multi-entity.md).

```yaml
post:
  operationId: <реальный operationId из OpenAPI-спецификации проекта>
  summary: <напр. Подписать заявку>
  x-controls:
    source: "{{<service>:[<префикс>] BE Контроли <исходный заголовок>}}"
    group: <N>
  responses:
    '422':
      description: Проверки сущности (событие «<имя>») не пройдены
      content:
        application/json:
          schema: { $ref: '#/components/schemas/ControlViolation' }
```
```

## Заметки по оформлению

- Заголовки разделов — без уточнений в скобках («Группы триггеров», не «Группы
  триггеров (события формы)»). Документ — рабочий, в том виде, в каком станет
  шаблоном.
- Условие в таблице — **дословно**, без перефразирования и обрезки; сохраняй
  пошаговую структуру условия (`<br>`/маркеры), не склеивай в одну строку.
- Неактуальные контроли помечай маркером ⚠️ в колонке объекта и оставляй в
  таблице (не удаляй).
