# Чеклист финальной сверки

Пройди перед выдачей результата.

## Полнота

- [ ] Каждый исходный № встречается хотя бы в одном производном документе
      (в колонке «Ориг.»). Ни одна строка не потеряна.
- [ ] **BE содержит непустой раздел «Связь с OpenAPI»** (≥1 операция `x-controls`
      на каждый триггер-группу, все на ОДИН `source`). Без него BE невалиден.
- [ ] Дублирование на форму — по критерию объёма, не механически: в `SF-NN`
      вынесено лишь то, что форма проверяет на поле (обязательность/формат/значения/
      видимость) + ранний отказ по внешним сущностям и входным гейтам (статус
      организации, полномочия, блокировка клиента); чистые межатрибутные/статусные/
      межсистемные правила без полевой реакции и без раннего read-доступа остались
      DM-only (с пометкой «на бэке»). Решение о мирроринге — по двум осям (цена
      позднего отказа × ранний read-доступ), не по категории. Принятая политика
      объёма едина по всему разбору.
- [ ] Неактуальные контроли перенесены и помечены ⚠️, а не удалены.
- [ ] Контроли вне групповой матрицы перенесены и оговорены.
- [ ] Группы, делегированные ссылкой в другой документ: присутствующие в таблице
      строки перенесены (не потеряны), отсутствующие — оставлены указателем;
      пересечение с внешним документом помечено для аналитика.

## Идентификаторы

- [ ] ID сквозные и последовательные внутри каждого документа (`SF-01…`,
      `DM-01…`).
- [ ] В каждой строке заполнена колонка «Ориг.» с исходным №.
- [ ] Дубли и опечатки исходной нумерации сохранены в «Ориг.» как есть, но
      новые ID корректны.

## Структура документов

- [ ] Оба документа следуют одному формату: frontmatter → врезка-зона →
      «Назначение» → «Группы триггеров» (каждая группа отдельным пунктом) →
      «Контроли заявки» → (для сущности) «Связь с OpenAPI».
- [ ] Заголовки без уточнений в скобках.
- [ ] frontmatter наследует поля исходника; `control_kind` проставлен верно
      (`screen-form` / `data-model`).
- [ ] `related` — строка со смарт-ссылками через запятую (не YAML-массив); при
      1↔N в BE перечислены **все** парные FE, в каждом FE — один BE; связь
      двусторонняя.
- [ ] Группы триггеров описаны словами (акцент на событие-триггер, не на таблицу);
      **ссылки исходника** (страницы/процессы/функции) в описаниях сохранены, не
      сведены к голому тексту.
- [ ] **Описание групп в data-model — на языке BE, не скопировано из FE.** В
      «Группах триггеров» BE-документа нет кнопки/вкладки/перехода как части
      определения группы (только целевой статус сущности и/или API-операция);
      если экранное событие упомянуто — это необязательное примечание со ссылкой
      на парный FE-документ, а не формулировка группы. См. SKILL.md, «Правило
      раздельного описания групп».
- [ ] **Все группы одного data-model документа названы единообразно** —
      либо все через целевой статус сущности, либо (только при отсутствии
      персистентного статуса у конкретной группы) через операцию; нет
      документа, где часть групп — «статус X», а часть — «операция Y» без
      причины (см. «Статус — приоритетнее операции» в SKILL.md). Каждая
      формулировка группы — **одна фраза-триггер**, без пересказа механики
      вычисления итога по нескольким группам и без обработки непройденных
      проверок (это — предмет процесса жизненного цикла, не определения группы).

## Содержание

- [ ] Условия перенесены **дословно** (без перефразирования и усечения), с
      сохранённой пошаговой структурой («если-то-иначе» читаемы); никаких «…».
- [ ] Парные «обязательность + пустота» обработаны согласованно с выбранной
      политикой (раздельно в сущности; в форме — по договорённости).
- [ ] Согласия отнесены по принятому решению (по умолчанию data-model); выбор
      оговорён, если спорный.
- [ ] В документе сущности у каждой строки проставлен `Тип` (`field`/`rule`) по
      критерию «достаточно ли одного значения поля»: справочник/межатрибут/другая
      сущность/внешняя система/полномочия → `rule`; автономное ограничение значения →
      `field` (см. formatting.md).
- [ ] Блок `x-controls` корректен: `source` — смарт-ссылка на сам документ; `group` —
      номер события; `operationId`/путь — **реальные операции из OpenAPI-спецификации
      проекта** (не выдуманные; при отсутствии — `TODO`, не фикция).
- [ ] Колонка «Проверяемый атрибут» заполнена во **всех** строках (имя не
      потеряно при снятии тегов); скобки в колонке убраны, в тексте условия у
      второго атрибута сохранены.
- [ ] Имена атрибутов дочерних сущностей — без префикса родителя; родительский
      атрибут (если нужен) — как `<родитель>.<атрибут>`.
- [ ] Многосущностный data-model файл: сквозные ID, **единые** номера групп,
      разделители-строки сущностей через `colspan`.
- [ ] Один screen-form файл на каждую экранную форму/страницу; терминальная
      страница без группы-перехода тоже оформлена отдельным FE-файлом (триггер
      описан кнопкой-действием, а не переходом).
- [ ] Дробление FE — по страницам, не по группам: несколько групп **одной** страницы
      сведены в **один** файл несколькими столбцами групп (не раздроблены на файл-на-
      группу).
- [ ] Имена файлов построены из **оригинального** имени (исходник для BE, файл
      ЭФ для FE) минимальной правкой, без пробелов, с `.md` (годны для URL): BE —
      `BE-` перед «Контроли»; FE — `FE-Контроли` после префикса.
- [ ] `title` и `doc_id` согласованы с именем файла (та же основа `BE`/`FE Контроли`,
      отличие только в оформлении); формат «Проверки сущности — …» не использован;
      `related` и смарт-ссылки указывают на парный документ его `doc_id`.
- [ ] Выходные файлы — в том же каталоге, что и обрабатываемый оригинал.
- [ ] В FE-документе «Назначение»/врезка ссылаются на экранную форму (имя из
      описания триггера), а не на функции создания/редактирования.
- [ ] Многосущностный BE-файл: под каждую сущность отдельный подраздел `###`,
      нумерация и группы сквозные.
- [ ] Внешние сущности и входные гейты (статус организации, полномочия, блокировка
      клиента) продублированы в оба документа как ранний отказ; binding-решение на BE;
      проставленные по смыслу бэковые группы помечены для аналитика.
- [ ] Сущность каждого контроля определена по префиксу атрибута, а не по
      разделителю исходника; смешанный блок «ЭФ Клиента» в data-model отсутствует.
- [ ] Агрегаты формы («Обязательные поля вкладки», «Наличие хотя бы…») — только в
      screen-form, не в data-model.
- [ ] Ссылки (`[текст](href)`) сохранены в атрибутах, логике и сообщениях; теги
      `<strong>` и пр. вычищены.
- [ ] Текст не обрезан: условия, сообщения об ошибке и названия проверок перенесены
      целиком, без усечения по длине и без «…» (нет обрывов на середине слова).
- [ ] Импорт реестра: значения растворены по объекту, импорт — отдельный
      столбец-группа; проверки формата файла — в бэк с пометкой; отдельного файла
      под импорт нет.

## Сообщить пользователю

- [ ] Перечислить принятые спорные решения (согласия, схлопывание парных строк,
      классификация неоднозначных контролей) — это предмет согласования с командой.
- [ ] Указать обнаруженные дефекты исходника (дубли/опечатки номеров) — полезно
      для очистки первоисточника.
