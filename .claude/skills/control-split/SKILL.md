# Разделение документа контролей (API First / Doc as Code)

## Что делает скилл

На входе — один экспортированный документ контролей заявки (единая таблица:
группы триггеров + матрица «контроль × группа» со значками «V»). На выходе —
набор markdown-документов:

1. **Проверки сущностей модели данных** (`control_kind: data-model`, Зона 1) —
   контроли, оперирующие **атрибутами сущностей**. На них ссылается операция
   OpenAPI через `x-controls`. Один файл может покрывать **несколько связанных
   сущностей** (иерархия) со сквозными номерами групп — см.
   references/multi-entity.md.
2. **Проверки экранных форм** (`control_kind: screen-form`, Зона 3) — контроли,
   оперирующие **полями форм**. В OpenAPI не проецируются. По умолчанию
   **отдельный файл на каждую экранную форму/страницу** — см.
   references/multi-entity.md.

Ось разделения — **объект проверки**: поле экранной формы или атрибут логической
сущности. Это, а не технология «фронт/бэк», определяет, в какой документ попадает
контроль. Обоснование и связь с зонами — в стратегии перехода на API First
(раздел «Контроли»).

## Когда применять

Применяй, когда пользователь предоставляет документ контролей и хочет его
разделить, разметить для API First, присвоить ID или мигрировать в Doc as Code.
Если документа контролей нет в контексте, но он загружен — сначала прочитай его
(см. правила чтения файлов в окружении).

## Процедура

### Шаг 1. Разобрать исходный документ

Прочитай исходный документ и извлеки:

- **frontmatter** (`doc_id`, `service_code`, `source`, `confluence_page_id` и пр.)
  — он понадобится для производных документов;
- **историю изменений** — сохраняется только в одном из производных (см. шаг 5),
  либо опускается, если пользователь не просит её переносить;
- **описание групп триггеров** — текстовый список «Группа №N — событие», **со
  ссылками исходника** (на страницы ЭФ, процессы, функции); не перефразируй их в
  голый текст — ссылки несут навигацию и смысл (например, ссылка на процесс приёма
  заявки объясняет, что ФЛК выполняется при подписи);
- **таблицу контролей** — каждая строка: №, проверяемый реквизит, название
  проверки, логика, сообщение об ошибке, отметки «V» по группам.

Исходная таблица часто приходит как HTML-таблица внутри markdown (экспорт
Confluence). Разбирай её по строкам `<tr>`; учитывай `rowspan` (один реквизит на
несколько строк-проверок), `colspan` (имя сущности на всю ширину строки) и сбои
нумерации (см. references/edge-cases.md).

**Критично — имя проверяемого атрибута.** Имя атрибута часто записано в колонке
реквизита в нотации треугольных скобок `<Имя атрибута>` с `rowspan` на группу
проверок. При наивном снятии HTML-тегов треугольные скобки исчезают вместе с
именем — это типичная и опасная потеря данных. Извлекай имя атрибута **до**
очистки тегов (защити `<Кириллица/латиница…>` от вырезания), протягивай его по
`rowspan` на все строки группы. Колонка «Проверяемый атрибут» не должна быть
пустой ни в одной строке. Парсинг — самая хрупкая часть процесса; после разбора
сверь несколько строк с исходником глазами.

### Шаг 2. Классифицировать каждый контроль

Для каждой строки определи объект проверки по правилам из
**references/classification.md**. Кратко:

- **screen-form** — проверка оперирует полями формы: «обязательные поля вкладки»,
  отображение/подтверждение поля, формат вводимого поля, переходы между вкладками
  (группы-события формы).
- **data-model** — проверка оперирует атрибутами сущности: справочники, другие
  сущности, внешние системы, межатрибутные и статусные правила, полномочия.

Один и тот же реквизит часто порождает **две** проверки — одну на форме (поле),
другую в сущности (атрибут). Это нормально: они идут в разные документы как
самостоятельные. Но дублируй на форму **не механически**: в screen-form попадает лишь
то, что форма реально проверяет на поле (обязательность, формат, набор значений,
видимость) плюс ранний отказ по внешним сущностям и **входным гейтам** (статус
организации, полномочия, блокировка клиента); чистые межатрибутные/статусные/
межсистемные правила без полевой реакции остаются **только** в data-model, даже если
помечены форменной группой. Решение о мирроринге на FE — по двум осям (цена позднего
отказа × ранний read-доступ), не по категории контроля. Критерий объёма и пограничные
случаи — references/classification.md.

**Неоднозначные триггеры.** По описанию группы не всегда ясно, событие это формы
или жизненного цикла сущности. Кнопка («Подписать и отправить», «Продолжить») —
действие на форме, но переход чаще всего сопровождается созданием/сохранением
сущности; тогда проверка идёт в **оба** документа. Признак сохранения сущности —
наличие отдельной процедуры/метода/шагов инициализации атрибутов, используемых на
форме. Для разрешения неоднозначности привлекай **опорные документы**: описания
процессов (создание/редактирование/подписание), описания экранных форм,
диаграммы состояний сущности (статусы DRAFT/NEW/подписан…), описания иерархии
сущностей. Полный разбор — references/classification.md.

### Шаг 3. Присвоить стабильные ID

- Контроли формы: `SF-01`, `SF-02`, … (сквозная нумерация по документу формы).
- Контроли сущности: `DM-01`, `DM-02`, … (сквозная нумерация по документу
  сущности).

ID **стабильны**: не сдвигаются при добавлении/удалении контролей. В каждой строке
сохраняй колонку **Ориг.** с исходным № Confluence-страницы — для прослеживаемости
миграции и сверки.

### Шаг 4. Собрать документы по шаблону

Используй шаблоны из **references/templates.md**. Каждый документ содержит:
frontmatter → врезку о зоне → раздел «Назначение» → раздел «Группы триггеров»
(каждая группа отдельным пунктом — внимание на триггеры, а не на таблицу) →
раздел «Контроли заявки» (таблица) → для документа сущности — **обязательный**
раздел «Связь с OpenAPI» с `x-controls` (≥1 операция на каждый триггер-группу). Без
этого раздела data-model документ невалиден — это его функция для API First, а не
оформление (легко пропустить при скриптовой генерации; проверяется на Шаге 5). Формат
всех документов держи максимально одинаковым.

**Форматирование таблицы (см. references/formatting.md):**
- Имя атрибута в колонке «Проверяемый атрибут» — **без** треугольных скобок
  (там и так ясно, что это проверяемый атрибут). Внутри текста условия скобки у
  *второго* атрибута **сохраняй** — это разрешение двусмысленности, не шум.
- Имя атрибута дочерней/связанной сущности — **без** префикса родителя. Если в
  проверке нужен атрибут родителя, пиши его как `<родительская_сущность>.<атрибут>`.
- Одинаковые имена атрибутов в соседних строках **объединяй** через `rowspan`.
- В многосущностном файле имя сущности — строкой на всю ширину (через `colspan`).
  Если файл описывает **одну** сущность, имя сущности в таблице **не используется**
  (оно уже в названии файла).

**Сколько файлов (см. references/multi-entity.md):**
- Несколько связанных сущностей → **один** data-model файл, сквозные ID, **единые
  номера групп** по всему файлу.
- Несколько экранных форм/страниц → по умолчанию **отдельный** screen-form файл на
  каждую форму. Но дроби **по страницам, а не по группам**: несколько групп-триггеров
  одной страницы идут в **один** файл несколькими столбцами групп, не в файл-на-группу.
- Терминальная страница без группы-перехода (её событие — «Подписать и отправить»,
  а не «Продолжить») → тоже **отдельный** FE-файл; триггер описывается действием, не
  переходом. См. references/multi-entity.md.

### Шаг 5. Финальная сверка

Пройди по **references/checklist.md**. Ключевое: ни одна строка исходника не
потеряна (каждый исходный № встречается хотя бы в одном производном документе);
неактуальные контроли помечены, а не удалены; парные «обязательность + пустота
при условии» обработаны согласованно.

## Важные принципы

- **Не менять формат требований без необходимости.** У документов требований есть
  действующие шаблоны; производные документы должны быть узнаваемы аналитиками.
  Сохраняй привычную структуру (история изменений, «Назначение», «Группы»,
  таблица), добавляя только то, что нужно для API First (ID, колонка «Ориг.»,
  `control_kind`, `x-controls`).
- **Дублирование осознанное, не цитирование.** Проверка поля на форме и проверка
  атрибута на бэке — две самостоятельные записи в двух документах. Не выноси одну
  как ссылку на другую: их состав и реализация могут расходиться.
- **Тяжесть проверки не влияет на её документ.** Простая проверка формата и
  тяжёлая проверка через внешнюю систему классифицируются одинаково — по объекту
  (поле/атрибут), а не по сложности реализации.
- **Условие переноси дословно — не перефразируй и не обрезай.** Текст условия
  критичен к смыслу: в нём вложенные «если-то-иначе» и завязка на форматирование, а
  после снятия HTML вложенность хрупка — перефразирование (тем более слабой моделью)
  вносит двусмысленность и искажает требование. Поэтому **условие, название проверки
  и сообщение об ошибке переноси целиком**, без усечения по длине (`[:N]`) и без «…».
  Допустима только техническая нормализация: `<a href>` → markdown-ссылка,
  восстановление `<…>`, схлопывание лишних пробелов. **Сохраняй пошаговую структуру**
  условия (вложенные «если-то-иначе» через `<br>`/маркеры списка), не склеивая в одну
  строку: для контролей читаемость условия повышает шанс поймать ошибку на ревью —
  высота ячейки вторична. Если генерируешь таблицу скриптом, не оставляй в нём срезов
  `[:60]`/`[:90]`/`[:160]`. Сжатие условия «до сути» — отдельный ручной шаг аналитика
  с ревью, не часть автоматического разделения.
- **Правило внешних сущностей и входных гейтов.** Проверка против сущности,
  выходящей за рамки создаваемой/изменяемой (Организация, Клиент Банка, Соглашение,
  справочники), и входной гейт (статус организации, полномочия пользователя,
  блокировка клиента) идут в **оба** документа: на форме — ранний отказ пользователю
  (binding-решение всё равно на BE), в сущности — проверка атрибута-ссылки. «Источник
  истины — BE» значит «авторитетное решение на бэке», а не «только на бэке»: FE-копия
  оптимизирует клиентский путь, но устаревает/обходится/частична. Зеркалить на FE решай
  по двум осям — цена позднего отказа и наличие раннего read-доступа, — а не по
  категории контроля. Подробно — references/classification.md.
- **Правило агрегата (API First).** Несколько связанных сущностей сводятся в один
  data-model файл, когда они образуют **агрегат записи** (вложены в DTO корня, нет
  собственного контракта записи). Отдельные эндпоинты над таким агрегатом — это
  разные триггеры, отбирающие проверки по группе, а не повод дробить документ:
  в «Связь с OpenAPI» все операции ссылаются на ОДИН `source`, различаясь только
  `group`. Дробить — лишь если у под-сущности есть самостоятельный `create`/`update`
  со своим телом. Подробно — references/multi-entity.md.
- **Правило делегированных групп.** Если исходник отправляет группу в другой
  документ ссылкой («Группа N — см. …»), смотри на **факт наличия строк**, а не на
  ссылку: присутствующие в таблице строки переноси в агрегатный BE по сущности (не
  теряй, помечай пересечение с внешним документом); если строк нет — оставь только
  указатель. Полнота важнее «чистоты» делегирования. Подробно — references/edge-cases.md.
- **Правило импорта реестра.** Импорт — ускорение ввода, не отдельный канал
  создания сущностей: данные ложатся на экран, правятся и идут стандартным путём.
  Проверки значений растворяются в форме или сущности по объекту. Но импорт —
  самостоятельный триггер (отдельная группа-столбец), а проверки **формата самого
  файла реестра** (маппинг колонок, расположение полей) — это проверка структуры
  входного файла: их относи в **бэковый** файл с пометкой (или в описание функции
  импорта, если их немного). Отдельный файл под импорт не создавай.

## Справочные файлы

- `references/classification.md` — правила отнесения контроля к screen-form или
  data-model; неоднозначные триггеры и опорные документы; внешние сущности;
  пограничные случаи (согласия, агрегаты вкладок, парные проверки).
- `references/multi-entity.md` — несколько сущностей в одном data-model файле
  (сквозные группы); критерий агрегата и x-controls при нескольких операциях
  (API First); несколько экранных форм (отдельные файлы); правило импорта реестра.
- `references/formatting.md` — оформление таблицы: треугольные скобки, имена
  атрибутов дочерних сущностей, объединение ячеек (`rowspan`/`colspan`).
- `references/templates.md` — шаблоны frontmatter и разделов для документов.
- `references/edge-cases.md` — извлечение имён атрибутов из `<…>`, сбои нумерации,
  rowspan, неактуальные контроли, контроли вне групповой матрицы.
- `references/checklist.md` — финальная сверка перед выдачей результата.
