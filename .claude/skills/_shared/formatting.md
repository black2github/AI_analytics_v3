# Форматирование текста (ячейки таблиц и условия)

Цель — **сохранять смысловое разделение**, а не сливать текст в один поток. Правила
применяются одинаково в таблице полей формы и в таблице контролей.

## Переносы строк в ячейках

Внутри ячейки markdown-таблицы пустая строка невозможна — перенос делается тегом
`<br>`. Визуальный «отдельный блок» (пустая строка сверху и снизу) эмулируется
двойным переносом `<br><br>`.

Вне таблиц (обычные разделы) — обычные переводы строк и пустые строки, не `<br>`.

## Обязательные выделения

1. **Ключевые слова Если-то-иначе.** «Если», «то», «иначе» (в любом регистре и
   форме: «Если», «Иначе», «иначе если», «то») — **всегда жирным**: `**Если**`,
   `**то**`, `**иначе**`.

2. **Именованные блоки-ярлыки.** Служебный ярлык, вводящий смысловой блок,
   оформляется отдельным блоком: перенос перед, **жирный** ярлык, перенос после.
   Обязательно для:
   - `**По умолчанию:**`
   - `**Источник списка:**`
   - по аналогии, если встречаются: `**Формат отображения:**`, `**Плейсхолдер:**`,
     `**Подсказка:**`, `**Значение:**`, `**Пример:**`.
   В ячейке: `… <br><br>**По умолчанию:**<br>текст`.

3. **Последовательные шаги.** «Шаг 1.», «Шаг №2», … — перед каждым перенос строки,
   само слово **жирным**: `…<br>**Шаг 1.** …<br>**Шаг 2.** …`.

4. **Перечисления.** Каждый самостоятельный пункт — с новой строки (`<br>` или
   список `- `), не сливать подряд.

## Присваивание `=` и сравнение `==`

В исходнике знак `=` двусмыслен — им пишут и присваивание, и сравнение. В целевом
тексте это **разные операции**, различай их нотацией:

- **присваивание / инициализация значения** — один знак `=`. Колонка «Логика
  установки значения», умолчания, вычисляемые заголовки: `= [Заявка].<Номер заявки>`;
  `**По умолчанию:** = "RUB"`; `= "Заявка" + " №" + <Номер>`.
- **логическое сравнение** — двойной `==`. Внутри условий Если-то-иначе (и в колонке
  «Условие» контролей): `**Если** <Код типа счета> == 2, **то** …`.

Не оставляй двусмысленный исходный `=` в условиях — заменяй на `==`. (Здесь —
каноничная формулировка нотации; скиллы-потребители повторяют её у себя как
принцип и как проверку чек-листа, ссылаясь сюда.)

## Стрелка `→` — только «найти/взять запись»

Символ `→` (синоним `->`) зарезервирован **исключительно** за операцией
«найти/взять запись» по ключу: `операнд_A → Операнд_B` — найти запись сущности
`Операнд_B` по ключу `операнд_A`. Определение, приоритеты и примеры — в
авторитетном своде «Правила оформления аналитики ДБО», §2 (не дублируются здесь).
Резолверы и линтеры трактуют `→` как обращение к **данным** (там же, §6), поэтому
любое иное употребление — ложный сигнал для машинной обработки.

**Запрещено** использовать `→` для: последовательности шагов/проверок,
причины-следствия («событие → реакция»), иерархии, перехода между экранами/меню,
соответствия/маппинга «значение поля → параметр запроса». Замены:

- последовательность — список, «затем», `**Шаг N)**`;
- причина-следствие — словами («при …», «вызывает …»);
- переход экрана/меню — стрелка `=>` (по §2.1 свода);
- маппинг/соответствие — присваивание `=` (`параметр = «Офис обслуживания»`) или
  словами («передаётся в …»).

Как и `=`/`==`, это **оформительская нормализация поверх дословного переноса**:
исходный текст со стрелкой-связкой переписывается в допустимую форму, канонический
поиск записи (`<GUID> → [Клиент Банка].<Идентификатор записи>`) — сохраняется.

## Пример ячейки «Логика установки значения»

```
**Источник списка:**<br>[ЭКО_Получение списка счетов](...) по Организации и Типу карты.<br><br>**По умолчанию:**<br>**Если** в списке один счёт, **то** поле заполняется этим счётом и блокируется.
```

Читается как три отдельных блока (источник · умолчание · условие), а не сплошной
строкой.

## Сохранение форматирования при переписывании (v3.0)

Реструктуризация — это **семантическое дополнение, а не переписывание с потерей**.
Когда содержимое исходной ячейки переносится в целевую колонку, **сохраняются все
несущие смысл элементы**. Терять/игнорировать их запрещено.

- **Иконки** `<img src="..." width="...">` — переносятся **как есть** в ту колонку,
  где были (часть UI-спецификации). Пример: `Кодовое слово <img src="img/d42064ab.png" width="20">`.
- **Ссылки** `<a href="URL">текст</a>` заменяются на markdown `[текст](URL)`
  (навигация и прослеживаемость). Не превращать в голый текст.
- **Списки** `<ul>/<ol>/<li>` заменяются на markdown-пункты, разделённые `<br>`
  (`<br>- пункт1<br>- пункт2`). Не склеивать перечисление в строку через запятые.
- **Выделения** `<strong>` заменяется на `**…**`, `<em>` — на `*…*`.
- **Вложенность условий** — сохраняется, не сплющивается. Уровень показывается
  отступом из `&nbsp;` (4 на уровень) после `<br>`. Пример:
  `**Если** поле заполнено, **то**:<br>&nbsp;&nbsp;&nbsp;&nbsp;**Если** тип = 2, **то** отображается<br>&nbsp;&nbsp;&nbsp;&nbsp;**иначе** не отображается<br>**иначе** скрыто`.
- **Токены `<Имя атрибута>`** — сохраняются дословно (см. parsing.md рядом:
  защита треугольных скобок). Не терять как «неизвестный тег».

## Пометка при неуверенности (приоритет — смысл)

Сохранение смысла — высший приоритет. Если фрагмент описания:
- не удаётся **передать без потери смысла** (переформулировать в полный эквивалент),
- или разложить по колонкам однозначно,
- или ты **не уверен в эквивалентности** результата,

— **не сглаживай и не выбрасывай**, оставь в подходящей ячейке маркером с
**дословным** исходным текстом:

```
[НЕ РАСПОЗНАНО: <исходный текст фрагмента дословно>]
```

Тихая потеря/огрубление смысла — худший исход. Явная пометка — приемлемо: человек
или другой агент доразберёт место, ничего не потеряв.

## Что НЕ дублируется

Вынесенное в контроль условие (`rule`/`visibility`) в ячейке формы **не
повторяется** — только указатель `см. SF-NN`. Единственный источник истины для
вынесенного условия — FE-документ контролей.
