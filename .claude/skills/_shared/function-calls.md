# Вызовы функций/методов — с инициализацией входных параметров

Общее ядро скиллов описания ЭФ (screen-form-restructure,
event-driven-screen-form-design). Каноническая формулировка принципа;
скиллы-потребители ссылаются сюда, не дублируют текст.

Любой упомянутый на ЭФ вызов функции/метода (кнопки, «Дополнительные действия
при открытии ЭФ», кнопки сообщений, автозаполнение поля из вызова) описывается
**вместе с тем, как инициализируются его входные параметры**, — иначе
требование нереализуемо. Правила:

- **Не терять то, что есть в источнике:** если исходник указал входные (и
  значимые исходящие) параметры и способ их получения (значение поля / атрибут
  сущности / предварительный шаг вычисления или запроса) — переноси полностью,
  дословно.
- **Дополнять, если источник опустил** (частая ошибка описаний — вызов дан
  «голым»): восстанови **состав** входных параметров по **логическому описанию
  функции** (приоритет) и **OpenAPI-спецификации** метода сервиса; для каждого
  параметра укажи **источник значения на этой форме** — поле формы (нотация
  `<Метод>.<ПутьКПолю>` / `= <Поле>`), атрибут сущности, либо **шаг получения
  значения** (если параметр добывается предварительным вызовом/запросом, а не
  берётся напрямую).
- **Приоритет по обязательности:** логическое описание функции **>** OpenAPI
  (в OpenAPI обязательность параметров проставлена не всегда).
- **Различай два вопроса:** (1) *состав и обязательность* параметров — из
  функции/OpenAPI (почти всегда восстановимо); (2) *источник значения* каждого
  параметра на форме — из описания ЭФ; если его нет и из семантики поля не
  выводится — помечай `[НЕ РАСПОЗНАНО: …]` / `уточняется`, а не опускай молча.
- **Граница:** скилл формы фиксирует наполнение параметров вызова **из ЭФ**
  (чем наполняются входы). Маппинг параметра вызова **в атрибут сущности** при
  сохранении (из DTO в сущность) — API-слой, вне объёма скиллов форм.
