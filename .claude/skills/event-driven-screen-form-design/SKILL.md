# Событийное описание экранной формы (Е-N / П-N / М-N) — v1.0

> **Верховный принцип (reverse): СОХРАНЕНИЕ СМЫСЛА — высший приоритет.**
> Переформатирование — семантическое преобразование через понимание: каждый
> фрагмент исходника должен появиться в целевой структуре как полный смысловой
> эквивалент (условия, ограничения, умолчания). Переформулировать можно; терять
> и огрублять — нельзя. Не уверен в эквивалентности — пометь
> `[НЕ РАСПОЗНАНО: <дословный исходный текст>]`, не сглаживай.
>
> **Верховный принцип (forward): ЗАЗЕМЛЕНИЕ — каждый элемент имеет
> основание-источник.** Шаг события — ссылку на требование (PRC/FUN/CTL/ENT)
> либо пометку `уточняется` с формулировкой открытого вопроса; вызов внешнего
> метода — карточку EXTINT (нет карточки — создаётся заглушка; выдуманный метод
> запрещён); сообщение — заполненный «Показ». Элемент без основания и без
> пометки — дефект генерации. `[НЕ РАСПОЗНАНО]` в forward не используется —
> его место занимают `уточняется` и открытые вопросы (по конвенциям репозитория
> документации — вне docs/, не inline-TODO).

## Что делает скилл

Выход — **один** документ ЭФ в событийной структуре (см. references/templates.md):

1. **Описание ЭФ** — дословная таблица (зачем / как перейти / доп. действия /
   макеты); строка «Дополнительные действия при открытии» ссылается на Е-1.
2. **Назначение** — цель формы, ссылки на сущность (ENT), экран, функции
   открытия (FUN).
3. **Реестр полей** — только `№ | Блок | Поле | Тип | Обяз.`.
4. **Видимость блоков** — условия по блокам списком.
5. **Правила полей (статика)** — подраздел **П-N** на поле, ярлыки-буллеты.
6. **События** — реестр **Е-N** + детальные подразделы; шаги — со стабильными
   ID `Е-N.M` (идентификатор, не порядковый номер: порядок задаёт позиция в
   документе, ID не перенумеровываются); кнопки — тоже события.
7. **Сообщения** — реестр **М-N** с обратной ссылкой «Показ».

Режимы:

- **reverse** — вход: страница ЭФ (экспорт Confluence или markdown). Дословный
  перенос смысла в событийную структуру; технический разбор HTML — по
  `../screen-form-restructure/references/parsing.md`.
- **forward** — вход: процесс (PRC), модель данных (ENT/EXT), функции (FUN),
  контроли (CTL), интеграции (EXTINT), соседние формы сервиса как образцы.
  Агент проектирует, человек ревьюит; новый путь в существующем сервисе —
  **дельта**: новые события + ссылки на существующие П-N/исходы/EXTINT/CTL.

## Ключевые принципы

- **Нотация — по действующему formatting.md** скилла screen-form-restructure
  (`../screen-form-restructure/references/formatting.md`), это общий авторитет:
  жирные **Если**/**то**/**иначе**; `=` — присваивание, `==` — сравнение, `≠`;
  шаги с новой строки жирным; `→` — только «найти/взять запись» (свод «Правила
  оформления аналитики ДБО» §2); ярлыки-блоки жирным. **Эталон-пример формата
  содержит нотационные ошибки (одиночные `=` в сравнениях и др.) — из эталона
  берётся структура, нотация — из formatting.md; ошибки не воспроизводить.**
- **Вызовы функций/методов — с инициализацией входных параметров** (авторитет —
  SKILL.md screen-form-restructure, одноимённый принцип): параметры источника не
  терять; отсутствующие восстанавливать по логическому описанию функции
  (приоритет) и OpenAPI; источник значения каждого параметра указывать
  (поле / атрибут / шаг получения) либо `уточняется`. Вызовов «голыми» нет.
- **Видимость — всегда инлайн** (раздел «Видимость блоков» + ярлык «Видимость»
  в П-N). Модель выноса видимости в контроли (SF-NN, R/V) в событийном формате
  **не используется**.
- **Контроли скилл не создаёт** — их создаёт control-split; здесь только ссылки
  на CTL-NN / CTL-GRP комплекта контролей сервиса (обязательность полей реестра —
  ссылкой на контроль группы, с оговоркой «проверяются только отображаемые поля»).
- **Ссылки на артефакты SRS-комплекта** (ENT/EXT/EXTINT/FUN/PRC/CTL) вместо
  сырых Confluence-ссылок. Нет карточки — заглушка с пометкой («карточка-заглушка,
  страница метода вне выгрузки»), не выдуманный артефакт и не голый текст.
- **Реакции на ошибки не описываются в шагах сверх показа сообщения** — текст и
  кнопки сообщения живут в М-N, шаг только ссылается («информер М-6, исход
  "Скрытие полей"»).

## Процедура

### Шаг 1. Собрать вход

- reverse: разобрать исходную страницу (frontmatter, история изменений, поля,
  действия, сообщения, встроенные проверки); HTML — по parsing.md.
- forward: собрать основания — процесс, функции открытия формы, сущность и
  справочники, карточки EXTINT, комплект контролей, соседние формы-образцы.
  Чего не хватает — список заглушек и открытых вопросов ДО проектирования.

### Шаг 2. Инвентаризация фрагментов (2a) — целевой адрес

По каждому фрагменту исходника (reverse) или каждому спроектированному факту
(forward) зафиксируй целевой адрес в структуре: раздел (§2–§6) + строка
(П-N / Е-N шаг / М-N) + ярлык (для П-N). Не пиши документ, пока инвентарь не
завершён — иначе поперечные эффекты «прилипают» к полю-источнику. Правила
распределения по разделам — references/templates.md («Куда что кладётся»).

### Шаг 3. Собрать документ по шаблону

references/templates.md: скелет, правила каждого раздела, словарь ярлыков П-N,
правила шагов и переиспользования, именованные исходы, реестр сообщений.

### Шаг 4. Предпроверка инвариантов

Прогнать **все** структурные инварианты references/checklist.md (биекции
поле↔П-N, событие↔подраздел, сообщение↔показ; достижимость шагов; резолвинг
ссылок на шаги/EXTINT/CTL; парность исходов; нотация). В forward-режиме отчёт
предпроверки прикладывается к передаче на ревью.

### Шаг 5. Отчёт пользователю

Перечислить: пометки `уточняется` и открытые вопросы; созданные заглушки;
спорные решения распределения; (reverse) `[НЕ РАСПОЗНАНО]` и дефекты исходника.

## Разделение труда со смежными скиллами

- **control-split** — создаёт комплект контролей (BE/FE, CTL). Этот скилл на
  него только ссылается.
- **screen-form-restructure** — легаси-формат «единая таблица полей»; общие
  авторитетные правила (formatting.md — нотация; parsing.md — разбор HTML;
  type-dictionary.md — UI-типы; принцип инициализации параметров вызовов)
  живут там и здесь не дублируются.

## Справочные файлы

- `references/templates.md` — скелет документа и правила каждого раздела
  (обязателен при сборке).
- `references/checklist.md` — финальная сверка и инварианты предпроверки
  (обязателен перед сдачей).
