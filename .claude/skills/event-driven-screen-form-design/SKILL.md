  нуля по процессу, модели данных и функциям, с ревью человеком. Используй этот
  скилл, когда пользователь просит описать ЭФ в событийном формате / событийной
  модели, спроектировать новую экранную форму или клиентский путь, перевести
  описание ЭФ на события, привязать событийную ЭФ к реестру контролей (CTL).
  Привязка по умолчанию — к готовому реестру CTL сервиса; конвейер с
  control-split — только по явному запросу разделения контролей.
  Для легаси-формата «единая таблица полей» используй
  screen-form-restructure; для разделения контролей — control-split (этот скилл
  контроли НЕ создаёт, только ссылается).

# Событийное описание экранной формы (FR / EV / MSG) — v1.1

> **Верховный принцип (reverse): СОХРАНЕНИЕ СМЫСЛА — высший приоритет.**
> Переформатирование — семантическое преобразование через понимание: каждый
> фрагмент исходника должен появиться в целевой структуре как полный смысловой
> эквивалент (условия, ограничения, умолчания). Переформулировать можно; терять
> и огрублять — нельзя. Не уверен в эквивалентности — пометь
> `[НЕ РАСПОЗНАНО: <дословный исходный текст>]`, не сглаживай.
>
> **Верховный принцип (forward): ЗАЗЕМЛЕНИЕ — каждый элемент имеет
> основание-источник.** Шаг события — ссылку на требование (PRC/FUN/CTL/ENT)
> либо пометку `уточняется` с формулировкой открытого вопроса; вызов внешнего
> метода — карточку EXTINT (нет карточки — создаётся заглушка; выдуманный метод
> запрещён); сообщение — заполненный «Показ». Элемент без основания и без
> пометки — дефект генерации. `[НЕ РАСПОЗНАНО]` в forward не используется —
> его место занимают `уточняется` и открытые вопросы (по конвенциям репозитория
> документации — вне docs/, не inline-TODO).

## Что делает скилл

Выход — **один документ на одну вкладку/страницу ЭФ** (гранулярность, скелет и
правила всех разделов — references/templates.md, это авторитет по структуре):

1. **Описание ЭФ** — дословная таблица; строка «Дополнительные действия при
   открытии» ссылается на событие EV-1.
2. **Назначение**.
3. **Реестр полей**.
4. **Видимость блоков**.
5. **Правила полей (статика)** — подразделы **FR-N**.
6. **События** — реестр **EV-N** + подразделы; шаги — со стабильными ID
   **EV-N.M**; кнопки — тоже события.
7. **Сообщения** — реестр **MSG-N** с обратной ссылкой «Показ».

Режимы:

- **reverse** — вход: страница ЭФ (экспорт Confluence или markdown). Дословный
  перенос смысла в событийную структуру; технический разбор HTML — по
  `../screen-form-restructure/references/parsing.md`.
- **forward** — вход: процесс (PRC), модель данных (ENT/EXT), функции (FUN),
  контроли (CTL), интеграции (EXTINT), соседние формы сервиса как образцы.
  Агент проектирует, человек ревьюит; новый путь в существующем сервисе —
  **дельта**: новые события + ссылки на существующие FR-N/исходы/EXTINT/CTL.
- **замыкание** — фаза 3 конвейера контролей (только в варианте «конвейер»,
  см. ниже): вход — готовый документ формы с маркерами кандидатов + парный
  FE-файл контролей (выход control-split); механическая замена пометок
  ссылками на карточки, соответствие берётся из колонки «Ориг.» FE-файла;
  правила — templates.md, «Кандидаты в контроли и замыкание ссылок».

**Привязка к контролям** — независимая от reverse/forward ось, два варианта:

- **прямая привязка к реестру CTL — вариант по умолчанию**: контроли сервиса
  уже существуют чистовым каталогом (реестр `CTL-NNN` с матрицей групп
  `CTL-GRP-N` + карточки); ссылки на карточки и группы ставятся сразу при
  сборке, фазы конвейера не нужны; правила — templates.md, «Прямая привязка
  к реестру CTL».
- **конвейер с control-split** — включается **только по явному указанию
  пользователя** (просьба разделить контроли FE/BE, слова «control-split»,
  «конвейер», «фаза 1»/«замыкание»): фаза 1 — маркеры кандидатов, фаза 3 —
  замыкание.

Слова reverse/forward выбирают **источник документа** (переформатирование vs
проектирование) и вариант привязки **не переключают**. Без явного указания
конвейер не включается — даже если на вход подан сырой единый файл контролей
без стабильных CTL-ID (как действовать в этом случае — templates.md, там же).

## Идентификаторы

Политика комплекта (детали шагов — templates.md, §5, авторитет):

- **Только латиница** (`A–Z`, `0–9`, дефис, точка); кириллица в ID запрещена.
  Префиксы — ≥2 символов, не состоящие целиком из букв-омоглифов кириллицы
  (задокументированное исключение — `EXT`).
- **Локальные ID** — `FR-N` (правило поля), `EV-N` (событие), `EV-N.M` (шаг),
  `MSG-N` (сообщение); уникальны **в пределах документа**. Полный адрес извне —
  `<id документа>/<локальный ID>`, напр. `SCR-CL-03.1/EV-1.9`.
- **Стабильность**: ID не перенумеровываются и не переиспользуются.
- **Ось классификации — не в ID**: изменчивые свойства (FE/BE контроля,
  item/list формы) живут во frontmatter/реестрах, не в префиксе.

## Словарь артефактов комплекта

Скилл ссылается на артефакты SRS-комплекта сервиса (канонический реестр
префиксов — схема ID в конвенциях целевого репозитория документации):

- `ENT-NNN` — сущность модели данных сервиса;
- `EXT-NNN` — справочник **или сущность платформы** (вне сервиса; реестр —
  `dictionaries.md`); не путать с EXTINT;
- `EXTINT-NNN` — карточка внешней интеграции: точка обмена со смежной
  банковской АС (`external-integrations/`);
- `FUN-CL-NN` / `FUN-BNK-NN` / `FUN-SYS-NN` — функция (контур Клиент / Банк /
  системная);
- `PRC-NNN` — процесс; `STM-NNN` — статусная модель (диаграмма состояний);
- `CTL-NNN` / `CTL-GRP-N` — контроль / группа контролей (`controls/`);
- `SCR-CL-NN[.K]` / `SCR-BNK-NN[.K]` — экранная форма (`.K` — вкладка);
- `NTF-NNN` — нотификация; `PRN-NNN` — печатная форма;
- `DE-NNN` — элемент данных: бизнес-реквизит клиентского пути; реестр —
  `data-model/data-elements.md` сервиса (см. templates.md, «Реестр элементов
  данных»). Поля форм — представления DE, атрибуты ENT — хранение; глобальная
  трассировка полей данных идёт через DE, а не через локальные FR-N.

## Ключевые принципы

- **Нотация — по действующему авторитету**
  `../screen-form-restructure/references/formatting.md` (жирные Если/то/иначе,
  `=`/`==`/`≠`, шаги, `→` только «найти/взять запись»). **Эталон-пример формата
  содержит нотационные ошибки — из него берётся структура, нотацию не копировать.**
- **Вызовы функций/методов — с инициализацией входных параметров**; авторитет —
  SKILL.md скилла screen-form-restructure, одноимённый принцип (параметры не
  терять; отсутствующие — по описанию функции (приоритет) и OpenAPI; источник
  значения каждого параметра указан либо `уточняется`).
- **Видимость — всегда инлайн** («Видимость блоков» + ярлык «Видимость» в FR-N).
  Модель выноса видимости в контроли (SF-NN, R/V) не используется.
- **Контроли скилл не создаёт — только привязывает.** По умолчанию — прямая
  привязка к реестру CTL: найденная в реестре проверка сразу получает
  постоянную ссылку на карточку/группу; ненайденная остаётся inline дословно
  **без маркера**, кандидаты в новые CTL — только в отчёт и open-questions;
  карточки CTL не редактируются (у каталога свой скилл-писатель). Маркеры
  `CAND-N` в этом варианте запрещены. В варианте «конвейер» (только по явному
  указанию) карточки контролей создаёт control-split; этот скилл: **(фаза 1)**
  при сборке документа сверяет каждую встроенную в текст ЭФ проверку с общим
  файлом контролей сервиса (вход на чтение): найдена — пометка с исходным №,
  не найдена — маркер кандидата `CAND-N` — **только для проверок полевой
  природы**; проверки-шаги процедур событий не картуются и остаются шагами
  (templates.md, «Кандидаты в контроли»);
  **(фаза 3, режим «замыкание»)** после прогона control-split заменяет пометки
  ссылками на карточки FE-файла. **Ссылки формы — только в FE-файл** (полевые
  контроли, зеркала ранних отказов, группы); ссылок формы на BE-файл не
  ставить: серверное исполнение проверок обеспечивается на стороне методов и
  BE-документа — вне объёма этого скилла.
- **Ссылочная политика — по месту работы:**
  - **в SRS-комплекте** (`output/<service>/docs/`): ссылки только на артефакты
    комплекта; нет карточки — создаётся **заглушка** с пометкой («карточка-
    заглушка, страница метода вне выгрузки»);
  - **вне комплекта** (reverse-прогон рядом с исходником): допустимы исходные
    ссылки (Confluence/файлы исходника); заглушки не создаются.
  Выдуманный артефакт запрещён в обоих случаях.
- **Тексты сообщений живут только в реестре MSG** — шаг/правило ссылается
  (`информер MSG-6, исход «Скрытие полей»`), не дублирует текст.
- **Расположение — дизайн, не логика.** Позиционные уточнения («(у поля X)»,
  «(в шапке)») в именах полей и событий не используются — расположение задают
  макеты; логическое описание не меняется при перестановке элементов
  (templates.md, §4).

## Процедура

### Шаг 1. Собрать вход

- reverse: разобрать исходную страницу (frontmatter, история изменений, поля,
  действия, сообщения, встроенные проверки); HTML — по parsing.md. Плюс
  **контроли сервиса — на чтение**, для сверки встроенных проверок: по
  умолчанию — каталог реестра CTL с карточками (прямая привязка); при явно
  запрошенном конвейере — общий файл контролей (фаза 1).
- forward: собрать основания — процесс, функции открытия формы, сущность и
  справочники, карточки EXTINT, комплект контролей, соседние формы-образцы.
  Чего не хватает — список заглушек и открытых вопросов ДО проектирования.

### Шаг 2. Инвентаризация фрагментов — целевой адрес

По каждому фрагменту исходника (reverse) или каждому спроектированному факту
(forward) зафиксируй целевой адрес в структуре: раздел + строка
(FR-N / EV-N.M / MSG-N) + ярлык (для FR-N). Не пиши документ, пока инвентарь не
завершён — иначе поперечные эффекты «прилипают» к полю-источнику. Правила
распределения — references/templates.md («Куда что кладётся»).

### Шаг 3. Собрать документ по шаблону

references/templates.md: скелет, правила каждого раздела, словарь ярлыков FR-N,
правила шагов EV-N.M, именованные исходы, реестр сообщений.

### Шаг 4. Предпроверка инвариантов

Прогнать **все** структурные инварианты references/checklist.md. В
forward-режиме отчёт предпроверки прикладывается к передаче на ревью.

### Шаг 5. Отчёт пользователю

Перечислить: пометки `уточняется` и открытые вопросы; созданные заглушки;
спорные решения распределения; (reverse) `[НЕ РАСПОЗНАНО]` и дефекты исходника.

## Разделение труда со смежными скиллами

- **control-split** — создаёт комплект контролей (BE/FE, CTL). Этот скилл на
  него только ссылается.
- **screen-form-restructure** — легаси-формат «единая таблица полей»; общие
  авторитетные правила (formatting.md — нотация; parsing.md — разбор HTML;
  type-dictionary.md — UI-типы; принцип инициализации параметров вызовов)
  живут там и здесь не дублируются.

## Справочные файлы

- `references/templates.md` — гранулярность, скелет документа и правила каждого
  раздела (обязателен при сборке).
- `references/checklist.md` — финальная сверка и инварианты предпроверки
  (обязателен перед сдачей).
