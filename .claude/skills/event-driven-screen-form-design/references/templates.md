# Шаблон событийного описания ЭФ

Нотация всех примеров и целевого текста — по
`../../screen-form-restructure/references/formatting.md` (жирные
**Если**/**то**/**иначе**; `=` присваивание / `==` сравнение / `≠`; шаги с новой
строки жирным; `→` — только «найти/взять запись»). Эталон-пример формата содержит
нотационные ошибки — не воспроизводить (см. SKILL.md).

## Скелет документа

```markdown
---
id: SCR-<роль>-<NN>[.<M>]
type: screen-form
status: draft
service: <код сервиса>
version: <версия сервиса>
---

# <Название формы / вкладки>

## Описание ЭФ

| | |
|---|---|
| **Зачем нужна ЭФ** | … |
| **Как перейти на ЭФ** | Вызовом функций (входные параметры — по описанию функции):<br>- [FUN-…](…) — … |
| **Дополнительные действия, выполняемые при открытии ЭФ** | Событие [Е-1](#е-1-открытие-формы). |
| **Макеты ЭФ** | … |

## 1. Назначение

## 2. Реестр полей

## 3. Видимость блоков

## 4. Правила полей (статика)

## 5. События

## 6. Сообщения
```

При reverse-режиме frontmatter дополняется полями исходника
(`source`, `confluence_page_id`), переносится история изменений. Имя файла:
в SRS-комплекте — по конвенции комплекта (`scr-<роль>-NN-<слаг>.md`); вне
комплекта (прогон рядом с исходником) — оригинальное имя с префиксом `EV-`
(после префикса в квадратных скобках, если он есть), исходник не перезаписывать.

## Куда что кладётся (распределение осей)

| Смысл фрагмента | Адрес |
|---|---|
| зачем форма, как открыть, макеты | «Описание ЭФ» (дословная таблица) |
| цель, сущность, экран, функции открытия | §1 Назначение |
| перечень полей, блок, UI-тип, обязательность | §2 Реестр |
| показ/скрытие целого блока | §3 Видимость блоков |
| значения/источник списка, отображение, предзаполнение, редактируемость, доступность, формат, сортировка, поиск, подсказка, видимость поля | §4, П-N поля |
| открытие формы, изменение поля, чекбокс, кнопка — и их реакции | §5, Е-N |
| текст, тип и кнопки сообщения | §6, М-N; шаг только ссылается |
| правила проверок (контроли) | НЕ здесь — ссылка на CTL-NN / CTL-GRP |

## Раздел «Описание ЭФ»

Обязательный. Табличная форма (2 колонки `ярлык | содержимое`) сохраняется —
не разворачивать в прозу (правило и обоснование — templates.md скилла
screen-form-restructure, раздел «Описание ЭФ»). Отличие от легаси-формата одно:
строка «Дополнительные действия, выполняемые при открытии ЭФ» не содержит
алгоритм, а ссылается на событие Е-1 (можно с кратким перечнем проверок).
«Как перейти на ЭФ» — с входными параметрами функций открытия и источниками
их значений.

## §1 Назначение

1–3 предложения: что заполняет форма (ссылка на ENT), из чего состоит
(вкладка/экран, ссылка на README экрана), какими функциями открывается
(ссылки FUN). Без пересказа поведения.

## §2 Реестр полей

Вводный абзац: обязательность проверяется контролем
`[CTL-NN …](…)` в точках группы `[CTL-GRP-N …](…)`; проверяются только поля,
отображаемые в момент проверки. Далее навигация: видимость блоков — §3,
правила полей — §4 (П-N), поведение — §5 (Е-N).

| № | Блок | Поле | Тип | Обяз. |
|---|---|---|---|---|

- **Тип** — UI-тип из `../../screen-form-restructure/references/type-dictionary.md`,
  при необходимости с уточнением способа заполнения («текст, автоматически»,
  «чекбокс, вручную»); служебные (не отображаемые) поля включаются в реестр с
  типом «служебное (не отображается)».
- **Обяз.** — Да/Нет/«—» (для автозаполняемых и служебных). Условная
  обязательность здесь не расписывается — она в контроле CTL.
- Блоки перечисляются в порядке отображения; поле вне блока — «—».

## §3 Видимость блоков

Маркированный список: блок — условие. Тривиальное — одной строкой («всегда»);
составное — вложенным списком условий («если одновременно: …»). Условия — над
полями формы, атрибутами сущностей и настраиваемыми параметрами, в нотации
formatting.md. Видимость отдельного поля — не здесь, а в П-N поля.

## §4 Правила полей (статика) — П-N

Подраздел `### П-N. «<Имя поля>»` на каждое поле реестра (биекция с §2,
включая служебные; допускается один П-N на пару однотипных служебных полей).
Содержимое — ярлыки-буллеты, только применимые к полю:

- **Значения:** — источник списка/enum (справочник EXT, ответ EXTINT, литералы);
- **Отображение значения:** — что показывается (атрибут, приоритет при пустом);
- **Формат:** / **Формат в списке:** / **Формат в поле:** — маска, шаблон,
  пример, плейсхолдер;
- **Сортировка:** / **Поиск:** — для списков;
- **Предзаполнение:** — умолчание при создании; **Заполнение при открытии на
  редактирование:** — маппинг из атрибутов ENT;
- **Редактируемость:** / **Доступность:** / **Активность:** — когда поле
  нередактируемо/недоступно (условия — `==`);
- **Видимость:** — показ/скрытие поля и лейбла («Название поля на форме не
  отображается» — явная оговорка);
- **Подсказка:** — иконка/текст;
- **Особый случай:** — исключительная ситуация с ссылкой на М-N;
- **Действие:** — для текстов-ссылок (что открывает нажатие).

Вычисляемое значение без списка ярлыков допускается формулой первой строкой:
`= <выражение>` (присваивание — одиночный `=`). Каскадные пересчёты
(«переопределяется при изменении X») здесь не описываются процедурно — правило
вычисления в П-N, момент срабатывания — в шаге события Е-N.

## §5 События — Е-N

Сначала реестр:

| ID | Триггер | Суть реакции |
|---|---|---|

Затем — именованные **исходы**, если используются (определяются один раз до
подразделов): `**Исход «<Имя>»**: <эффект>; обработка события завершается.`

Затем `### Е-N. <Триггер>` на каждое событие реестра (биекция). Виды событий:

- **Е-1 «Открытие формы» — обязательное** (маршрутизация по статусу,
  инициализация, проверки, запросы, заполнение полей).
- Изменение поля / установка чекбокса.
- **Кнопка — тоже событие**: сначала свойства `**Видимость:** …`
  `**Активность:** …`, затем `**При нажатии:**` с шагами.

Правила шагов:

- **Стабильный ID шага `Е-N.M`** — каждый шаг начинается со своего ID жирным:
  `**Е-1.1.** <Смысл шага>: …`. ID — идентификатор, а не порядковый номер:
  - **порядок выполнения задаёт позиция в документе**, а не величина M;
  - ID **никогда не перенумеровываются и не переиспользуются**; при вставке
    шага между `Е-1.9` и `Е-1.10` новый шаг получает **следующий свободный
    номер** события (например `Е-1.24`) и ставится физически между ними;
  - удалённый ID больше не присваивается.
  Это закрывает риск молчаливого сдвига ссылок при правках.
- Ветвление в нотации formatting.md; **переходы — только явными ID**
  (`**то** — Е-1.10`, не «следующий шаг»): при вставке шага автор сознательно
  перепроводит переходы, а не полагается на соседство.
  `**Е-1.9.** <Смысл шага>:<br>- **Если** <условие с ==>,<br>- **то** — Е-1.10,<br>- **иначе** <эффект>, конец.`
- Каждая ветвь завершается: переходом на шаг (`Е-N.M`), «конец», именованным
  исходом или показом сообщения М-N (+ исход).
- **Вызов внешнего метода** — ссылка на карточку EXTINT + инициализация входных
  параметров (`<параметр> = <источник>`) и разбор исходящих (условия — `==`);
  нет карточки — заглушка. **Контроль** — ссылка на CTL-NN / CTL-GRP; текст
  правила не дублируется.
- **Переиспользование**: другое событие ссылается диапазоном по стабильным ID
  («Выполняются шаги Е-1.7–Е-1.23»), с оговорками об исключениях
  («без повторного вызова …»). Диапазон означает **все шаги между граничными ID
  в порядке документа** — вставленный между ними шаг входит в диапазон
  автоматически; если не должен входить — у переиспользующего события
  дописывается оговорка-исключение. При интенсивном переиспользовании выделяй
  именованную подпроцедуру вместо диапазона.
- (forward) Каждый шаг — с основанием-источником (ссылка PRC/FUN/CTL/ENT)
  либо `уточняется` (см. SKILL.md, «Заземление»).

## §6 Сообщения — М-N

Вводный абзац: что означает «Показ»; семантика скрытия/очистки полей при
показе (общее правило + перечень исключений).

| ID | Тип | Показ | Содержание |
|---|---|---|---|

- **Тип** — Информер / Блоккер / Плашка / Модальное окно (словарь сервиса).
- **Показ** — обратная ссылка на стабильный ID: `Е-N.M` (можно несколько)
  или `П-N`. Пустого «Показа» не бывает.
- **Содержание** — заголовок/текст (reverse — дословно), кнопки с действиями;
  действие кнопки — ссылка на функцию/шаг («повтор шага Е-1.12»),
  с заземлением в forward. Плейсхолдеры вида `<телефон>` — с указанием
  источника (настраиваемый параметр).
- В шагах и правилах текст сообщения не дублируется — только `М-N`.
