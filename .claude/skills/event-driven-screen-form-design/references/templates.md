# Шаблон событийного описания ЭФ

Нотация всех примеров и целевого текста — по авторитету
`../../screen-form-restructure/references/formatting.md`; нотационные ошибки
эталона-примера не воспроизводить (см. SKILL.md, «Ключевые принципы»).

## Гранулярность и идентификаторы

- **Один документ = одна вкладка/страница ЭФ.** Экран с вкладками — каталог:
  `README.md` (обзор, ID экрана) + файл на вкладку с суб-ID `SCR-<роль>-NN.K`
  (роль: `CL` — Клиент, `BNK` — Банк; имя файла `scr-<роль>-nn.k-<слаг>.md` —
  по конвенциям комплекта).
- **Локальные ID** (уникальны в пределах документа, стабильны — см. §5):
  `FR-N` — правило поля, `EV-N` — событие, `EV-N.M` — шаг события,
  `MSG-N` — сообщение. Кириллица в ID запрещена.
- **Ссылки:** на артефакты комплекта (ENT/EXT/EXTINT/FUN/PRC/STM/CTL/SCR) —
  **всегда markdown-ссылками** на файл/якорь. Локальные ID внутри документа —
  **простым текстом** (`FR-1`, `EV-1.9`, `MSG-3`), без markdown-якорей;
  проверка такой ссылки — существование ID в документе. Вне SRS-комплекта
  (reverse рядом с исходником) вместо артефактов — исходные ссылки
  (SKILL.md, «Ссылочная политика»).

## Скелет документа

```markdown
---
id: SCR-<роль>-<NN>[.<K>]
type: screen-form
status: draft
service: <код сервиса>
version: <версия сервиса>
---

# <Название формы / вкладки>

## Описание ЭФ

| | |
|---|---|
| **Зачем нужна ЭФ** | … |
| **Как перейти на ЭФ** | Вызовом функций (входные параметры — по описанию функции):<br>- [FUN-…](…) — … |
| **Дополнительные действия, выполняемые при открытии ЭФ** | Событие EV-1. |
| **Макеты ЭФ** | … |

## 1. Назначение

## 2. Реестр полей

## 3. Видимость блоков

## 4. Правила полей (статика)

## 5. События

## 6. Сообщения
```

При reverse-режиме frontmatter дополняется полями исходника
(`source`, `confluence_page_id`); **история изменений** исходника переносится
блоком сразу после frontmatter (перед H1-заголовком — строкой
`**История изменений:** …`); при частичном прогоне (одна вкладка многовкладочной
страницы) допускается указатель на исходник вместо переноса. Имя файла вне
комплекта — оригинальное имя с префиксом `EV-` (после префикса в квадратных
скобках, если он есть); для вкладки — суффикс `-вкладка-<название-вкладки>`
(слагом, **без номера** — роль номера выполняет название вкладки). Исходник не
перезаписывать.

**Прослеживаемость (reverse):** каждый шаг события снабжается пометкой
`*(исх. Шаг N)*` с номером шага исходника; каждое сообщение реестра —
`*(исх. Сообщение X "<название>")*`, а при отсутствии номера в исходнике —
`*(исходный номер отсутствует)*`.

## Куда что кладётся (распределение осей)

| Смысл фрагмента | Адрес |
|---|---|
| зачем форма, как открыть, макеты | «Описание ЭФ» (дословная таблица) |
| цель, сущность, экран, функции открытия | §1 Назначение |
| перечень полей, блок, UI-тип, обязательность | §2 Реестр |
| показ/скрытие целого блока | §3 Видимость блоков |
| значения/источник списка, отображение, предзаполнение, редактируемость, доступность, формат, сортировка, поиск, подсказка, видимость поля | §4, FR-N поля |
| открытие формы, изменение поля, чекбокс, кнопка — и их реакции | §5, EV-N |
| текст, тип и кнопки сообщения | §6, MSG-N; шаг только ссылается |
| правила проверок (контроли) | НЕ здесь — ссылка на CTL-NNN / CTL-GRP-N |
| каскад «изменение A меняет B» | шаг события изменения A; правило вычисления — в FR-N поля B |

## Раздел «Описание ЭФ»

Обязательный; табличная форма (2 колонки `ярлык | содержимое`) сохраняется, в
прозу не разворачивается — авторитет правила: templates.md скилла
screen-form-restructure, раздел «Описание ЭФ». Отличий от легаси-формата два:
строка «Дополнительные действия…» не содержит алгоритм, а ссылается на EV-1
(можно с кратким перечнем проверок); «Как перейти на ЭФ» — с входными
параметрами функций открытия и источниками их значений.

## §1 Назначение

1–3 предложения: что заполняет форма (ссылка на ENT), из чего состоит
(вкладка/экран, ссылка на README экрана), какими функциями открывается
(ссылки FUN). Без пересказа поведения.

## §2 Реестр полей

Вводный абзац: обязательность проверяется контролем
`[CTL-NNN …](…)` в точках группы `[CTL-GRP-N …](…)`; проверяются только поля,
отображаемые в момент проверки. Далее навигация: видимость блоков — §3,
правила полей — §4 (FR-N), поведение — §5 (EV-N).

| № | Блок | Поле | Тип | Обяз. |
|---|---|---|---|---|

- **В реестре — только поля.** Кнопки — не строки реестра (они события, §5);
  сообщения — не строки реестра (они в §6). Это отличие от легаси-формата,
  где сообщения были строками таблицы полей.
- **№** — порядковый номер отображения, не идентификатор (при вставке поля
  допустимо перенумеровать). Для ссылок на поле используется его `FR-N`.
- **Тип** — **чистый UI-тип** из словаря
  `../../screen-form-restructure/references/type-dictionary.md` (Текст, Список,
  Переключатель, Чекбокс, Заголовок, Иконка, Сумма, Текст со ссылками, …) с
  нормализацией (Confluence «Текст» + логика «выбор из списка» — это «Список»).
  **Способ заполнения в колонку не примешивать** («текст, автоматически»,
  «чекбокс, вручную» — ошибка: способ заполнения — стандартный атрибут FR-N;
  эталон-пример этот тип со способом смешивает — не воспроизводить). Служебные
  поля без UI-представления включаются в реестр с псевдо-типом
  «служебное (не отображается)».
- **Обяз.** — Да / Нет / «—» (для автозаполняемых и служебных). Условная
  обязательность здесь не расписывается — она в контроле CTL.
- Блоки перечисляются в порядке отображения; поле вне блока — «—».

## §3 Видимость блоков

Маркированный список: блок — условие. Тривиальное — одной строкой («всегда»);
составное — вложенным списком условий («если одновременно: …»). Условия — над
полями формы, атрибутами сущностей и настраиваемыми параметрами, в нотации
formatting.md. Видимость отдельного поля — не здесь, а в FR-N поля.

## §4 Правила полей (статика) — FR-N

Подраздел `### FR-N. «<Имя поля>»` на каждое поле реестра. Соответствие
поле ↔ FR-N — взаимно однозначное, с одним допущением: **пара однотипных
служебных полей может описываться одним общим FR-подразделом** (обе строки
реестра тогда указывают на него).

**Имя поля — логическое, без позиционных уточнений** («(у поля X)», «(слева
внизу)», «(в шапке)»): расположение элементов задают макеты («Описание ЭФ»),
логическое описание не должно меняться при перестановке элементов. Поля-омонимы
различаются блоком: в реестре — колонкой «Блок», в заголовке FR — уточнением
«(блок «<Имя блока>»)»; ссылки в тексте — всегда по FR-N.

Содержимое — **стандартные атрибуты** (наследие колонок табличной формы; ID и
наименование ушли в заголовок подраздела). Атрибуты указываются **строго в этом
порядке**, у каждого поля; неприменимый атрибут опускается:

1. **Тип:** — UI-тип из словаря type-dictionary (тот же, что в реестре §2).
2. **Формат:** — маска/шаблон/пример/плейсхолдер; при различии представлений —
   под-ярлыки **В списке:** / **В поле:**.
3. **Способ заполнения:** — Вручную / Выбор из списка / Автоматически; здесь же
   редактируемость/доступность/активность, в т.ч. условная (условия — `==`).
4. **Обязательность:** — Да / Нет / «—» (та же, что в реестре §2).
5. **Видимость:** — показ/скрытие поля и лейбла («Название поля на форме не
   отображается» — явная оговорка).
6. **Логика установки значения:** — откуда значение; внутри — под-ярлыки:
   **Источник списка:** (правила отбора записей), **По умолчанию:**
   (предзаполнение при создании), **При открытии на редактирование:** (маппинг
   из атрибутов сущности), **Переформирование списка:** (когда список
   пересобирается).

**Дополнительные атрибуты — только после стандартных.** «Тип» и
«Обязательность» дублируются в реестре §2 как свод: авторитет — FR-N,
расхождение реестра и FR — дефект (проверяется инвариантом).

**Дополнительные атрибуты** — только когда требование не относится ни к одному
стандартному; имя дополнительного атрибута **едино для всех полей документа**
с аналогичной характеристикой (синонимы не изобретать). Действующий реестр
дополнительных (расширять осознанно): **Сортировка:**, **Поиск:**,
**Подсказка:**, **Действие:** (что открывает/вызывает нажатие на текст-ссылку),
**Особый случай:** (исключительная ситуация со ссылкой на MSG-N).

Вычисляемое значение без списка атрибутов допускается формулой первой строкой:
`= <выражение>` (присваивание — одиночный `=`). Условное вычисление — той же
формой с ветвлением: `= true, **если** <условие с ==>; **иначе** = false`.
Каждый атрибут — отдельным буллетом: не сливать разные атрибуты (например,
видимость и логику значения) в одну строку. Каскадные пересчёты
(«переопределяется при изменении X») здесь не описываются процедурно —
правило вычисления в FR-N, момент срабатывания — в шаге EV-N.M.

## §5 События — EV-N

Сначала реестр:

| ID | Триггер | Суть реакции |
|---|---|---|

- **Суть реакции** — одна строка без ветвлений (перечень эффектов или диапазон
  шагов); детальная логика — только в подразделе.

Затем — именованные **исходы**, если используются (определяются один раз до
подразделов): `**Исход «<Имя>»**: <эффект>; обработка события завершается.`
Исход — это **имя** (по-русски, в кавычках), не идентификатор.

Затем `### EV-N. <Триггер>` на каждое событие реестра (взаимно однозначно).
Виды событий:

- **EV-1 «Открытие формы» — обязательное** (маршрутизация по статусу,
  инициализация, проверки, запросы, заполнение полей).
- Изменение поля / установка чекбокса.
- **Кнопка — тоже событие**: сначала свойства `**Видимость:** …`
  `**Активность:** …`, затем `**При нажатии:**` с шагами. Название события —
  без позиционных уточнений («в шапке», «левый нижний угол»): расположение
  задают макеты.

### Правила шагов (авторитет по стабильным ID)

- **Шаги — НЕ нумерованный markdown-список.** Оформление: последовательность
  абзацев (или пунктов маркированного списка `- `), каждый начинается со
  стабильного ID жирным: `**EV-1.1.** <Смысл шага>: …`. Никакой второй
  (визуальной) нумерации рядом с ID быть не должно.
- **ID шага `EV-N.M` — идентификатор, а не порядковый номер:**
  - порядок выполнения задаёт **позиция в документе**, а не величина M;
  - ID **никогда не перенумеровываются и не переиспользуются**; при вставке
    шага между `EV-1.9` и `EV-1.10` новый шаг получает **следующий свободный
    номер** события (например `EV-1.24`) и ставится физически между ними;
  - удалённый ID больше не присваивается.
- **Переходы — только явными ID** (`**то** — EV-1.10`, не «следующий шаг»):
  при вставке шага автор сознательно перепроводит переходы. Пример шага:
  `**EV-1.9.** <Смысл шага>:<br>- **Если** <условие с ==>,<br>- **то** — EV-1.10,<br>- **иначе** <эффект>, конец.`
- Каждая ветвь завершается: переходом на шаг (`EV-N.M`), «конец», именованным
  исходом или показом сообщения MSG-N (+ исход).
- **Вызов внешнего метода** — ссылка на карточку EXTINT (вне комплекта —
  исходная ссылка) + инициализация входных параметров
  (`<параметр> = <источник>`) и разбор исходящих (условия — `==`).
  **Контроль** — ссылка на CTL-NNN / CTL-GRP-N; текст правила не дублируется.
- **Переиспользование**: другое событие ссылается диапазоном по стабильным ID
  («Выполняются шаги EV-1.7–EV-1.23»), с оговорками об исключениях
  («без повторного вызова …»). Диапазон означает **все шаги между граничными ID
  в порядке документа** — вставленный между ними шаг входит в диапазон
  автоматически; если не должен входить — у переиспользующего события
  дописывается оговорка-исключение. При интенсивном переиспользовании выделяй
  именованную подпроцедуру вместо диапазона.
- (forward) Каждый шаг — с основанием-источником либо `уточняется`
  (SKILL.md, «Заземление»).
- **(reverse) Унаследованная недостижимость:** шаг, недостижимый уже в
  исходнике (переход на него нигде не описан), переносится **с явной пометкой
  дефекта исходника** — не чинится молча и не выбрасывается; инвариант
  достижимости считается выполненным при наличии пометки.
- **(reverse) Сужение ссылки на процедуру:** если исходник ссылается на
  процедуру целиком («выполняются проверки раздела X») без границ, а часть
  шагов к событию заведомо неприменима, ссылку допустимо сузить до диапазона
  применимых шагов **только с явной пометкой-интерпретацией** (какие шаги
  исключены и почему); без пометки — переносить ссылку на процедуру целиком.

## §6 Сообщения — MSG-N

Вводный абзац: что означает «Показ»; семантика скрытия/очистки полей при
показе (общее правило + перечень исключений).

| ID | Тип | Показ | Содержание |
|---|---|---|---|

- **Тип** — Информер / Блоккер / Плашка / Модальное окно (словарь сервиса).
- **Показ** — обратная ссылка на стабильный ID: `EV-N.M` (можно несколько)
  или `FR-N`. Пустого «Показа» не бывает; **если условие показа в исходнике
  не определено или противоречиво** (reverse), допустима пометка `уточняется`
  с указанием причины — это фиксация дефекта исходника, не пропуск.
- **Содержание** — заголовок/текст (reverse — дословно), кнопки с действиями;
  действие кнопки — ссылка на функцию/шаг («повтор шага EV-1.12»),
  с заземлением в forward. Плейсхолдеры вида `<телефон>` — с указанием
  источника (настраиваемый параметр).
- В шагах и правилах текст сообщения не дублируется — только `MSG-N`.
