# Примеры запросов пользователя (формулировки для предсказуемого результата)

Два варианта использования скилла — по варианту привязки к контролям
(SKILL.md, «Привязка к контролям»). Формулировки ниже — рекомендуемые
шаблоны: пользователь подставляет пути и объём.

## Вариант А — конвейер с разделением контролей (три запроса)

Для команд, переходящих на разделение контролей FE/BE. Три запроса,
строго по порядку; фаза 2 — скилл control-split.

**Запрос 1 — событийное описание ЭФ (фаза 1, разметка кандидатов):**

> «Примени скилл event-driven-screen-form-design (reverse) **в конвейере с
> control-split, фаза 1 — разметка кандидатов**, к странице ЭФ
> `<путь к экспорту страницы>`. Общий файл контролей сервиса для сверки:
> `<путь>` (только чтение). Каталог OpenAPI-спецификаций для параметров
> вызовов: `<путь>`. Объём: вкладка «X» (или: вся страница — все вкладки,
> отдельные экраны и сквозные части). Выходные файлы
> положи в `<каталог>`.»

Входы: страница ЭФ + общий файл контролей (на чтение).
Выход: EV-документ на каждую вкладку (имя
`[…]-ЭФ-<Роль>-EV-…-вкладка-<название>.md`) с маркерами `CAND-N` и
пометками «исх. №», реестр `data-elements.md`, отчёт с перечнем кандидатов.

**Запрос 2 — раскол контролей (фаза 2, скилл control-split):**

> «Примени скилл control-split к общему файлу контролей `<путь>` в событийном
> конвейере: документы экранных форм с кандидатами — `<пути EV-документов>`.
> Ссылки на процесс/модель данных/API — `<если есть>`.»

Входы: общий файл контролей + EV-документы фазы 1.
Выход: BE-файл (с x-controls) + FE-файл на каждую вкладку — в каталоге
общего файла контролей (правило скилла, каталог указывать не нужно) +
резюме «кандидат → ID».

**Запрос 3 — замыкание (фаза 3):**

> «Скиллом event-driven-screen-form-design выполни замыкание ссылок:
> документы экранных форм — `<пути>`, парные файлы FE-контролей — `<пути>`.»

Входы: EV-документы + их FE-файлы.
Выход: чистовые документы без маркеров, все ссылки — в FE-файлы; отчёт по
инвариантам.

## Вариант Б — прямая привязка к реестру CTL (умолчание, один запрос)

Для команд, которые контроли не разделяют и ведут единый чистовой каталог
CTL (реестр + карточки).

> «Примени скилл event-driven-screen-form-design (reverse) к странице ЭФ
> `<путь к экспорту страницы>`. Контроли сервиса — каталог
> `<путь к docs/srs/controls>` (реестр CTL и файлы карточек; только чтение,
> не менять). Каталог OpenAPI-спецификаций: `<путь>`. Объём: вкладка «X»
> (или: вся страница — все вкладки, отдельные экраны и сквозные части).
> Выходные файлы положи в `<каталог>`.»

Входы: страница ЭФ + каталог CTL (реестр + карточки).
Выход — сразу чистовой, одним прогоном: EV-документ на вкладку с постоянными
ссылками `CTL-NNN`/`CTL-GRP-N`, реестр `data-elements.md`, отчёт; проверки,
не найденные в реестре, — inline дословно; кандидаты в новые CTL — списком
в отчёте и open-questions (документ они не загрязняют).

**Важное «не» для варианта Б:** в запросе не должно быть слов «раздели
контроли», «control-split», «конвейер», «фаза», «замыкание» — любое из них
переключает скилл в вариант А. Слово «(reverse)» безопасно в обоих
вариантах — оно задаёт только источник (переформатирование существующей
страницы, а не проектирование с нуля) и вариант привязки не выбирает.

## Общие правила предсказуемости (оба варианта)

- **Объём указывать всегда** («вкладка X» / «вся страница») — без него агент
  может остановиться на первой вкладке. «Вся страница» включает все вкладки,
  отдельные экраны и сквозные части (боковая панель, шапка); формулировка
  «все вкладки» будет понята буквально — не-вкладочные секции останутся вне
  объёма.
- **Выходной каталог**: если не указан, EV-документы кладутся рядом с
  исходником; FE-файлы конвейера в любом случае кладутся в каталог общего
  файла контролей.
