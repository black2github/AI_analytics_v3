# Чеклист финальной сверки и инварианты предпроверки

Прогоняется перед сдачей результата. В forward-режиме отчёт по инвариантам
прикладывается к передаче на ревью (человек проверяет смысл, не структуру).

## Смысл / заземление (ВЫСШИЙ ПРИОРИТЕТ)

- [ ] (reverse) Каждый фрагмент исходника — в целевой структуре как **полный
      смысловой эквивалент**; неуверенность — `[НЕ РАСПОЗНАНО: <дословно>]`,
      не сглажено, не выброшено.
- [ ] (forward) **Каждый шаг события** имеет основание-источник (ссылка на
      PRC/FUN/CTL/ENT) либо пометку `уточняется` с формулировкой вопроса;
      шагов «из воображения» нет.
- [ ] (forward) Открытые вопросы — вне документа (по конвенциям репозитория
      документации), в тексте только `уточняется`; inline-TODO нет.
- [ ] Каждый вызов функции/метода — с инициализацией входных параметров и
      источником значения каждого параметра (поле / атрибут / шаг) либо
      `уточняется` (принцип — SKILL.md screen-form-restructure). «Голых»
      вызовов нет.

## Инварианты структуры (машинная предпроверка)

1. [ ] **Поле ↔ П-N**: каждое поле §2 «Реестр полей» имеет подраздел П-N;
       каждый П-N — строку в §2 (биекция, включая служебные поля).
2. [ ] Каждый блок из §2 упомянут в §3 «Видимость блоков».
3. [ ] **Событие ↔ подраздел**: каждое событие реестра §5 имеет детальный
       подраздел; каждый подраздел — строку в реестре (биекция). Е-1
       «Открытие формы» присутствует.
4. [ ] Каждый шаг каждого события **достижим** (последовательный или на него
       есть переход) и **завершим** (каждая ветвь оканчивается переходом,
       «конец» или именованным исходом).
5. [ ] **Все ссылки на шаги резолвятся**: диапазоны «шаги N–M события Е-K»,
       колонка «Показ» в М-N, кнопки «Повторить шаг …». После ЛЮБОЙ
       вставки/удаления/перестановки шагов — перепроверка всех ссылок на шаги
       этого события (главная защита от сдвига нумерации).
6. [ ] **Сообщение ↔ показ**: каждое М-N имеет непустой «Показ»; каждое
       сообщение, упомянутое в шагах/правилах, есть в реестре §6 (биекция);
       тексты сообщений вне §6 не дублируются.
7. [ ] Каждый вызов внешнего метода ссылается на карточку EXTINT (или
       заглушку с пометкой); выдуманных методов и голых упоминаний нет.
8. [ ] Каждый упомянутый контроль/группа резолвится в CTL-NN / CTL-GRP
       комплекта контролей; текст правил контролей не дублируется.
9. [ ] **Исходы**: каждый использованный именованный исход определён; каждый
       определённый — использован.
10. [ ] Перекрёстные ссылки §4 ↔ §5 ↔ §6 резолвятся (П-N из шагов, Е-N из
        правил, М-N отовсюду).
11. [ ] Ссылки на артефакты SRS-комплекта (ENT/EXT/EXTINT/FUN/PRC/CTL)
        ведут на существующие файлы/якоря.

## Нотация (авторитет — formatting.md screen-form-restructure)

- [ ] «Если»/«то»/«иначе» — жирным во всех условиях (§3, П-N, шаги Е-N).
- [ ] Сравнение — `==` / `≠`; присваивание и инициализация параметров —
      одиночный `=`. Двусмысленных `=` в условиях нет (в т.ч. в текстах,
      перенесённых из эталона/исходника, — эталон содержит такие ошибки,
      они исправляются при переносе).
- [ ] `→` — только «найти/взять запись» (`ключ → Сущность`); нет `→` как
      связки последовательности/следствия/маппинга.
- [ ] Шаги — нумерованный список, ветвления читаемы, вложенность сохранена.
- [ ] Иконки/ссылки/списки/токены `<Имя атрибута>` сохранены (reverse).

## Структура документа

- [ ] Раздел «Описание ЭФ» присутствует и оформлен **таблицей** (2 колонки
      `ярлык | содержимое`), не прозой; строка «Дополнительные действия…»
      ссылается на Е-1.
- [ ] Обязательность в §2 делегирована контролю CTL (ссылка на группу), с
      оговоркой «проверяются только отображаемые поля».
- [ ] Модель SF-NN / R / V **не используется**; видимость — только §3 / П-N.
- [ ] Кнопки описаны событиями (Видимость / Активность / При нажатии), не
      отдельным разделом «Действия».
- [ ] frontmatter: `id/type/status/service/version` (+ `source`/
      `confluence_page_id` при reverse); имя файла — по конвенции комплекта
      либо `EV-`-префикс вне комплекта.
