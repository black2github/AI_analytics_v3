# План правок скилла перед калибровочным прогоном

Статус: **ВЫПОЛНЕН** (блоки 1–6 применены одной итерацией; политика
идентификаторов — принят вариант A).

Принятые решения по политике идентификаторов:
- **Вариант A подтверждён**: локальные ID — `FR-N` (правило поля), `EV-N`
  (событие), `EV-N.M` (шаг), `MSG-N` (сообщение); глобальные — по пилотной
  схеме конвенций (ENT/EXT/EXTINT/FUN/PRC/STM/CTL/SCR/NTF/PRN, SEC/NFR — при
  появлении типов). Принципы: только латиница; анти-омоглифы; ось
  классификации — не в ID; стабильность; область уникальности локальных —
  документ.
- **`EXT` остаётся задокументированным исключением** из анти-омоглифного
  правила (в платформе есть и справочники, и сущности — Dictionary по смыслу
  не всегда подходит).
- **`PROC` унифицирован**: процессы — `PRC-NNN` (по факту файлов); статусная
  модель — `STM-001` (диаграмма состояний по варианту A; `PRC-001` занят
  процессом ЖЦ). Правится в конвенциях и комплекте целевого репозитория.

Критерии аудита: чёткость/недвусмысленность, отсутствие дублирования (drift),
полнота, проверяемость — скилл будет исполняться агентами на более слабых LLM.

## 1. Критичные двусмысленности

1.1. **Markdown-нумерация vs стабильные ID шагов.** Явно указать: шаги — НЕ
     нумерованный markdown-список; последовательность абзацев/маркированных
     пунктов, начинающихся с `**<ID шага>.**`; второй (визуальной) нумерации нет.
1.2. **Гранулярность документа не зафиксирована.** Правило: один документ =
     одна вкладка/страница; экран с вкладками — каталог с README (суб-ID `.K`,
     как в пилотной схеме конвенций). Следствие: область уникальности локальных
     ID (правила/события/сообщения) — документ.
1.3. **Ссылочная политика reverse вне SRS-комплекта.** Развилка: в комплекте —
     ссылки на артефакты SRS + заглушки; вне комплекта (прогон рядом с
     исходником) — допустимы исходные ссылки, заглушки не создаются. Поправить
     инвариант 7 чек-листа (сейчас невыполним вне комплекта).
1.4. **Биекция «поле ↔ правило поля» vs оговорка о парных служебных полях.**
     Оговорку (один П-подраздел на пару однотипных служебных) повторить в
     инварианте 1 чек-листа — сейчас формальное противоречие.
1.5. **Кнопки и сообщения — не строки реестра полей.** Явный запрет в правилах
     §2 (высокий приор ошибки у моделей, видевших легаси-формат).

## 2. Дедупликация (схема «авторитет + указатель»)

2.1. Нотация перечислена трижды (SKILL.md, шапка templates.md, чек-лист) —
     оставить однострочные указатели на formatting.md SFR.
2.2. Семантика стабильных ID шагов изложена трижды — авторитет: templates §5;
     SKILL.md — одна строка; чек-лист — только проверка.
2.3. Заземление и инициализация параметров — авторитет: SKILL.md; в templates и
     чек-листе — указатели/форма проверки.
2.4. «Описание ЭФ» — один указатель на templates.md SFR + одна отличительная
     деталь (строка доп. действий ссылается на событие открытия).

## 3. Полнота

3.1. Правило формирования якорей внутренних ссылок (иначе инвариант 10
     непроверяем).
3.2. Место «Истории изменений» в скелете (reverse).
3.3. Ограничение «Сути реакции» в реестре событий: одна строка, без ветвлений.
3.4. Допустимые значения роли в id (`CL`/`BNK`) — указать или сослаться на
     конвенции комплекта.
3.5. Образец условного присваивания в правиле поля («= true, **если** …;
     **иначе** false»).

## 4. Проверяемость

4.1. Инвариант достижимости — самодостаточная формулировка: «каждый шаг, кроме
     первого в событии, упомянут хотя бы в одном переходе».
4.2. Маркеры `(reverse)`/`(forward)` в разделе «Сообщить пользователю».

## 5. Терминология артефактов (по вопросам пользователя)

Скилл использует префиксы EXTINT/ENT/EXT/FUN/PRC/CTL **без расшифровки** —
слабая LLM (и человек) не обязаны их знать. Скилл живёт в другом репозитории и
не может ссылаться на `_meta/conventions.md` целевого репо. Добавить в SKILL.md
краткий словарь артефактов комплекта (одна строка на префикс: EXTINT — карточка
внешней интеграции; ENT — сущность модели данных; EXT — платформенный
справочник/сущность; FUN — функция; PRC — процесс; CTL/CTL-GRP — контроль/группа;
SCR — экранная форма) с оговоркой «канонический реестр — в конвенциях целевого
репозитория документации».

## 6. Политика идентификаторов (ждёт решения пользователя)

Проблема: локальные ID событийного формата — кириллица (П-N, Е-N, М-N),
глобальные — латиница; смешение алфавитов + омоглифы (Е/E, М/M, Р/P, С/C…).
Решение: только латиница; предложены варианты (см. обсуждение); после выбора —
заменить во всех трёх файлах скилла: правила полей, события, шаги, сообщения,
исходы; обновить примеры и инварианты.
