# Категории характеристик поля: A / C / D, видимость, логика заполнения

Ось — **природа характеристики**, не место записи в исходнике.

## Триада «может ли провалиться»

| Категория | Что это | Где живёт | Может провалиться? |
|---|---|---|:---:|
| **A — Формат** | какие значения допустимы | колонка «Формат» (из типа или на поле) | да → сообщение об ошибке |
| **C — Контроль** | отношение полей / контекст | FE-документ контролей (`SF-NN`) | да → сообщение об ошибке |
| **D — Представление** | как отображается | колонка «Представление» | **нет** — чистый рендеринг |

Плюс две характеристики вне триады: **видимость** (см. ниже) и **логика
заполнения** (откуда значение).

## A — intrinsic-ограничение (Формат)

Свойство самого элемента данных, истинно везде, где он встречается: формат,
длина, паттерн, диапазон, набор литералов inline, обязательность значения.

Источник формата — пометка в колонке:

- `(из типа)` — поле один-к-одному соответствует атрибуту модели данных; формат
  наследуется из словаря типов (задан один раз, рассинхрон невозможен).
- `(на поле)` — поле фронтовое, с атрибутом расходится (адрес разбит на
  индекс/город/улицу против единого атрибута «Адрес»); формат задаётся на самом
  поле, общего типа нет.

Ловушка (унаследована из control-split): «значение из справочника» — это **не**
литеральный набор. Литеральный enum прямо в условии («М»/«Ж») → A; lookup во
внешний справочник → категория C (нужен контекст за пределами значения).

## C — relational-правило (Контроль)

Проверке нужен контекст за пределами самого значения поля: межполевое правило
(«дневной лимит ≤ месячный»), сверка со справочником, с другой сущностью, с
внешней системой, статус, полномочия. Выносится в FE-документ контролей с `SF-NN`,
условием и сообщением об ошибке.

## D — Представление

Как значение/опция отображается: шаблон опции списка
(`<номер счёта> — <валюта>, <остаток>, <тип счёта>`), маска ввода
(`+7 (___) ___-__-__`), разделители разрядов. Свойство категории: **не проверяет и
не ограничивает, провалиться не может, ни от каких полей не зависит**. Если у
характеристики есть условие и участники — это не D.

Слово «формат» в обиходе покрывает три смысла — разводи их: формат-ограничение
(A), маска ввода (D), шаблон отображения (D).

**Значение ≠ представление.** В поле-выборе значением является номер/идентификатор;
обогащение опции (валюта, остаток, тип счёта) — проекция атрибутов сущности для
чтения, категория D.

## Видимость: тривиальная в D, нетривиальная в контроли типом `visibility`

Условие видимости («поле скрыто, если тип счёта ≠ 2») — **правило**: у него есть
условие, участники, момент срабатывания. От контроля отличается только исходом:
не проваливается с сообщением, а молча меняет состав формы. Поэтому:

- **тривиальная** видимость (статичная или однострочное условие без ветвлений) —
  текстом в колонке «Представление»;
- **нетривиальная** (зависит от состояния других полей, многошаговая, с
  ветвлениями) — в FE-документ контролей строкой с типом **`visibility`**:
  `ID | Ориг. | Тип=visibility | Поля | Условие показа/скрытия | — (без сообщения) | Гр.`

Держи категорию D чистой: нетривиальная видимость в «Представлении» разрушает
свойство «можно не проверять».

## Логика заполнения

Отвечает на вопрос: **откуда у поля значение до и помимо ввода пользователя**.
Сюда идут:

- значение по умолчанию;
- **источник списка** для поля-выбора (какие опции предлагаются, включая
  фильтрацию: «счета клиента, валюта = валюте карты, не заблокированные»);
- вычисляемые/производные значения;
- наполнение из данных, возвращаемых функциями (включая разбор списков) —
  **семантика этой связи отложена**: помечай `уточняется`, нотацию не выдумывай.

**Источник списка — не контроль**: он ограничивает *предлагаемое*, провалиться не
может. Но фронтовая фильтрация обходима, поэтому у правила источника обязан
существовать **бэковый близнец** — data-model контроль («счёт принадлежит клиенту
и валюта совпадает»). Пометь его необходимость; сам близнец живёт в BE-документе
(зона ответственности control-split).

## Границы применимости

- A «из типа» — только при совпадении поля с атрибутом; иначе — «на поле».
- Наполнение данными (что возвращает навигационная функция, как операционные
  функции поставляют значения) — отложенный вопрос, скилл его не решает.
- Решение о задвоении форматов фронт/бэк через общий тип — отдельная развилка,
  не предмет этого скилла.
