# Переформатирование экранной формы (API First / Doc as Code) — v3.0

> **Модель v3.0.** Контроли — спецификации централизованных проверок (поле ·
> название · условие Если-то-иначе · событие), с режимами `R`/`V` и механическим
> критерием выноса видимости (`behavior` и реакции удалены; см.
> references/controls.md). Форма: колонки `Поле | Тип | Формат | Способ заполнения |
> Обяз. | Видимость | Логика установки значения`. **Формат** вобрал отображение
> (маска/шаблон/пример/подсказка), **Видимость** — показ/скрытие **наименования и
> значения** поля, **Логика установки значения** — откуда значение (формула с `=`).
>
> **Верховный принцип: СОХРАНЕНИЕ СМЫСЛА — высший приоритет.** При потере смысла
> всё остальное (формат, чистота колонок, читаемость) не имеет значения. Это
> **семантическое преобразование через понимание**, а не парсинг: пойми, что несёт
> исходный текст, и вырази это в целевой структуре как **ПОЛНЫЙ смысловой
> эквивалент** — со всеми условиями и ограничениями (именно они задают клиентский
> путь). Переформулировать текст **можно**; терять или огрублять смысл — **нельзя**.
> «Приблизительно» недопустимо. Один фрагмент исходника может нести несколько
> смыслов сразу (напр. «значение по умолчанию» = и способ получения значения → в
> «Логику установки значения», и блокировка = автоустановка → в «Способ
> заполнения») — разложи все, ничего не отбросив.
>
> **Если не можешь передать смысл без потери или не уверен в эквивалентности —
> НЕ СГЛАЖИВАЙ, а пометь** `[НЕ РАСПОЗНАНО: <исходный текст фрагмента дословно>]`.
> Тихая потеря — худшее; явная пометка — приемлемо (человек/агент доразберёт).
>
> Дополнительно: новая форма — основной документ; при переносе сохраняются все
> несущие смысл элементы (иконки `<img>`, ссылки, списки, вложенность условий,
> токены `<Имя атрибута>` — references/formatting.md). Раздел «Описание ЭФ» —
> дословно.

## Что делает скилл

На входе — страница экранной формы (экспорт Confluence или markdown): описание
полей с форматами, логикой показа, умолчаниями, часто вперемешку. Опционально
рядом — страница контролей этой формы.

На выходе — согласованная **пара** артефактов из **одного** разбора:

1. **Переформатированная ЭФ** — единая таблица полей с колонками
   `Поле | Тип | Формат | Способ заполнения | Обяз. | Видимость | Логика установки значения`
   (см. references/field-table.md).
2. **FE-документ контролей** (`control_kind: screen-form`) — relational-правила
   (`rule`) и вынесенная по механическому критерию видимость (`visibility`), с
   режимами `R`/`V` и привязкой к группам триггеров (см. references/controls.md).
   Совместим по формату с выходом скилла control-split.

При необходимости — список кандидатов в форматы значений (опционально; см.
references/type-dictionary.md). Колонка «Тип» формы — UI-типы, не форматы.

## Ключевой принцип: выделение проверок = переформатирование формы

Чтобы решить, что остаётся у поля (формат, способ заполнения, тривиальная
видимость, логика установки значения), а что выносится в контроли (relational-правила,
нетривиальная видимость), нужно разобрать каждую строку описания поля. Этот разбор
— и есть основная работа. Делать её дважды (сначала контроли, потом форма) — платить
дважды за самое дорогое (интерпретацию неструктурированного текста). Поэтому форма
и её контроли рождаются **согласованной парой из одного разбора**.

## Что уходит из формы и контролей: реакции

Реакции — подсветка поля, показ сообщения об ошибке, очистка/пересчёт полей,
блокировка, модальные окна — **не контроль** и **не колонка формы**. Контроль лишь
проверяет условие и формирует код/сообщение; что с ним делать — решает вызывающая
сторона. Единые правила информирования об ошибках ведутся отдельно (для КК —
`[КК] Правила информирования пользователя Клиента об ошибках`); чего не хватает —
дописывается туда. Исключение — **недоступность для редактирования**: это часть
«Способа заполнения» (= «Автоматически»), см. ниже.

## Процедура

### Шаг 1. Разобрать исходную страницу ЭФ

Извлеки: frontmatter (наследуется производными), историю изменений, список полей
со всеми характеристиками (тип, формат, маска, видимость, способ заполнения,
умолчание, источник списка, подсказки), действия (кнопки), встроенные проверки.
Технические правила разбора HTML (треугольные скобки, ссылки, rowspan, запрет
обрезки) — references/parsing.md.

### Шаг 2. Классифицировать каждую характеристику

По **references/categories.md** и **references/controls.md**:

- **Тип поля** — из словаря UI-типов ЭФ (references/type-dictionary.md): Текст,
  Число, Дата, Список, Multi-select, Переключатель, Чекбокс, Пароль, Поиск и т.д.
- **Формат** — ограничение **и формат отображения** значения: длина/паттерн/набор,
  маска, шаблон опции/строки, пример, плейсхолдер, подсказка → колонка «Формат»,
  пометка `(из типа)` / `(на поле)`. Атрибут-источник значения сюда **не** пишется.
- **Способ заполнения** — {Вручную | Выбор из списка | Автоматически}. «Автоматически»
  = недоступность для ручного ввода. Условную недоступность пиши здесь:
  «Автоматически, если <условие>; иначе Вручную».
- **Видимость** — показ/скрытие **лейбла и значения** (по умолчанию лейбл виден;
  условие без оговорки — для обоих; «Наименование поля не отображается» — явно).
  Тривиальная (по критерию) → колонка «Видимость» как Если-то-иначе, без маркера
  `R`. Нетривиальная (ветвление ∨ разделяют несколько полей ∨ внешнее состояние —
  references/controls.md) → контроль `visibility`, в форме указатель `см. SF-NN`.
  Размерность экрана — среда рендеринга, у поля. **Формул значения в «Видимость»
  не пиши.** См. references/categories.md, «Наименование и значение».
- **Логика установки значения** — как поле **получает значение**: умолчание (и логика
  расчёта), источник списка (фильтр), вычисляемая формула (со знаком `=`),
  автозаполнение из атрибута; `уточняется` — отложенная связь с функциями.
- **Контроль (rule)** — relational-правило (межполевое, контекстное, против
  сущности/справочника) → FE-документ контролей.
- **Разделители** (частые ошибки): опция-склейка → «Формат», опция-выбор-по-условию
  → «Логика установки значения»; пример/маска/подсказка → «Формат»; атрибут-источник
  и вычисляемый заголовок → «Логика установки значения».
- **Реакции** и **необходимость BE-близнеца** — не в колонки формы (реакции — в
  документе сервиса; BE-близнец — в FE-контролях).
- **Полный смысловой эквивалент (высший приоритет):** каждое условие/ограничение
  исходной ячейки должно присутствовать в результате как эквивалент (переформулировка
  допустима, огрубление/потеря — нет). Один фрагмент может нести несколько смыслов —
  разложи все по колонкам. Сохраняй и несущие смысл элементы (иконки `<img>`, ссылки,
  списки, вложенность, токены `<Имя атрибута>` — references/formatting.md).
- **Пометка при неуверенности:** если не можешь передать смысл без потери, разложить
  однозначно или не уверен в эквивалентности — **не сглаживай**, пометь
  `[НЕ РАСПОЗНАНО: <исходный текст дословно>]`. Тихая потеря недопустима.

### Шаг 3. Оформить контроли (references/controls.md)

Каждой строке: `ID` (`SF-01…` сквозной), **Название проверки** (суть/цель, не
пересказ условия), `Тип` (`rule`/`visibility`), «Поля» (для `visibility` —
`Объект ← зависимости`), «Условие» (Если-то-иначе, дословно, без реакций и логики
получения значения), «Сообщение об ошибке» (`—` для `visibility`), режим `R`
(отдельная колонка) и отметки `V` в столбцах групп триггеров. Группы объяви над
таблицей; при миграции — переиспользуй исходную нумерацию групп. `Ориг.` — исходный
№ для прослеживаемости.

### Шаг 4. Собрать пару артефактов

По **references/templates.md** и **references/naming.md** — создай **новую пару
SFR-файлов** рядом с исходником; исходный файл **не перезаписывай и не удаляй**.

- ЭФ: frontmatter → описание формы → **раздел «Описание ЭФ»** (обязательная таблица
  из исходника, переносится как есть) → единая таблица полей (порядок:
  `Поле | Тип | Формат | Способ заполнения | Обяз. | Видимость | Логика установки значения`)
  → пояснения к нетривиальным строкам → «Действия» → ссылка на FE-контроли.
  Форматирование текста в ячейках — по references/formatting.md.
- FE-контроли: врезка Зоны 3, «Назначение» со ссылкой на **экранную форму**,
  «Группы триггеров», таблица контролей (references/controls.md).

Именование, `title`/`doc_id`, `related`, каталог — references/naming.md.

### Шаг 5. Смысловая самопроверка + финальная сверка

**Смысловая самопроверка (главное, не механическая).** По каждому полю перечитай
исходную ячейку и результат и ответь себе: *присутствует ли в результате каждое
условие, ограничение и умолчание оригинала как смысловой эквивалент? не появилось
ли «приблизительно»?* Любой фрагмент, чью эквивалентность не можешь подтвердить, —
верни в оригинал и пометь `[НЕ РАСПОЗНАНО: …]`, а не оставляй огрублённым. Это
проверка смысла человеком-читателем, а НЕ сравнение строк.

Затем — формальная сверка по **references/checklist.md**.

## Важные принципы

- **Реструктуризация — семантическое дополнение, а не переписывание с потерей.**
  Новая форма обязана полностью воспроизводить клиентский путь оригинала. Иконки,
  ссылки, списки, вложенность условий, токены `<Имя атрибута>` — сохраняются
  (references/formatting.md). Нераспознанное — `[НЕ РАСПОЗНАНО: …]`, не выбрасывать.
- **Условия и тексты — дословно** (нормализация ссылок/`<…>`/пробелов допустима;
  усечение `[:N]`/«…» — нет).
- **Контроль не описывает реакцию и не описывает получение значения** — только
  проверку условия и код/сообщение при провале (references/controls.md).
- **Видимость выносится в контроли только по механическому критерию.** Размерность
  экрана — среда рендеринга (у поля).
- **Источник списка — не контроль** (ограничивает предлагаемое) → «Логика установки
  значения»; но нужен BE-близнец (пометь необходимость в FE-контролях; создаёт
  control-split).
- **Значение поля ≠ представление опции**: склейка-опция → «Формат», выбор-по-условию
  → «Логика установки значения».
- **A из типа — только при совпадении поля с атрибутом**; иначе `(на поле)`.
- **Не решай отложенное** (наполнение данными функций — `уточняется`).
- **Не менять формат требований без необходимости** — узнаваемость для аналитиков.

## Разделение труда со скиллом control-split

- Вход — страница **контролей** → control-split (BE/FE-разнесение, x-controls).
- Вход — страница **экранной формы** → этот скилл (структура полей, контроли формы,
  видимость, R/V).
- Вход — пара (ЭФ + контроли) → этот скилл ведёт разбор формы; выделенные правила
  согласуются с FE-документом control-split (единые SF-ID, единый формат).

## Справочные файлы

- `references/controls.md` — **модель контроля**: состав строки, запрет
  реакций/получения значения, `rule`/`visibility`, режимы `R`/`V`, группы
  триггеров, механический критерий видимости, формат таблицы.
- `references/categories.md` — оси характеристик (Формат / Контроль / Видимость /
  Способ заполнения / Логика установки значения); правила-различители; куда идут
  реакции.
- `references/field-table.md` — структура таблицы полей, порядок колонок, примеры.
- `references/formatting.md` — форматирование текста в ячейках: `<br>`-разделение,
  жирные «Если/то/иначе», блоки «По умолчанию:»/«Источник списка:», «Шаг N».
- `references/type-dictionary.md` — словарь UI-типов полей ЭФ.
- `references/templates.md` — шаблоны ЭФ и FE-контролей.
- `references/parsing.md` — технические правила разбора (общие с control-split).
- `references/naming.md` — именование, `title`/`doc_id`, `related`, каталог.
- `references/checklist.md` — финальная сверка.
