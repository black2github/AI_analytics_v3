# Переформатирование экранной формы (API First / Doc as Code)

## Что делает скилл

На входе — страница экранной формы (экспорт Confluence или markdown): описание
полей с форматами, логикой показа, умолчаниями, часто вперемешку и без единой
структуры. Опционально рядом — страница контролей этой формы.

На выходе — согласованная тройка артефактов из **одного** разбора:

1. **Переформатированная ЭФ** — единая таблица полей с колонками
   `Поле | Тип | Формат | Представление | Обяз. | Логика заполнения`
   (см. references/field-table.md).
2. **FE-документ контролей** (`control_kind: screen-form`) — relational-правила
   (категория C) и **нетривиальная видимость** (тип `visibility`). Совместим по
   формату с выходом скилла control-split.
3. **Кандидаты в словарь типов** — замеченные повторяющиеся форматы (ИНН, БИК,
   номер счёта…) отдельным списком-предложением (см.
   references/type-dictionary.md). Словарь не создаётся самовольно — только
   кандидаты на решение аналитика.

## Ключевой принцип: выделение проверок = переформатирование формы

Чтобы решить, что выносится в контроли (C), что остаётся у поля (A), что является
представлением (D), нужно разобрать каждую строку описания поля. Этот разбор — и
есть основная работа переформатирования. Делать её дважды (сначала выделение
контролей, потом миграция формы) — платить дважды за самое дорогое
(интерпретацию неструктурированного текста) и получить незафиксированное
промежуточное состояние. Поэтому форма и её контроли рождаются **согласованной
парой из одного разбора**.

## Процедура

### Шаг 1. Разобрать исходную страницу ЭФ

Извлеки: frontmatter (наследуется производными), историю изменений, список полей
со всеми характеристиками (формат, маска, видимость, умолчание, источник списка,
подсказки), действия (кнопки), встроенные проверки. Технические правила разбора
HTML (треугольные скобки, ссылки, rowspan, запрет обрезки) — те же, что в
control-split: см. references/parsing.md.

### Шаг 2. Классифицировать каждую характеристику по категориям

Для каждого фрагмента описания поля определи категорию по
**references/categories.md**. Кратко:

- **A (Формат)** — intrinsic-ограничение значения: формат, длина, паттерн,
  обязательность, набор литералов. Идёт в колонку «Формат». Пометь источник:
  `(из типа)` — если поле совпадает с атрибутом и формат наследуем из словаря
  типов; `(на поле)` — если поле фронтовое и с атрибутом расходится (случай
  адреса: индекс/город/улица против единого атрибута «Адрес»).
- **C (Контроль)** — relational-правило: межполевое, контекстное, против внешней
  сущности. Идёт в FE-документ контролей (`SF-NN`).
- **D (Представление)** — как отображается: шаблон опции, маска ввода,
  разделители. Не проверка, провалиться не может. Колонка «Представление».
- **Видимость**: тривиальная (статичная или однострочная) — текстом в
  «Представлении»; **нетривиальная** (зависит от состояния других полей,
  многошаговая) — в FE-контроли с типом `visibility` (без сообщения об ошибке —
  она не проваливается, а скрывает/показывает).
- **Логика заполнения** — откуда у поля значение помимо ввода: умолчание,
  источник списка, вычисление, наполнение из данных функций. Колонка «Логика
  заполнения». Семантика связи с операционными функциями — **отложенный вопрос**:
  колонка резервируется и заполняется тем, что уже ясно; неясное помечается
  `уточняется`, не блокируя переформатирование.

### Шаг 3. Присвоить ID контролям

FE-контроли: `SF-01, SF-02, …` сквозной нумерацией; для видимости — тип
`visibility` в колонке «Тип». Если у поля был исходный № или якорь — сохрани в
колонке «Ориг.» для прослеживаемости.

### Шаг 4. Собрать тройку артефактов

По шаблонам **references/templates.md**:

- ЭФ: frontmatter → описание формы → единая таблица полей (порядок колонок:
  `Поле | Тип | Формат | Представление | Обяз. | Логика заполнения`) →
  пояснения к нетривиальным строкам → «Действия» → ссылка на FE-контроли.
- FE-контроли: формат control-split (frontmatter, врезка Зоны 3, «Назначение» со
  ссылкой на **экранную форму**, «Группы триггеров», таблица с колонкой «Тип»
  (`rule` / `visibility`)).
- Кандидаты в словарь типов: отдельный список-артефакт.

Именование файлов, `title`/`doc_id`, `related`, каталог размещения — по правилам
control-split (references/naming.md): минимальная правка оригинального имени, без
пробелов, `FE-Контроли` после префикса, выходные файлы в каталог оригинала.

### Шаг 5. Финальная сверка

По **references/checklist.md**. Ключевое: ни одна характеристика исходника не
потеряна (каждый формат/умолчание/видимость нашли свою колонку или контроль);
текст перенесён дословно; категории A/C/D не смешаны.

## Важные принципы

- **Условия и тексты переноси дословно** — без перефразирования, усечения
  (`[:N]`) и «…». Допустима только техническая нормализация (ссылки → markdown,
  восстановление `<…>`, схлопывание пробелов). Сжатие — ручной шаг аналитика.
- **D — категория со свойством «провалиться не может».** Если у характеристики
  есть условие, участники и исход «не пройдено» — это не D. Не клади нетривиальную
  видимость в «Представление»: она разрушит чистоту категории.
- **Источник списка — не контроль.** Он ограничивает *предлагаемое*, а не
  проверяет *введённое* (провалиться не может) → колонка «Логика заполнения».
  Но у него обязан существовать **бэковый близнец** — data-model контроль
  (фронтовая фильтрация обходима). Близнеца создаёт/сверяет control-split по
  своей политике; здесь — пометь необходимость.
- **Значение поля ≠ представление опции.** В поле-выборе значением является
  идентификатор/номер; обогащение опции (валюта, остаток, тип) — представление
  (D), проекция атрибутов сущности для чтения.
- **A из типа — только при совпадении поля с атрибутом.** Где фронт дробит или
  склеивает атрибут (адрес), общего типа нет — формат задаётся на поле.
- **Не решай отложенное.** Механика наполнения данными (связь с операционными
  функциями, что возвращает навигационная функция) — вне рамок скилла. Колонка
  «Логика заполнения» резервирует место; помечай `уточняется`, не выдумывай
  нотацию.
- **Не менять формат требований без необходимости** — производные документы
  должны быть узнаваемы аналитиками (история изменений, привычные разделы).

## Разделение труда со скиллом control-split

- Вход — страница **контролей** → control-split (BE/FE-разнесение, многосущностные
  агрегаты, x-controls).
- Вход — страница **экранной формы** → этот скилл (структура полей, A/C/D,
  visibility, кандидаты типов).
- Вход — пара (ЭФ + контроли) → этот скилл ведёт разбор формы, а выделенные
  C-правила согласуются с FE-документом control-split (единые SF-ID, единый
  формат таблицы контролей); политика мирроринга внешних сущностей — по
  control-split.

## Справочные файлы

- `references/categories.md` — категории A/C/D, видимость (`visibility`), логика
  заполнения, источник списка; границы и триада «может ли провалиться».
- `references/field-table.md` — структура единой таблицы полей, порядок колонок,
  правила заполнения, примеры строк всех случаев.
- `references/type-dictionary.md` — как выделять кандидатов в словарь типов.
- `references/templates.md` — шаблоны переформатированной ЭФ и FE-контролей.
- `references/parsing.md` — технические правила разбора (общие с control-split).
- `references/naming.md` — именование файлов, `title`/`doc_id`, `related`,
  каталог (общие с control-split).
- `references/checklist.md` — финальная сверка.
