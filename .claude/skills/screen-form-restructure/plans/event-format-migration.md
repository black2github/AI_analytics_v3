# План перевода скилла SFR на событийный формат описания ЭФ

Статус: **предварительный, не в работе**. Эталон целевого формата —
`AI-docs-O2new/output/open-add-account/docs/srs/screen-forms/client/example-scr-cl-03.1-account.variant-7-events-detailed.md`
(вариант 7: событийная модель с детализацией вызовов).

## Суть перехода

От «всё в одной таблице» (7 колонок: Поле | Тип | Формат | Способ заполнения |
Обяз. | Видимость | Логика установки значения; динамика в TRG + «Действия» +
процедуры; сообщения — строки таблицы) — к разложению по осям в разделы:

1. **Назначение** — цель формы, ссылки на сущность/экран/функции открытия.
2. **Реестр полей** — только `№ | Блок | Поле | Тип | Обяз.`.
3. **Видимость блоков** — список условий по блокам.
4. **Правила полей (статика)** — подраздел **П-N** на поле; ярлыки-буллеты:
   Значения / Отображение значения / Предзаполнение / Редактируемость /
   Доступность / Формат / Сортировка / Поиск / Подсказка / Видимость / Особый случай.
5. **События** — реестр **Е-N** (ID | Триггер | Суть реакции) + детальные
   подразделы: нумерованные шаги, **Если/то/иначе**, переиспользование диапазонов
   шагов («шаги 7–23 события Е-1»), именованные исходы («Скрытие полей»),
   вызовы EXTINT с инициализацией параметров. Кнопки — тоже события
   (Видимость / Активность / При нажатии).
6. **Сообщения** — реестр **М-N**: ID | Тип | Показ (обратная ссылка на шаг
   события / правило поля) | Содержание.

Frontmatter упрощается: `id / type / status / service / version`.

## Изменения по файлам скилла

1. `references/templates.md` — новый скелет документа (6 разделов выше);
   раздел «Описание ЭФ» (дословная таблица) упраздняется — судьба «Макетов» и
   «Как перейти» решается отдельно (см. вопрос 3).
2. `references/field-table.md` → «field-registry + П-правила»: реестр из 5
   колонок; словарь ярлыков-буллетов. Колонка «Способ заполнения» расщепляется
   на Предзаполнение / Редактируемость / Доступность.
3. Раздел «События» заменяет «Триггеры»+«Действия»+процедуры: правила реестра
   Е-N, шаги, переиспользование, именованные исходы. Принцип «вызовы с
   инициализацией входных параметров» переносится без изменений.
4. Раздел «Сообщения»: правила извлечения М-N, колонка «Показ» с обратной
   ссылкой (замена «сообщения — строки таблицы полей»).
5. `references/controls.md` — смена роли: SFR не генерирует парный FE-документ;
   обязательность/правила — ссылками на CTL-NN / CTL-GRP существующего комплекта
   контролей; механический критерий выноса видимости упраздняется — видимость
   всегда инлайн (§3 / П-N).
6. Остаётся без изменений: двухпроходный разбор 2a/2b (меняются только целевые
   «оси» — разделы вместо колонок), верховный принцип смыслового эквивалента,
   `[НЕ РАСПОЗНАНО: …]`, formatting.md (жирные Если/то/иначе, `=`/`==`, `→`
   только «найти/взять запись»), инициализация параметров вызовов.

## Открытые вопросы (решить до начала правок)

1. **Судьба FE-контролей и модели SF-NN/R/V.** Целевой формат ссылается на
   готовый комплект CTL. Кто его создаёт: control-split остаётся отдельным шагом,
   SFR только ссылается? Что делать с уже введённой моделью SF-NN (R/V, группы)?
2. **Зависимость от SRS-комплекта.** Формат требует резолвинга ссылок в
   ENT/EXT/EXTINT/FUN/PRC/CTL. Для прогона по «голой» странице Confluence нужна
   политика заглушек (в эталоне намечена: «карточка-заглушка, страница метода
   вне выгрузки»).
3. **«Макеты ЭФ» и «Как перейти на ЭФ»** из нынешнего обязательного раздела
   «Описание ЭФ»: переход частично покрыт «Назначением», макетов в эталоне нет.

## Известные риски формата (учесть при правке)

- Хрупкость ссылок-диапазонов «шаги 7–23 события Е-1»: вставка шага сдвигает
  нумерацию и молча ломает все диапазоны. Кандидаты-решения: стабильные ID шагов
  (`Е-1.9`), запрет переиспользования диапазонами в пользу именованных
  подпроцедур, линтер диапазонов.
- Информация об одном поле разнесена по §2/§3/§4/§5/§6 — нужна дисциплина
  перекрёстных ссылок (П-N ↔ Е-N ↔ М-N) и проверка полноты в чек-листе
  (каждое поле реестра имеет П-N; каждое М-N имеет «Показ»; каждый Е-N из
  реестра имеет подраздел).

## Forward-режим (агент проектирует новую ЭФ с нуля — человек ревьюит)

Скилл сегодня заточен под reverse (дословный перенос исходника). Целевой формат
должен работать и в forward-режиме: новый сервис или новый клиентский путь,
исходника нет, агент содержимое проектирует (из процесса, модели данных, функций,
аналогов соседних форм), аналитик только ревьюит. В этом режиме структура
документа — не контейнер, а **методика проектирования**: порядок разделов — маршрут
(назначение → реестр → видимость → П-N → Е-N → М-N), обязательный Е-1 «Открытие
формы» вынуждает спроектировать самое сложное поведение, а не пропустить его.

### Правило заземления (анти-переинженеринг, анти-галлюцинации)

Каждый элемент спроектированного описания обязан иметь **основание-источник**:

- **шаг события** — ссылка на требование-источник: шаг процесса (PRC), функцию
  (FUN), контроль (CTL-NN / CTL-GRP), правило сущности (ENT) — либо явная пометка
  `уточняется` с формулировкой открытого вопроса. Шаг без основания и без пометки —
  дефект генерации, а не «творчество»;
- **вызов внешнего метода** — ссылка на карточку EXTINT; если карточки нет —
  создаётся **заглушка** (по политике эталона: «карточка-заглушка, страница метода
  вне выгрузки»), выдуманный метод без карточки запрещён;
- **параметры вызова** — по действующему принципу «вызовы с инициализацией
  входных параметров» (состав/обязательность — из описания функции и OpenAPI;
  источник значения каждого параметра — поле/атрибут/шаг, иначе `уточняется`);
- **сообщение М-N** — колонка «Показ» заполнена (шаг Е-N или правило П-N);
  сообщение, которое ниоткуда не показывается, — дефект;
- открытые вопросы — не inline-TODO, а по конвенциям репозитория документации
  (open-questions вне docs/), в документе — только пометка `уточняется`.

Для ревьюера правило симметрично: «шаг без основания — вопрос к агенту».

### Инварианты предпроверки (агент прогоняет ДО передачи человеку)

Механические проверки структуры; человеку остаётся только смысл:

1. Каждое поле §2 «Реестр полей» имеет подраздел П-N; каждый П-N — строку в §2
   (биекция, включая служебные поля).
2. Каждый блок из §2 упомянут в §3 «Видимость блоков».
3. Каждое событие реестра §5 имеет детальный подраздел; каждый подраздел — строку
   в реестре (биекция).
4. Каждый шаг каждого события **достижим** (на него есть переход или он
   последовательный) и **завершим** (ветви заканчиваются шагом-переходом,
   «конец» или именованным исходом).
5. Все ссылки на шаги (в т.ч. диапазоны «шаги N–M события Е-K» и колонка «Показ»
   в М-N) резолвятся в существующие шаги. (До решения о стабильных ID шагов —
   это главная проверка против churn нумерации.)
6. Каждое М-N имеет «Показ»; каждое сообщение, упомянутое в шагах/правилах,
   есть в реестре §6 (биекция).
7. Каждый вызов внешнего метода ссылается на существующую карточку EXTINT
   (или заглушку); каждый вызов — с инициализацией входных параметров.
8. Каждый упомянутый контроль/группа резолвится в CTL-NN / CTL-GRP.
9. Именованные исходы: каждый использованный — определён, каждый определённый —
   использован.
10. Ссылки на П-N из шагов и на Е-N из правил — резолвятся (перекрёстная
    целостность §4 ↔ §5 ↔ §6).
11. Нотация: `→` только «найти/взять запись», `==` в сравнениях, жирные
    Если/то/иначе, `[НЕ РАСПОЗНАНО: …]` — только в reverse-режиме (в forward
    вместо него `уточняется` + открытый вопрос).

Отчёт предпроверки прикладывается к передаче на ревью; ревью делимо по ролям:
статика П-N против модели данных / события против процесса / тексты М-N — UX и
комплаенс.

### Специфика forward vs reverse в скилле

- «Дословность» и `[НЕ РАСПОЗНАНО]` — только reverse; в forward их место занимают
  правило заземления и `уточняется`.
- Риск нумерации шагов в forward **блокирующий** (шаги переставляются в каждом
  черновике) — стабильные ID шагов (`Е-1.9`) или линтер диапазонов нужны до
  первого greenfield-прогона.
- Новый путь в существующем сервисе проектируется как **дельта**: новые события +
  ссылки на существующие П-N/исходы/EXTINT/CTL соседних форм; соседние формы
  сервиса — образцы структуры.

## Оценка

Перекладка, а не переосмысление: инвентаризация фрагментов 2a остаётся, меняется
адрес назначения фрагмента (колонка → раздел/ярлык). Переписать ~60–70%
справочного материала (templates, field-table, controls, куски SKILL.md и
чек-листа). Без изменений: formatting.md, parsing.md, принципы смысла и
параметров вызовов; type-dictionary.md — частично. Плюс один калибровочный
прогон (пересборка SFR-2 в новом формате) для проверки воспроизводимости.
Forward-режим добавляет: раздел «правило заземления» в SKILL.md и инварианты
предпроверки в чек-лист (сами инварианты полезны и в reverse — как проверка
структурной целостности результата).
