# Архитектура межсервисных ссылок Doc as Code — решения PoC

## Контекст

Система ДБО для юридических лиц состоит из ~100 микросервисов (бизнес-сервисы и
платформенные/системные). Каждый сервис ведёт одна команда (2–3 аналитика).
Требования сейчас в Confluence, мигрируются в Markdown (Doc as Code).
Параллельно идёт переход на API First (генерация OpenAPI-спецификаций) и активное
использование LLM, развёрнутой в банке (стоимость токенов в целевом процессе
не важна).

Ключевая проблема: в Confluence наименование = имя страницы = адрес (тройная связь),
что давало удобный wiki-механизм ссылок. При переходе на несколько Git-репозиториев
эта связь распадается и её нужно восстановить другими средствами — особенно для
ссылок на структуры описания требований (элементы) **других** сервисов.

Объём межсервисных ссылок высокий: платформенные справочники используются повсеместно.

---

## Решение 1. Нотация ссылок

### Внутренние ссылки (на свой сервис)
Обычная Markdown reference-ссылка на локальный файл — кликается нативно:

```markdown
Передаём [Счёт].<номер счёта> в запрос.

[Счёт]: ../data-model/Счёт.md
```

### Внешние ссылки (на чужой сервис) — фигурные скобки
смарт-ссылка с двойными фигурными скобками:

```markdown
{{сервис:Имя элемента}}
```

**Почему фигурные, а не квадратные:** квадратные скобки уже заняты — по правилам
оформления аналитики имя логического элемента пишется как `[Имя]`, плюс Markdown
использует `[...]` для ссылок. Фигурные скобки свободны, не трактуются Markdown,
визуально читаются как плейсхолдер-подстановка (конвенция Jinja/Handlebars/Vue).

Ключ резолва — полное имя элемента, как его пишет аналитик, включая код сервиса (например 'cc', префикс и
квадратные скобки имени:

```markdown
{{сс:[КК_ВК] Заявка}}
```

Ссылку физически вставляет **агент**, не аналитик руками — поэтому громоздкость
спецсимволов внутри `{{...}}` не является проблемой. Аналитик описывает намерение,
агент подбирает точный элемент (через подсказку RAG) и вставляет точную ссылку.

---

## Решение 2. Развилка «данные vs действие» (на что ведёт ссылка)

Разрешение ссылки зависит **не от типа сервиса**, а от природы того, на что
показывает ссылка:

| Природа ссылки                      | Цель резолва | Механизм                   |
|-------------------------------------|---|----------------------------|
| ДАННЫЕ (сущность, ЭФ...)           | публичная карточка сущности | `kind: card`       |
| ДЕЙСТВИЕ (вызов метода, интеграция) | swagger-операция | `kind: swagger` |

Даже внутри одного функционального требования могут встретиться обе: ссылка на вызов
метода ведёт в swagger, ссылка на атрибут данных — в карточку.

### Различение по синтаксису
Различение «данные/действие» в большинстве случаев определяется синтаксисом
(см. «Правила оформления аналитики ДБО»):

- `[Сущность].<атрибут>` и конструкции с операцией `→` (найти/взять запись) — ДАННЫЕ.
- `[Функция].<Исходящие параметры>.<…>`, а также слова-маркеры
  «вызывается / вызов / обращение» — ДЕЙСТВИЕ.

Это означает, что агенту-резолверу в большинстве случаев достаточно грамматики,
без глубокого понимания контекста.

## Решение 3. Гранулярность ссылки — элемент целиком, без якоря на атрибут

Ссылка ведёт на **сущность (или операцию) целиком**, не на конкретный атрибут.

Обоснование: в Confluence перехода по клику к конкретному атрибуту и сейчас нет,
аналитики в подавляющем большинстве случаев якоря не проставляют. Открытия файла
сущности достаточно — аналитик находит нужный атрибут глазами, агент —自самостоятельно.

Следствия (упрощения):
- Якоря (`#атрибут`) не нужны ни в файлах, ни в карточках.
- Маппинг «логический атрибут → поле DTO» в карточке не обязателен.
- Резолвер обрабатывает только `{{сервис:Имя}}`, часть `.<атрибут>` остаётся
  текстом после ссылки.

Единица ссылки = элемент (сущность или операция), не атрибут.

---

## Решение 4. Хранение смарт-ссылки и кликабельность

В Git хранится **смарт-ссылка** `{{сервис:Имя}}` — стабильна, не ломается при
переименовании цели, даёт чистые диффы.

Кликабельная (разрешённая) форма — производная, генерируется при показе:

| Артефакт | Назначение | Что показывает                   |
|---|---|----------------------------------|
| Исходный `.md` в IDE / GitLab MR | ревью изменений | смарт-ссылка `{{...}}` + git diff |
| Собранный MkDocs-сайт | чтение, навигация | разрешённые кликабельные ссылки  |

Для MkDocs кликабельность обеспечивается плагином-макросом (например,
`mkdocs-macros`), который на этапе сборки заменяет `{{...}}` на настоящую ссылку
по манифесту. Альтернатива — агент-препроцессор, генерирующий разрешённую копию
(в `.gitignore`). В обоих случаях в Git вливается смарт-ссылка.

## Решение 5. Ревью изменений — GitLab Merge Request (замена цветовой маркировки)

Ревью изменений проводится в **GitLab MR**, потому что только там есть git diff —
сравнение «было/стало». Это прямая и более строгая замена цветовой маркировки
Confluence (цвет = «изменено в задаче» → diff = что/когда/кем + полная история).

Семантика «отклонение от утверждённого состояния» обеспечивается моделью ветвления:
- **ветка = задача Jira**;
- **ревью = diff ветки против `main`**;
- тогда `main` = утверждённое (промышленное) состояние, diff = что меняет задача.

MkDocs-сайт diff не показывает и для ревью изменений не предназначен — он для чтения
утверждённых требований и навигации по кликабельным ссылкам. Это два разных
артефакта для двух разных задач.

## Решение 6. Манифест — минимальный состав

Манифест существует ради трёх операций: (1) резолв `{{сервис:Имя}}` в адрес,
(2) проверка существования (линтер ловит битые ссылки), (3) выбор механизма
разрешения (карточка vs swagger).

### Структура
Шапка + плоский список элементов:

```yaml
service_code: "РКО"
generated_at: "2026-06-12T03:00:00"
elements:
  - name: "[КК_ВК] Заявка"
    kind: card
    doc_id: "рко/data-model/КК_ВК-Заявка"
    url: "../external/рко/data-model/КК_ВК-Заявка.md"
  - name: "получитьСчёт"
    kind: swagger
    doc_id: "рко/api/получитьСчёт"
    url: "https://swagger.bank/рко/v2/#/operations/получитьСчёт"
```

### Поля записи (минимум)
- **`name`** — ключ резолва; точно совпадает с тем, что аналитик пишет в `{{...}}`, но без кода сервиса.
- **`kind`** — `card` | `swagger`; определяет цель (карточка / swagger).
- **`doc_id`** — стабильный идентификатор; переживает переименование (`name` может
  смениться), служит адресом карточки в Git.
- **`url`** — конечная цель кликабельной ссылки (путь к карточке либо swagger-URL).

### Поля шапки (минимум)
- **`service_code`** — какому сервису принадлежит манифест (резолвер сопоставляет
  префикс `{{рко:...}}`).
- **`generated_at`** — метка свежести (линтер предупреждает об устаревшем манифесте).

### Что намеренно НЕ включено в PoC
- `title` / развёрнутое описание — `name` уже человекочитаем.
- атрибутный состав сущности — он в самой карточке, манифест только адресует.
- версия/хеш — относится к синхронизации, не к резолву.
- права доступа как поле — регулируются правами GitLab на репозиторий; сам факт
  попадания в публичный манифест = «можно показывать».

### Ключ уникален
Заголовки (имена) уникальны в пределах сервиса — гарантировано правилом именования
с обязательными префиксами (`[КК_ВК] Заявка` vs `[КК_ЛК] Заявка`), если они есть (полный аналог наименования страницы Confluence). Поэтому `name`
достаточен как ключ, уточнители в нотации не нужны.

## Решение 7. Хранение манифестов и карточек — централизованное

Отдельная GitLab-группа (условно `dbo-registry`). CI каждого сервиса публикует туда
свой манифест и публичные карточки. Резолвер и агент читают оттуда.

Преимущества:
- **Дискаверабилити**: «какие есть сервисы» = «какие манифесты в `dbo-registry`».
  Отдельный реестр репозиториев не нужен — выводится из содержимого центра.
- Единая известная точка — все знают, где искать.

Источники резолвинга вынесены в **настройку** (`manifest_sources`), что оставляет
дверь для будущего гибрида без переписывания ядра:

```yaml
manifest_sources:
  - url: "https://gitlab.bank/dbo-registry/registry-open/"
    access: public
```

## Решение 8. Граница доступа на PoC — закрыта только логика

Принято упрощение для PoC: **закрыта только внутренняя логика**, а перечень
эндпоинтов/публичных элементов общедоступен (он и так известен потребителям).

- В публичную карточку НЕ попадает: алгоритмы, описание процессов, раздел
  «что делает функция», прочая внутренняя логика.
- В публичную карточку попадает: для сущностей — атрибутный состав и описание
  сущности; для экранных форм — описание полей, кнопок, фильтров.
- Для функций на целевом этапе публичная карточка может не генерироваться
  (есть OpenAPI); на переходный период — генерируется (входные/исходящие параметры).

Гибрид с раздельными правами (когда сам перечень элементов чувствителен) —
**отложен** до реальной потребности. Архитектурно поддержан через `manifest_sources`.
Обоснование: не тратить ресурсы преждевременно; переключение не мгновенно;
сначала доказать концепцию.

## Решение 9. Автоматизация публичности — через разметку разделов шаблона

Признак публичности навешивается не на каждый документ вручную, а **на разделы
шаблона типа требований один раз** (типов всего девять). Декларативно, например:

```
dataModel:   публичные разделы = [Атрибутный состав сущности, Описание сущности]
screenForm:  публичные разделы = [Описание полей, Кнопки, Фильтры]
function:    публичная карточка не генерируется (есть OpenAPI)
             # переходный период: [Входные параметры, Исходящие параметры]
```

(Указано в card_sections.json)
Команда аналитиков размечает шаблоны однократно; генератор карточек применяет
разметку к каждому документу автоматически. Ручная работа — разовая настройка
на тип, а не операция на каждый документ.

## Роль агента и RAG

- **RAG — ассистент написания, не адресная книга.** При наборе агент подсказывает
  («в РКО есть „Счёт“, „Счёт клиента“ — какой?»), аналитик выбирает, в текст попадает
  точная ссылка. RAG не используется как механизм адресации (т.к. вероятностный, а
  ссылка требует детерминизма).
- **Резолв — детерминированный, через манифест** (точное соответствие).
- **Агент-ревьюер резолвит `{{...}}` сам** через манифест; форма артефакта (MR/сайт)
  для него неважна. Требование: агент имеет доступ к публичному манифесту тех
  сервисов, на которые проверяет ссылки — в тех же границах доступа GitLab, что и
  человек.

## Открытые вопросы (следующие шаги)

1. **Генерация манифеста и карточек** — один CI-процесс: проходит по требованиям
   сервиса, по разметке разделов шаблона отбирает публичную часть, пишет строки
   манифеста и карточки, публикует в `dbo-registry`. Запуск при коммите в `main`. (Реализован - см. app/scripts/CI/build_cards.py)
2. **Механика синхронизации** — как и когда потребляющие сервисы получают свежие
   карточки/манифесты (ночная синхронизация зеркал? чтение из центра на лету?).
3. **Выбор IDE/редактора** для аналитиков с учётом работы агентов (отдельный вопрос).
4. **Поведение резолвера при переходном периоде**, когда у части сервисов ещё нет
   OpenAPI/манифеста — fallback-механизм.
