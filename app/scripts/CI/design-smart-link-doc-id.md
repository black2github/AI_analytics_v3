# Дизайн: location-независимый `doc_id` как смарт-ссылка (`{{SERVICE: label}}`)

> Возобновляемый документ. Если работа прервана — перечитай разделы 1–4 (решение и
> стратегия), затем 6 (чек-лист фаз со статусами) и продолжай с первой невыполненной

> явного пересмотра этого документа.

## 0. Статус и решения (ЗАФИКСИРОВАНО)

- **Цель: Вариант A** — `doc_id` и перекрёстные ссылки в теле документов имеют форму
  `{{SERVICE: label}}` и резолвятся через манифест в точке потребления. Полная
  location-независимость: файл можно двигать по дереву, ссылки не ломаются.
- **Способ выкатки: через промежуточную фазу B**, но B строится «A-ready» (см. §5),
  так что переход B→A — это в основном *вычитание* кода, а не переписывание.
- **Backward-compat не требуется**: миграция выполняется «с нуля» каждый прогон до
  перехода команды. Старые `doc_id` мигрировать не нужно; поддержка двух форматов
  одновременно не нужна.
- **Решено** (см. §7): `label` = полный заголовок Confluence, включая префикс
  (`[КК_ВК]`, `[КК_БК]` и т.п.); уникальность `label` внутри сервиса — ответственность
  аналитика. То есть `label == title == manifest name` (один объект, без вырезания/достройки).
- **Ещё не решено** (см. §7.4): схема имён файлов карточек.

## 1. Мотивация

Сейчас `doc_id` — путь файла относительно корня (`cc/[КК_ВК]-ЭФ/.../file`). Идентичность
документа завязана на расположение: переместить файл нельзя без смены `doc_id` и всех
ссылок на него. Цель — `doc_id` как смарт-ссылка `{{SERVICE: label}}`
(пример: `{{CC: [КК_БК] ЭФ Клиента: "Заявка на блокировку КК" в режиме просмотра}}`),
не зависящая от пути и совпадающая с **уже существующей** конвенцией межсервисной
ссылки `{{сервис:Имя}}` (`card_link_resolver.py:45`). Ссылка на документ == его `doc_id`.

## 2. Почему именно A, и почему B сам по себе недостаточен

Проверено по коду: и текущий пайплайн (Pass 2, `os.path.relpath`), и чужой
`converter.py` (`_resolve_page_link`, `converter.py:602-616`, `os.path.relpath` на 611)
резолвят ссылки в **относительные пути**, записываемые в .md. Оба ломаются при
перемещении файла. Двухфазный `page_map` converter'а решает *корректность внутри
прогона* (forward-refs, циклы), но **не** даёт стабильности после прогона.

Вывод: `converter.py` — это фактически Вариант B и **не образец** для A. Location-
независимость (резолв `{{...}}` через манифест в точке потребления) — новое свойство,
которого нет ни в converter, ни в текущем коде. Поэтому B повторяет статус-кво по части
перемещения, а заявленную мотивацию закрывает только A.

За периметр репозитория относительные пути не выразимы в принципе — межсервисные
ссылки **уже** имеют форму `{{...}}`. То есть `{{...}}`-адресация в системе неизбежна;
A лишь распространяет её на внутренние ссылки тоже.

## 3. Целевая модель (A)

Единая адресация: `doc_id = {{SERVICE: label}}`, **манифест — единственный источник
истины «doc_id → расположение/url»**.

- `SERVICE` — короткий код сервиса (`CC`, `SBP`, `INT`; он уже короткий в системе).
- `label` — **полный заголовок Confluence, включая префикс** (`[КК_ВК] Заявка`,
  `[КК_БК] Заявка` — разные документы). Префикс держит имена короткими и подчёркивает
  общее в разном. Без вырезания/нормализации: `label == title == manifest name`.
- **Имя файла на диске развязано с `doc_id`**: файл = `safe_filename(title)` (или иная
  стабильная схема), лежит где угодно; связь `doc_id → путь/url` держит манифест,
  перестраиваемый из текущих путей при каждом прогоне.
- Ссылки в теле = `{{...}}` цели; резолвятся прямым `lookup_by_doc_id` в манифесте.

**Парсинг `{{SERVICE: label}}`:** `SERVICE` — короткий код без двоеточий, поэтому
разбор идёт по **первому** `:` (`SERVICE` слева, всё остальное — `label`). `label` может
содержать двоеточия и кавычки (`[КК_БК] ЭФ Клиента: "Заявка..."`) — это нормально.
Регекс линтера (`[^:{}]+:` для кода, `.+` для label) это учитывает. Заголовки с `}}`
внутри не ожидаются (footgun низкого риска; при необходимости — экранировать позже).

## 4. Стратегия B→A как «вычитание»

B и A отличаются только тем, в каком виде живут **ссылки в теле** документа:
- B: относительные пути (кликабельны в обычном просмотрщике/IDE);
- A: `{{...}}` (location-независимы, но требуют резолвера/плагина для навигации).

Всё остальное (формат `doc_id`, развязка имени файла, манифест, резолв `{{...}}`)
**одинаково** и делается один раз в B. Поэтому при правильной инженерии B (см. §5)
переход к A сводится к:
1. удалить ветку резолва относительных путей в `card_link_resolver`
   (`_resolve_target_doc_id` + индекс манифеста по пути + их тесты);
2. переключить подстановку в Pass 2 миграции с относительного пути на `doc_id` цели
   (для from-scratch это всё, что нужно — отдельный проход-переписыватель не обязателен);
3. (опционально) запустить standalone-проход замены относительных ссылок на `{{...}}`
   для рукописных .md — он же становится действием «convert to smart-link» в IDE-плагине.

### Про последовательность с IDE-плагином
Плагин (навигация по `{{...}}`, в т.ч. за периметр репозитория) — это **A-внутренняя
работа**, а не накладной расход двухшаговости: он нужен для A в любом случае. Его
фундамент (резолв `{{...}}` против глобального манифеста) нужен уже в B для межсервисных
ссылок. Последовательность «сначала плагин → потом флип ссылок в `{{...}}`» **снижает
риск**: аналитики ни на одном этапе не остаются без рабочей навигации.

## 5. Инварианты «A-ready» — соблюдать при реализации B (КРИТИЧНО)

Эти решения делают B→A вычитанием, а не рерайтом. Нарушение любого превращает переход
в переписывание.

- **И-1. `doc_id` уже в финальном формате `{{SERVICE: label}}`** начиная с B (frontmatter,
  `build_doc_id`, валидация линтера). В B это меняем сразу, не откладывая на A.
- **И-2. Имя файла карточки развязано с `doc_id`** уже в B (`build_cards` не строит путь
  из `doc_id`). Манифест хранит `doc_id → имя/путь файла`.
- **И-3. Манифест несёт реальный путь/имя файла** (а не только `doc_id`) — нужен и для
  path-ветки B, и для url карточки в A.
- **И-4. Резолвер строится `{{...}}`-first**: `{{...}}` → `manifest.lookup_by_doc_id`
  (первичный путь, нужен для кросс-сервиса в любом случае); резолв относительных путей —
  **отдельная, явно изолированная ветка** в `card_link_resolver`, которую в A удаляют
  целиком, не задевая остальное.
- **И-5. Подстановка в Pass 2 миграции вынесена в одну точку** (формирование строки
  замены), чтобы флип «относительный путь → `{{doc_id}}`» был однострочным переключением.
- **И-6. Path-ветку не «прорастить» в остальной код** — держать локально в
  `card_link_resolver`, чтобы удаление в A было чистым.

## 9. Приложение: текущее состояние (как есть, до изменений)

### 9.1 Генерация `doc_id`
- `migrate_confluence_page.py:39-46` `build_doc_id` → `f"{service_part}/{subdir}/{filename}"`.
- `migrate_confluence_tree.py:161-166` → `doc_id = filepath.relative_to(OUTPUT_ROOT)` без расширения.

### 9.2 Кто трактует `doc_id` как путь (ломается при смене формата)
| Файл | Строки | Операция | Эффект |
|---|---|---|---|
| `build_cards.py` | 145 | `out_path / f"{doc_id}.md"` | **краш**: `:`, `"`, `{}`, `[]` недопустимы в имени файла (Windows) |
| `card_link_resolver.py` | 69-109 | `_resolve_target_doc_id` — арифметика путей | теряет смысл для смарт-ссылки |
| `manifest_builder.py` | 84 | `_doc_id_to_url` → `f"{doc_id}.md"` | неверный url |
| `card_link_resolver.py` | 61-66 | `_norm_doc_id` (срез `.md`, слеши) | безвреден, бессмыслен |
| `migrate_confluence_tree.py` | 161-166 | вывод `doc_id` из пути | требует переписи (Фаза 1) |

### 9.3 Две схемы адресации (сосуществуют сейчас; A сольёт в одну)
1. `doc_id`-как-путь — расположение карточки + резолв относительных ссылок (класс 3).
2. `{{сервис:Имя}}` — межсервисная ссылка, резолв по `name`/заголовку через манифест (класс 2).
   Уникальность `name` внутри сервиса уже контролируется (`manifest_builder.py:166-172`).

### 9.4 Два этапа резолва ссылок сейчас
- Этап 1 — миграция (Pass 2, `resolve_confluence_links`): `confluence://ID|title` →
  относительный путь (`os.path.relpath`).
- Этап 2 — сборка карточек (`card_link_resolver`): относительный путь → `doc_id` цели
  (арифметика) → url карточки из манифеста.
