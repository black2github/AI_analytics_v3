# Дизайн: location-независимый `doc_id` как смарт-ссылка и перевод card-пайплайна на манифест

Статус: черновик к обсуждению. Реализацию не начинать до принятия решений из раздела «Открытые вопросы».

## 1. Цель и мотивация

Сейчас `doc_id` — это путь файла относительно корня (`cc/[КК_ВК]-ЭФ/.../file`). Из-за этого:

- переместить файл по дереву каталогов нельзя без смены `doc_id` (а значит, и всех ссылок на него);
- идентичность документа завязана на его физическое расположение.

Цель — сделать `doc_id` **смарт-ссылкой вида `{{SERVICE: label}}`** (пример: `{{CC: [КК_БК] ЭФ Клиента: "Заявка на блокировку КК" в режиме просмотра}}`), которая:

- не зависит от пути файла → файл можно двигать без смены `doc_id`;
- совпадает с **уже существующей** в проекте конвенцией межсервисной ссылки `{{сервис:Имя}}` (см. `card_link_resolver.py:45`, `_INTERSERVICE_RE`). То есть ссылка на документ становится буквально его `doc_id`.

Backward-compat не требуется: миграция выполняется «с нуля» при каждом прогоне до перехода команды. Это снимает целый класс задач (миграция старых `doc_id`, поддержка двух форматов одновременно).

## 2. Текущее состояние (как есть)

### 2.1 Генерация `doc_id`
- `migrate_confluence_page.py:39-46` `build_doc_id` → `f"{service_part}/{subdir}/{filename}"`.
- `migrate_confluence_tree.py:161-166` → `doc_id = filepath.relative_to(OUTPUT_ROOT)` без расширения, прямые слеши.

### 2.2 Кто трактует `doc_id` как путь (ломается при смене формата)
| Файл | Строки | Операция | Эффект |
|---|---|---|---|
| `build_cards.py` | 145 | `out_path / f"{doc_id}.md"` | **краш**: `:`, `"`, `{}`, `[]` в имени файла недопустимы (Windows) |
| `card_link_resolver.py` | 69-109 | `_resolve_target_doc_id` — `PurePosixPath(doc_id).parent`, схлопывание `..` по `/` | теряет смысл: у смарт-ссылки нет структуры пути |
| `manifest_builder.py` | 84 | `_doc_id_to_url` → `f"{doc_id}.md"` | неверный url/путь |
| `card_link_resolver.py` | 61-66 | `_norm_doc_id` (срез `.md`, слеши) | безвреден, но бессмыслен |
| `migrate_confluence_tree.py` | 161-166 | вывод `doc_id` из пути | требует переписи (см. §4) |

### 2.3 Две схемы адресации ссылок (сосуществуют)
1. **`doc_id`-как-путь** — для расположения карточки и резолва относительных ссылок между документами (класс 3 в `card_link_resolver`).
2. **`{{сервис:Имя}}`** — межсервисная ссылка, резолвится по `name` (заголовку) через манифест (класс 2). Манифест уже контролирует уникальность `name` внутри сервиса (`manifest_builder.py:166-172`).

### 2.4 Два этапа резолва ссылок
- **Этап 1 — миграция (Pass 2, `resolve_confluence_links`)**: `confluence://ID|title` → **относительный путь** к целевому .md (через `os.path.relpath`).
- **Этап 2 — сборка карточек (`card_link_resolver`)**: относительный путь → `doc_id` цели (арифметика путей) → url карточки из манифеста.

Замысел п.1 — **слить обе схемы в одну** (`{{...}}`) и убрать арифметику путей.

## 3. Целевая модель

**Единая адресация: `doc_id = {{SERVICE: label}}`, манифест — единственный источник истины «doc_id → расположение».**

- `SERVICE` — короткий код сервиса (`CC`, `SBP`, `INT`), он уже короткий в системе (`build_doc_id` его лишь нормализует).
- `label` — человекочитаемое имя документа = тот же `name` (заголовок), что уже используется в манифесте. Ссылка на документ == его `doc_id`.
- **Имя файла на диске развязано с `doc_id`**: файл может называться `safe_filename(title)` и лежать где угодно; связь `doc_id → путь/url` хранит манифест, который строится из текущих путей при каждом прогоне.

## 4. Развязка перекрёстных ссылок — РАЗВИЛКА (главное решение)

Цель «двигать файл без смены `doc_id`» выполняется для **собственной идентичности** документа в обоих вариантах ниже. Разница — в том, что происходит со ссылками *между* документами.

### Вариант A — ссылки в теле становятся `{{...}}` (полная location-независимость)
- Pass 2 миграции резолвит `confluence://ID|title` не в относительный путь, а в **`doc_id` цели** (`{{SERVICE: Title}}`). Реестры миграции отображают `page_id`/`title` → `doc_id` (смарт-ссылка), а не `Path`.
- `card_link_resolver` для `{{...}}` делает **прямой `manifest.lookup_by_doc_id`** → url карточки. Арифметика путей (`_resolve_target_doc_id`) удаляется.
- ➕ перемещение любого файла не ломает ссылки (они логические, резолв в точке потребления).
- ➖ в исходных .md (`conf-requirements`) ссылки выглядят как `{{...}}` — **не кликабельны** в обычном просмотрщике Markdown/в IDE при ревью аналитиком. Кликабельный вид появляется только в финальных карточках после резолва.

### Вариант B — ссылки в теле остаются относительными путями (кликабельны), `doc_id` в frontmatter — смарт-ссылка
- Pass 2 миграции работает как сейчас (относительные пути).
- `card_link_resolver` заменяет арифметику путей на lookup через **новый индекс `путь → doc_id`**, который отдаёт миграционный манифест; затем `doc_id → url`.
- ➕ исходные .md остаются кликабельными для аналитиков.
- ➖ перемещение файла ломает относительные ссылки *в теле* до следующей ре-миграции (но `doc_id` в frontmatter стабилен, и манифест перестроится).

**Рекомендация:** Вариант A — он единственный реализует заявленную мотивацию в полном объёме и убирает целый механизм (арифметику путей), сводя всё к одной конвенции `{{...}}` + манифест. Потерю кликабельности в исходниках считаю приемлемой: исходники — промежуточный артефакт, конечный навигируемый вид даёт MkDocs по карточкам. Но это **решение пользователя** (см. §7).

## 5. Инвентарь правок (для Варианта A)

### Миграция
1. `build_doc_id` (`migrate_confluence_page.py:39-46`) → `f"{{{{{service_code}: {label}}}}}"`, где `label` = заголовок (или его нормализованная форма). Решить про `subdir`/контекст в label (см. §7 уникальность).
2. `migrate_confluence_tree.py:161-166` — `doc_id` больше не выводится из `filepath`; берётся той же функцией из `(service_code, title)`.
3. Реестры (`page_registry`, `title_registry`) и `seed_registries_from_disk`: значение — `doc_id` цели (смарт-ссылка), а не `Path`.
4. Pass 2 (`resolve_confluence_links`): подстановка `[text]({{SERVICE: Title}})` вместо относительного пути. Регексы и `_title_key` остаются; меняется только формируемая замена.

### Card-пайплайн
5. `build_cards.py:145` — имя файла карточки выводить из `safe_filename(title)` (или иной стабильной схемы), **не** из `doc_id`. Записать соответствие `doc_id → имя файла` в манифест.
6. `manifest_builder._doc_id_to_url` (`:66-84`) — `url` берётся из реального имени/пути карточки, а не из `f"{doc_id}.md"`.
7. `card_link_resolver` — удалить `_resolve_target_doc_id` (арифметика путей) и `_norm_doc_id` (срез `.md`); класс `{{...}}` перевести из «keep as-is» в «lookup в манифесте → redirect» (`_classify_and_resolve`, `:130-169`). Ссылки на цели не из манифеста по-прежнему расплющиваются.

### Линтер
8. `lint_frontmatter.py` — добавить валидацию формата `doc_id`: `^\{\{\s*[^:{}]+:\s*.+\}\}$` (вместо «только непустой»).

### Тесты
9. Обновить/добавить: формирование `doc_id` (миграция), Pass 2 → `{{...}}`, `card_link_resolver` lookup по `{{...}}`, запись карточки по новому имени, валидация линтера.

## 6. Что НЕ затрагивается
- Эмбеддинги/ChromaDB — используют `page_id`, не `doc_id` (идентичность вектора не меняется; `embedding_store.py:270-278`).
- Резолв картинок (`image_migrator`) — отдельная схема `confluence-attachment://`.

## 7. Открытые вопросы (решить до реализации)
1. **Развилка §4: Вариант A или B?** (рекомендую A).
2. **Уникальность `label` внутри сервиса.** `{{CC: Title}}` требует уникальности заголовка в пределах сервиса. Манифест это уже предполагает и предупреждает о дублях (`manifest_builder.py:167-171`). Варианты: (а) считать уникальность ответственностью аналитика (как сейчас для `{{...}}`); (б) при коллизии добавлять дизамбигуатор (родительский раздел/числовой суффикс) в `label`. Что выбираем?
3. **Состав `label`.** Чистый заголовок или с контекстным префиксом раздела? Влияет и на уникальность (п.2), и на читаемость.
4. **Схема имён файлов карточек** (после развязки с `doc_id`): `safe_filename(title)` плоско в каталоге сервиса? с разрешением коллизий суффиксом? Манифест хранит `doc_id → имя файла`.
