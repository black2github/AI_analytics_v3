# Стратегия перехода на API First — Экосистема

## 1. Контекст и отправная точка

### 1.1. Текущее состояние

Система ДБО для юридических лиц реализована как экосистема ~100 микросервисов.
Аналитика ведётся на логическом уровне: сущности, алгоритмы, функции и
интеграционные методы описываются в Confluence по нотации, зафиксированной в
документе `[Правила оформления аналитики ДБО]`.

Параллельно ведётся миграция аналитики на Doc as Code (MkDocs), включая механизм
смарт-ссылок `{{сервис:элемент}}` для межсервисных зависимостей.

Технические характеристики текущей реализации:

- Swagger генерируется из кода (code-first) и полностью актуален.
- Вся экосистема использует единый OAuth-токен (единый слой аутентификации и авторизации).
- Навигация между сервисами — URL-переходы с прозрачной передачей токена. Для
  пользователя работа в системе полностью прозрачна вне зависимости от того, какой
  микросервис его обслуживает в данный момент.
- Практика contract testing существует: автоматические, нагрузочные и ручные тесты.

### 1.2. Инициатива

Переход на API First — решение на уровне всего департамента, охватывает все
~100 микросервисов. Ресурсы на переход выделены. Несколько пионерных команд
прокладывают путь, вырабатывают шаблоны и runbook, после чего остальные команды
выполняют переход самостоятельно по готовым шаблонам.

Ориентировочный срок полного перехода — несколько месяцев.

## 2. Что такое API First в контексте ДБО

API First в данном контексте — **не** «перенести весь функционал в swagger». Это:

> Межсервисные контракты формализованы в OpenAPI **до** начала реализации,
> а не выводятся из уже написанного кода.

Логический уровень при этом **не упраздняется** — он остаётся как implementation
guide: описывает, как сервис реализует опубликованный контракт внутри.

Итог: два дополняющих слоя документации с чёткой зоной ответственности каждого.

| Слой | Содержание | Основной потребитель |
|---|---|---|
| OpenAPI (swagger) | Что сервис публикует наружу: endpoint'ы, схемы запросов и ответов | Разработчики соседних сервисов, contract testing |
| Логическая аналитика (Doc as Code) | Как сервис реализует контракт: алгоритмы, модель данных, экранные формы | Разработчики своего сервиса, аналитики |

## 3. Карта маппинга: логические элементы → API First

### 3.1. Функции

Функция в логической аналитике — **навигационный endpoint**: точка входа в
UI-домен сервиса, после вызова которой управление клиентским путём полностью
передаётся принимающему сервису. Одна функция = один swagger endpoint.

Логическая роль функции двойная: разграничение прав пользователя на использование
функционала и описание клиентского пути. Входящие параметры функции — контекст
перехода (с какими данными пользователь пришёл). Выходных параметров у
навигационных функций в логической аналитике нет: управление не возвращается
через этот же канал.

Аналитики соседних сервисов ориентируются на логическое описание функции
(входящие параметры) как на межсервисный контракт. API First формализует этот
де-факто существующий контракт в машиночитаемый формат OpenAPI.

```
[Функция просмотра выписки]              →  GET  /statements/view
[Функция].<Входные параметры>.<param>    →  query param / path param
```

Операционные функции (действия, не навигация):

```
[Функция подачи заявки на выпуск карты]  →  POST  /corporate-cards/requests
```

### 3.2. Интеграционные методы

Интеграционные методы описывают взаимодействие с внешними (смежными) системами.
По своей природе они уже являются контрактами — фаза API First для них состоит
в документальной фиксации маппинга `[Метод]` → swagger endpoint внешней системы.

### 3.3. Экранные формы

Экранная форма (`screenListForm`, `screenItemForm`) описывает структуру UI.
В API First контексте — это фактически **response schema** соответствующего
GET-endpoint'а (view-модель). Связь фиксируется в логическом документе формы
ссылкой на swagger operation.

### 3.4. Внутренние алгоритмы и модель данных

Алгоритмы, манипулирующие внутренней моделью данных сервиса, — это
implementation detail. В swagger не попадают. Остаются в логической документации
без изменений. Именно в этом и состоит смысл слабой связности в микросервисах:
внутренняя логика — дело самого сервиса.

Модель данных частично маппируется на JSON Schema компонентов swagger
(схемы request/response body) в той мере, в которой атрибуты сущностей
входят в публичный контракт.

### 3.5. Сводная таблица

| Логический элемент | Объект API First? | API First эквивалент |
|---|---|---|
| Функция (навигационная) | ✅ Да | `GET /service/resource?params` |
| Функция (операционная) | ✅ Да | `POST /service/resource` |
| Интеграционный метод | ✅ Да (уже контракт) | Маппинг на существующий endpoint |
| Экранная форма | ⚡ Частично | Response schema функции (view-модель) |
| Внутренний алгоритм | ❌ Нет | Остаётся в логической документации |
| Модель данных сервиса | ⚡ Частично | JSON Schema (компоненты swagger) |

## 4. Три зоны функционала

### Зона 1 — Публичный API (объект перехода)

То, что сервис публикует наружу и что другие сервисы вызывают:

- Функции (навигационные и операционные).
- Интеграционные методы с внешними системами.

**Действие:** формализовать в OpenAPI до начала разработки нового функционала;
для существующего — провести маппинг и задокументировать связь.

### Зона 2 — Внутренние endpoint'ы сервиса (не объект перехода)

Эндпоинты, которые сервис дёргает для реализации своего UI: получение сущностей,
обновление атрибутов и т.д. Уже описаны в swagger (code-first, актуальны).
Не являются публичным контрактом для других сервисов.

**Действие:** не трогать, swagger актуален.

### Зона 3 — Внутренние алгоритмы (не объект перехода)

Бизнес-логика и манипулирование своей моделью данных.

**Действие:** не трогать, продолжают описываться в логической аналитике.

## 5. Двуслойная модель документации

Два слоя связаны взаимными ссылками. Смарт-ссылка `{{сервис:элемент}}`
в логическом документе резолвится в swagger operation. В swagger `description`
(или OpenAPI extension) — обратная ссылка на логический документ.

```
[Логический уровень]                       [API уровень / swagger]
────────────────────────────────           ────────────────────────────────
doc_type: function                         GET /corporate-cards/{id}
[КК_ВК Просмотр карты]                    parameters:
  .<Входные параметры>                       - cardId (path)
    .<идентификатор карты>                 responses:
  Алгоритм: ...                              200: CardViewModel
  → {{КК:GET /corporate-cards/{id}}} ──→  description: |
                                             Логический документ:
                                             [ссылка на Doc as Code]
```

### Процесс для нового функционала (API First)

1. Аналитик описывает endpoint в OpenAPI: path, params, request/response schema.
2. Пишет логический документ функции со ссылкой на swagger endpoint.
3. Разработчик реализует, ориентируясь на swagger как источник истины.
4. Contract testing покрывает соответствие реализации контракту.

### Процесс для существующего функционала (маппинг)

1. Найти в swagger endpoint, соответствующий каждой функции сервиса.
2. Добавить в логический документ функции смарт-ссылку на swagger operation.
3. Добавить в swagger description обратную ссылку на логический документ.

## 6. Операционный план перехода

### Фаза 0 — Подготовка: пионерные команды

**Задача:** выбрать 1–2 сервиса-пилота, пройти переход, выработать все шаблоны
и задокументировать опыт.

Критерии выбора пилотного сервиса:

- Средний по размеру и сложности (не минимальный, чтобы шаблон был репрезентативным,
  и не самый крупный, чтобы не затягивать фазу).
- Содержит разнообразие типов функций: и навигационные, и операционные.
- Имеет несколько интеграционных методов.
- Команда мотивирована и готова документировать опыт, включая краевые случаи.

Результат фазы: полный набор артефактов (см. раздел 7).

### Фаза 1 — Малые и средние сервисы

Команды выполняют переход по шаблону. Фиксируют краевые случаи в общем реестре.
Реестр пополняется, шаблон при необходимости уточняется держателем.

### Фаза 2 — Крупные и сложные сервисы

К этому моменту шаблон дополнен опытом Фазы 1 и покрывает нетипичные ситуации.
Сложные сервисы используют уточнённую версию.

### Параллельный процесс: новый функционал

С момента старта инициативы весь **новый функционал всех команд** описывается
по API First процессу вне зависимости от фазы перехода для существующего.

## 7. Артефакты пионерных команд

| Артефакт | Содержание |
|---|---|
| Конвенция именования endpoint'ов | Как логическое имя функции (`[КК_ВК Просмотр карты]`) транслируется в URL path |
| Шаблон описания функции в OpenAPI | Пример заполненного path item с пояснениями для аналитика |
| Шаблон логического документа функции | Функция с правильно оформленными смарт-ссылками на swagger |
| Runbook перехода сервиса | Пошаговая инструкция: что делать, в каком порядке, кто отвечает |
| Чеклист перехода | Как команда проверяет, что переход выполнен корректно |
| Реестр краевых случаев | Нестандартные ситуации и принятые решения; пополняется всеми командами в процессе |
