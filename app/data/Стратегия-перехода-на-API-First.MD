# Стратегия перехода на API First — Экосистема

## 1. Контекст и отправная точка

### 1.1. Текущее состояние

Система ДБО для юридических лиц реализована как экосистема ~100 микросервисов.
Аналитика ведётся на логическом уровне: сущности, алгоритмы, функции и
интеграционные методы описываются в Confluence по нотации, зафиксированной в
документе `[Правила оформления аналитики ДБО]`.

Параллельно ведётся миграция аналитики на Doc as Code (MkDocs), включая механизм
смарт-ссылок `{{сервис:элемент}}` для межсервисных зависимостей.

Технические характеристики текущей реализации:

- Текущий swagger генерируется из кода (code-first) и полностью актуален.
- Вся экосистема использует единый OAuth-токен (единый слой аутентификации и авторизации).
- Навигация между сервисами — URL-переходы с прозрачной передачей токена. Для
  пользователя работа в системе полностью прозрачна вне зависимости от того, какой
  микросервис его обслуживает в данный момент.
- Практика contract testing стандартная: автоматические, нагрузочные и ручные тесты.

### 1.2. Инициатива

Переход на API First — решение на уровне всего департамента, охватывает все
~100 микросервисов. Ресурсы на переход выделены. Несколько пионерных команд
прокладывают путь, вырабатывают шаблоны и runbook, после чего остальные команды
выполняют переход самостоятельно по готовым шаблонам.

Ориентировочный срок полного перехода — несколько месяцев.

## 2. Что такое API First в контексте ДБО

API First в данном контексте — **не** «перенести весь функционал в swagger». Это:

> Межсервисные контракты формализованы в OpenAPI **до** начала реализации,
> а не выводятся из уже написанного кода.

Логический уровень при этом **не упраздняется** — он остаётся как implementation
guide: описывает, как сервис реализует опубликованный контракт внутри.

Итог: два дополняющих слоя документации с чёткой зоной ответственности каждого.

| Слой | Содержание | Основной потребитель |
|---|---|---|
| OpenAPI (swagger) | Что сервис публикует наружу: endpoint'ы, схемы запросов и ответов | Разработчики соседних сервисов, contract testing |
| Логическая аналитика (Doc as Code) | Как сервис реализует контракт: алгоритмы, модель данных, экранные формы | Разработчики своего сервиса, аналитики |

## 3. Карта маппинга: логические элементы → API First

### 3.1. Функции

Функция в логической аналитике — **навигационный endpoint**: точка входа в
UI-домен сервиса, после вызова которой управление клиентским путём полностью
передаётся принимающему сервису. Одна функция = один swagger endpoint.

Логическая роль функции двойная: разграничение прав пользователя на использование
функционала и описание клиентского пути. Входящие параметры функции — контекст
перехода (с какими данными пользователь пришёл). Выходных параметров у
навигационных функций в логической аналитике нет: управление не возвращается
через этот же канал.

Аналитики соседних сервисов ориентируются на логическое описание функции
(входящие параметры) как на межсервисный контракт. API First формализует этот
де-факто существующий контракт в машиночитаемый формат OpenAPI.

```
[Функция просмотра выписки]              →  GET  /statements/view
[Функция].<Входные параметры>.<param>    →  query param / path param
```

Операционные функции (действия, не навигация):

```
[Функция подачи заявки на выпуск карты]  →  POST  /corporate-cards/requests
```

### 3.2. Интеграционные методы

Интеграционные методы описывают взаимодействие с внешними (смежными) системами.
По своей природе они уже являются контрактами — фаза API First для них состоит
в документальной фиксации маппинга `[Метод]` → swagger endpoint внешней системы.

### 3.3. Экранные формы

Экранная форма (`screenListForm`, `screenItemForm`) описывает структуру UI.
В API First контексте — это фактически **response schema** соответствующего
GET-endpoint'а (view-модель). Связь фиксируется в логическом описании формы
ссылкой на swagger operation.

### 3.4. Внутренние алгоритмы и модель данных

Алгоритмы, манипулирующие внутренней моделью данных сервиса, — это
implementation detail. В swagger не попадают. Остаются в логической документации
без изменений. Именно в этом и состоит смысл слабой связности в микросервисах:
внутренняя логика — дело самого сервиса.

Модель данных частично маппируется на JSON Schema компонентов swagger
(схемы request/response body) в той мере, в которой атрибуты сущностей
входят в публичный контракт.

### 3.5. Сводная таблица

| Логический элемент | Объект API First? | API First эквивалент |
|---|---|---|
| Функция (навигационная) | ✅ Да | `GET /service/resource?params` |
| Функция (операционная) | ✅ Да | `POST /service/resource` |
| Интеграционный метод | ✅ Да (уже контракт) | Маппинг на существующий endpoint |
| Экранная форма | ⚡ Частично | Response schema функции (view-модель) |
| Внутренний алгоритм | ❌ Нет | Остаётся в логической документации |
| Модель данных сервиса | ⚡ Частично | JSON Schema (компоненты swagger) |

## 4. Три зоны функционала

Весь функционал сервиса делится на три зоны с принципиально разным решением при
переходе. Граница зоны определяется тем, **кто владелец и потребитель** элемента:
пересекает ли он границу сервиса (является контрактом) или остаётся его внутренним
делом.

### Зона 1 — Публичный API (объект перехода)

То, что сервис **публикует наружу** и что вызывают другие сервисы. Сюда входят:

- Функции — навигационные (точка входа в UI-домен сервиса, после вызова которой
  управление клиентским путём передаётся принимающему сервису) и операционные
  (действие над ресурсом).
- Интеграционные методы с внешними (смежными) системами — по своей природе уже
  являются контрактами.

Это и есть настоящий объект API First: контракт существует **до** реализации и
зафиксирован в OpenAPI. Именно на элементы этой зоны ориентируются аналитики и
разработчики соседних сервисов.

**Действие:** формализовать в OpenAPI до начала разработки нового функционала;
для существующего — провести маппинг функции на существующий endpoint и
задокументировать связь.

### Зона 2 — Внутренние endpoint'ы сервиса (не объект перехода)

Эндпоинты, которые сервис **дёргает сам** для реализации собственного UI:
получить сущность, обновить атрибут, проверить доступность. Они уже описаны в
swagger (code-first, актуальны), но не являются публичным контрактом для других
сервисов — это деталь реализации, которая не пересекает границу сервиса.

**Действие:** не трогать, swagger актуален. Объектом API First не являются.

### Зона 3 — Внутренние алгоритмы и клиентский путь (не объект перехода)

Бизнес-логика, манипулирующая внутренней моделью данных сервиса, и проверки
клиентского пути. Это implementation detail: как именно сервис обрабатывает то,
что принял через публичный контракт. В swagger не попадает. Именно в этом и
состоит смысл слабой связности микросервисов — внутренняя логика есть дело
самого сервиса.

**Действие:** не трогать, продолжают описываться в логической аналитике как
implementation guide для команды сервиса.

> **Различение клиентского пути и бизнес-процесса.** Проверки клиентского пути
> (реакция на ввод, доступность кнопок, переходы между вкладками) принадлежат
> Зоне 3. Проверки бизнес-процесса и безопасности (целостность процесса,
> полномочия, статусные правила) принадлежат Зоне 1, потому что фронтовую
> проверку клиент может обойти — гарантия обязана жить на стороне сервиса.
> Подробнее это раскрыто в разделе 6 о контролях.

## 5. Двуслойная модель документации

Два слоя связаны взаимными ссылками.

- **Логический → API.** Логический элемент, который **использует** операцию,
  ссылается на неё смарт-ссылкой `{{сервис:[Имя элемента]}}`, где ключ — это `name`
  элемента из манифеста (человекочитаемое имя с префиксом). Смарт-ссылка резолвится в операцию: swagger-URL
  `…#/operations/{operationId}` (этап API First) либо файл операции после сплита
  YAML спецификации (этап Doc-as-Code).
- **API → логический.** Операция несёт обратные ссылки: `externalDocs` (первичный
  документ), `x-logical-doc` (полный список использующих элементов) и
  человекочитаемый перечень в `description`.
  - **Кликабельность в обоих рендерерах.** Один href не может одновременно
    открывать локальный файл (IDE/MkDocs) и работать в standalone Swagger UI,
    поэтому форма ссылки выбирается по рендереру поля:
    - `externalDocs.url` и ссылки в `description` — **абсолютные URL** (Swagger UI
      рендерит только их; абсолютная ссылка кликабельна и в IDE — открывается в браузере).
    - `x-logical-doc` несёт все формы: `ref` (смарт-ссылка, канон/источник),
      `path` (относительный — локальная навигация в IDE/MkDocs), `url` (абсолютный — swagger).
    - Относительные ссылки для перехода к локальному файлу живут в самих `.md`
      (их рендерит MkDocs/IDE).
  - Абсолютные `url` и относительные `path` — производные от `ref`; их проставляет
    тот же сборщик, что генерирует `x-logical-doc` (база/ветка — настройка деплоя).
- Функция, реализуемая операцией 1:1, **на себя не ссылается**: её связь с
  операцией фиксируется записью в манифесте сервиса (функция — элемент
  `kind: swagger`) и обратной ссылкой в самой операции.

```
[Логический уровень]                         [API уровень / swagger]
──────────────────────────────              ──────────────────────────────
requirement_type: screenItemForm             POST /.../set-limits/save
[КК_ЛК] ЭФ Клиента: Заявка ...               operationId: createOrUpdate
  использует операцию сохранения             externalDocs:  → логич. описание процесса
  → {{CC:[КК_ЛК] Клиент: Функция             x-logical-doc:
     создания заявки ...}} ──────────────→     - function / screenItemForm (имена)
                                              description: |
                                                Назначение функции: ...
```

### Процесс для нового функционала (API First)

1. Аналитик описывает endpoint в OpenAPI: path, params, request/response schema.
2. Формирует лог. описание функции; связь функции с операцией фиксируется
   записью в манифесте (`kind: swagger`) и обратной ссылкой в самой операции.
3. Разработчик реализует, ориентируясь на swagger как источник истины.
4. Contract testing покрывает соответствие реализации контракту.

### Процесс для существующего функционала (маппинг)

1. Найти в swagger endpoint, соответствующий каждой функции сервиса.
2. Зафиксировать связь функции и операции: запись манифеста (`kind: swagger`)
   + обратная ссылка в операции. Другие элементы, использующие операцию,
   ссылаются на неё `{{сервис:[Имя элемента]}}`.
3. Добавить в swagger операцию обратные ссылки (`externalDocs` + `x-logical-doc`)
   на использующие её логические документы. Указать назначение лог. функции в `description`.

## 6. Контроли (проверки) — разделение по владельцу

### 6.1. Проблема единой страницы

Сегодня контроли описываются на единой странице (пример — `[O2+NEW] Контроли
заявки`): один раздел перечисляет группы триггеров, второй — таблицу проверок,
где на пересечении проверки и группы стоит «V». Логическое описание функции или
процесса ссылается на страницу и указывает событие, что лаконично определяет
весь набор проверок в данной точке.

Единая страница смешивает под одной крышей проверки с разным «домом» в парадигме
API First:

- **Контроли клиентского пути** (фронтовые) — реакция на ввод, доступность
  кнопок, переходы между вкладками. Удобство UX: подсказать пользователю раньше,
  чем он отправит заявку. Клиент их может обойти.
- **Контроли бизнес-процесса и безопасности** (бэковые) — целостность процесса,
  проверка полномочий, статусные и межполевые правила, проверки по справочникам.
  Обязаны выполняться на стороне сервиса независимо от поведения клиента.

### 6.2. Принятое решение

> **Контроли разделяются по владельцу на два документа.** Ось деления — не
> технология (фронт/бэк), а смысл: **клиентский путь** против **бизнес-процесса
> и безопасности**. Это деление почти точно совпадает с границами зон.

**Документ «Контроли клиентского пути»** (`requirement_type: control`, подтип
`ui`). Живёт рядом с экранными формами. Триггеры — кнопки, смена фокуса,
изменение поля, переходы между вкладками. Относится к **Зоне 3**, в OpenAPI не
проецируется.

**Документ «Контроли бизнес-процесса»** (`requirement_type: control`, подтип
`rule` + `field`). Живёт рядом с публичными операциями сервиса. Триггеры — смена
статуса документа в жизненном цикле (сохранение, отправка на подпись, приём в
банк). Относится к **Зоне 1**: на него ссылается операция OpenAPI. Гарантирует
целостность бизнес-процесса и безопасность независимо от клиента.

Оба документа **остаются централизованными** (отдельная страница на набор
контролей, не растворяются в описаниях функций и процессов). Причина: одни и те
же проверки выполняются в рамках разных функций и процессов; привязка их к
конкретным точкам вызова через описание чревата ошибками и рассинхронизацией.
Централизация на отдельной странице сохраняет переиспользование набора между
сколь угодно многими точками вызова — это исходная ценность модели контролей.

### 6.3. Единый источник истины для field-контролей

`field`-контроли (обязательность, формат, длина, допустимое значение из
справочника) физически принадлежат **документу бизнес-процесса** — это контракт
данных, который сервис обязан соблюдать. Документ клиентского пути их **не
копирует, а цитирует по ID** для тех полей, где нужна ранняя реакция на фронте.
Так у каждого контроля один источник истины, а присутствие на клиенте выражается
ссылкой, а не дубликатом.

Для цитирования контролям присваиваются **стабильные сквозные ID** (например,
`CTRL-03`), переживающие правки и добавление новых контролей (нумерация не
сдвигается).

### 6.4. Связь с OpenAPI

Операция ссылается на документ бизнес-процесса одной общей ссылкой `x-controls`
с указанием применимой группы — **без перечисления каждого сообщения об ошибке**
в контракте. Сообщений может быть очень много, и их текстовки меняются; контракт
не должен дёргаться от смены формулировки. Стабильна **структура** ответа
(`code` + `message`), а не его наполнение.

```yaml
post:
  operationId: signRequest
  summary: Подписать заявку
  x-controls:
    source: "{{o2Plus:[O2+NEW] Контроли бизнес-процесса заявки}}"
    group: 3
  responses:
    '422':
      description: Контроли бизнес-процесса (группа №3) не пройдены
      content:
        application/json:
          schema: { $ref: '#/components/schemas/ControlViolation' }
```

Из логического описания функции или процесса ссылка на группу даётся смарт-ссылкой
с якорем группы (машинно-резолвимый primary) и событием как человекочитаемым
алиасом:

```
На шаге №2 процесса выполняются
{{o2Plus:[O2+NEW] Контроли бизнес-процесса заявки#группа-3}}
(событие: «Приём заявки в банк»).
```

Инверсия «где применяется набор контролей» не ведётся вручную — генерируется
агентом из графа смарт-ссылок (тот же механизм, что и для `x-logical-doc`).

## 7. Операционный план перехода

### Фаза 0 — Подготовка: пионерные команды

**Задача:** выбрать 1–2 сервиса-пилота, пройти переход, выработать все шаблоны
и задокументировать опыт.

Критерии выбора пилотного сервиса:

- Средний по размеру и сложности (не минимальный, чтобы шаблон был репрезентативным,
  и не самый крупный, чтобы не затягивать фазу).
- Содержит разнообразие типов функций: и навигационные, и операционные.
- Имеет несколько интеграционных методов.
- Команда мотивирована и готова документировать опыт, включая краевые случаи.

Результат фазы: полный набор артефактов (см. раздел 8).

### Фаза 1 — Малые и средние сервисы

Команды выполняют переход по шаблону. Фиксируют краевые случаи в общем реестре.
Реестр пополняется, шаблон при необходимости уточняется держателем.

### Фаза 2 — Крупные и сложные сервисы

К этому моменту шаблон дополнен опытом Фазы 1 и покрывает нетипичные ситуации.
Сложные сервисы используют уточнённую версию.

### Параллельный процесс: новый функционал

С момента старта инициативы весь **новый функционал всех команд** описывается
по API First процессу вне зависимости от фазы перехода для существующего.
