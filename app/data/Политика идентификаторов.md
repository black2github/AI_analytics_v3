# Политика идентификаторов

### Принципы (инвариантны к выбору префиксов)

1. **Алфавит**: только `A–Z`, `0–9`, дефис, точка. Кириллица в ID запрещена полностью.
2. **Анти-омоглифы**: запрещены однобуквенные префиксы и префиксы, целиком состоящие из букв, совпадающих по начертанию с кириллицей (`A B C E H K M O P T X Y`). Каждый префикс — ≥2 символов и содержит хотя бы одну «однозначно латинскую» букву (`D F G I J L N Q R S U V W Z`). Это делает опечатку-кириллицу видимой и грепаемой. Именно поэтому **прямая транслитерация П→P, Е→E, М→M — худший вариант**: получаются чистые омоглифы.
3. **Структура**: глобальные — `PREFIX[-КОНТУР]-NN[.K]`; локальные (внутри документа) — `PREFIX-N[.M]`. Полный адрес локального: `<id документа>/<локальный id>`, например `SCR-CL-03.1/EV-1.9`.
4. **Ось классификации — не в ID.** Изменчивое свойство (FE/BE у контроля, навигационная/операционная у функции, item/list у формы) кодируется в frontmatter/реестре, а не в префиксе: переклассификация не должна ломать ссылки. Мы это уже проходили — контроли мигрируют между зонами.
5. **Стабильность**: ID не перенумеровываются и не переиспользуются (правило Е-N.M обобщается на все ID).
6. **Область уникальности**: глобальные — комплект сервиса (сервис самодостаточен — политика фиксируется в `docs/README.md` комплекта); локальные — документ.

### Таблица префиксов (вариант A — рекомендуется; совместим с действующей пилотной схемой от Прохорова)

| Тип требования | Префикс | Статус |
| --- | --- | --- |
| Процессы | `PRC-NNN` | есть (⚠️ conventions §5 пишет `PROC`, файлы — `prc-002` — рассинхрон в самом репо, предлагаю унифицировать на `PRC`) |
| Диаграммы состояний | `STM-NNN` (state machine); пока модель одна — фикс-имя `status-model.md` без ID | новый |
| Контроли (FE и BE) | `CTL-NNN`, группы `CTL-GRP-N`; зона FE/BE — свойство карточки (принцип 4) | есть |
| Функции (навигационные и операционные) | `FUN-CL-NN` / `FUN-BNK-NN` / `FUN-SYS-NN`; нав/опер — свойство | есть |
| Интеграционные методы | `EXTINT-NNN` | есть |
| Нотификации | `NTF-NNN` | шаблон есть |
| Модель данных | `ENT-NNN`; платформенные — `EXT-NNN` ⚠️ | есть |
| ИБ | `SEC-NNN` | новый |
| НФТ | `NFR-NNN` | новый |
| ЭФ (сущность и журнал) | `SCR-CL-NN[.K]` / `SCR-BNK-NN[.K]`; item/list — свойство (`requirement_type`) | есть |
| Глоссарий | фикс-имя `glossary.md`, термины — якорями; `GLS-NNN` только если понадобится адресация терминов извне | есть |
| Печатные формы | `PRN-NNN` | есть |

⚠️ **EXT** нарушает принцип 2 (все три буквы — омоглифы: Е,Х,Т) и путается с EXTINT (в conventions даже есть оговорка «не путать»). Пока предлагается оставить `EXT` как задокументированное исключение.

### Локальные ID событийной ЭФ (замена кириллицы)

| Было | Стало | Мнемоника |
| --- | --- | --- |
| П-N | `FR-N` | Field Rule |
| Е-N | `EV-N` | EVent |
| Е-N.M (шаг) | `EV-N.M` |  |
| М-N | `MSG-N` | MeSsaGe |
| Именованные исходы | остаются **именами** («Скрытие полей») — это названия, не ID; `OUT-N` вводим только если понадобится машинная адресация |  |
| № строки реестра полей | остаётся порядковым номером *отображения* (не идентификатор); идентификатор поля для ссылок — его `FR-N` |  |

Все три префикса проходят анти-омоглифный фильтр (F, R, V, S, G — однозначно латинские).

### Отвергнутые варианты (чтобы решение было осознанным)

- **B: транслитерация 1:1** (П→P, Е→E, М→M) — омоглифы, неотличимы от кириллицы глазом, срыв принципа 2.
- **C: чисто числовая локальная адресация** (`4.1` = правило поля 1, `5.2.9` = шаг) — машинно удобно, для человека нечитаемо и конфликтует с нумерацией разделов.
