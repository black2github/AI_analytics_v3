## Проблема

OpenAPI спецификация микросервиса по поддержке работы с корпоративными картами в системе ДбО банка для юридических лиц занимает почти 15000 строк. Это много и неудобно для анализа с помощью LLM. 
В рамках микросервиса есть разделы, такие как выпуск карт, управление лимитами, изменение статуса карт и просто операции просмотра данных по картам и заявкам. 
Какие есть стандартные способы разделения спецификации на части? Рассмотри их плюсы и минусы.

Есть подозрение, что для спецификации на 15 000 строк, особенно с четкими доменами (выпуск, лимиты, изменение статуса, просмотр карт и заявок), самым эффективным решением будет разделение по бизнес-доменам (Domain-based split).
Это решит проблему анализа LLM и упростит командную работу.

## Вот сравнение основных подходов:

### 🏗️ Разделение по бизнес-доменам (Domain-based Split)

Как работает: Группировка эндпоинтов по URL-префиксам (/cards/issue, /cards/limits, /cards/status). Каждый домен — в отдельную папку со своими схемами и сервисами.

· Плюсы: 

✅ Максимальная изоляция — идеально для LLM (загружаете только нужный домен). 

✅ Параллельная разработка — команды не конфликтуют. 

✅ Масштабируемость — легко добавлять новые модули.
· Минусы: 

❌ Требует начальной настройки правил маршрутизации. 

❌ Может потребоваться кастомизация генерации клиентов.

### 📂 Компонентное разделение (Component-based Split)

Как работает: Техническое дробление по типам (paths/, schemas/, parameters/) с помощью $ref.

· Плюсы: 

✅ Высокий переиспользование схем (например, CardStatus). 

✅ Стандарт — поддерживается большинством инструментов (Redocly, Bump.sh).
· Минусы: 

❌ LLM все еще сложно — для анализа операции нужно собирать много файлов. 

❌ Риск цикличных ссылок. 

❌ Не решает проблему конфликтов при работе над одной схемой.

### 🏛️ Смешанный подход (Рекомендуемый)

1. Физически разбиваем на доменные папки (/card-issuance, /card-limits).
2. Внутри каждой папки используем компонентный метод: схемы в /schemas, эндпоинты в /paths.
3. Результат: Логика домена изолирована, а внутри поддерживается чистота за счет $ref.

### 🛠️ Инструменты и "сложные случаи"

· Инструменты: Redocly CLI (split/bundle), openapi-splitter, openapi-domainify. Redocly CLI — самый надежный для стандартного OpenAPI.
· Важно: Некоторые генераторы кода могут "схлопывать" $ref в анонимные типы, ухудшая читаемость. Проверьте это сразу.

## Итог:

Начните с разделения по доменам, чтобы "подружить" спецификацию с LLM. Внутри доменов используйте компонентный подход для переиспользования схем.

Если нужен совет по настройке $ref для ваших операций — пишите.

## Все домены используют единые структуры данных. Как в таком случае избегать дублирования?

Это классическая проблема «Cross-Domain Reuse». Если просто разложить домены по папкам, вы неминуемо получите дублирование Card или Transaction, что приведет к рассинхрону.

### Вот золотое правило: Общие схемы живут на верхнем уровне, а доменные — внутри своих папок.

📁 Идеальная структура файлов

```
openapi/
├── components/                # ⬅️ ОБЩИЙ КОРЕНЬ (только переиспользуемое)
│   ├── schemas/
│   │   ├── common/            # Базовые объекты
│   │   │   ├── Card.yaml      # (номер, срок, статус-енум)
│   │   │   └── Money.yaml     # (сумма, валюта)
│   │   └── parameters/        # Общие параметры (page, limit)
│   └── responses/             # Общие ошибки (400, 403)
│
├── domains/                   # ⬅️ БИЗНЕС-ДОМЕНЫ
│   ├── issuance/
│   │   ├── paths/             # POST /cards/issue
│   │   └── schemas/           # IssueRequest (использует $ref к общему Card)
│   ├── limits/
│   │   ├── paths/             # PATCH /cards/{id}/limit
│   │   └── schemas/           # LimitSettings (использует общий Money)
│   └── status/                # (аналогично)
│
└── index.yaml                 # Точка входа (сборка)
```

🔗 Как избежать дублей: два подхода к $ref

- **Вариант 1. «Глубокий» реф (для человека)**
    
    Используйте абсолютные пути в $ref, чтобы сразу видеть, где лежит схема:
    
    ```yaml
    # В domains/issuance/schemas/IssueRequest.yaml
    properties:
      card:
        $ref: '../../../components/schemas/common/Card.yaml'
      limit:
        $ref: '../../../components/schemas/common/Money.yaml'
    ```
    
- **Вариант 2. «Плоский» реф (для LLM и сборщиков)**
    
    Сделайте алиасы в index.yaml через components.schemas, тогда везде пишите коротко:
    
    ```yaml
    # В domains/issuance/schemas/IssueRequest.yaml
    properties:
      card:
        $ref: '#/components/schemas/Card'  # Коротко и красиво
    ```
    
    Чтобы это работало, ваш index.yaml должен собирать все через $ref:
    
    ```yaml
    components:
      schemas:
        Card: ./components/schemas/common/Card.yaml
        Money: ./components/schemas/common/Money.yaml
    paths:
      /cards/issue:
        $ref: ./domains/issuance/paths/issue.yaml
    ```
    

#### ⚠️ Главный риск: контекстные расширения

Например, в домене «Выпуск» у карты есть поле delivery_address, а в домене «Статус» — block_reason.
Ошибка: добавлять все поля в общий Card.yaml — он распухнет.

Правильное решение — композиция через allOf:

```yaml
# domains/issuance/schemas/IssuanceCard.yaml
allOf:
  - $ref: '../../../components/schemas/common/Card.yaml'
  - type: object
    properties:
      delivery_address: { type: string }  # Только для выпуска
```

#### 🧠 Для LLM это идеально

· При анализе выпуска — загружаете только issuance/ + общий Card.yaml.
· При правке общей схемы — правите в одном месте, и все домены подхватывают изменения (LLM видит это через $ref).

### 🛠️ Чек-лист для внедрения

1. Вынесите в common только неизменяемые сущности (id, status, balance).
2. Доменные расширения кладите в allOf внутри своей папки.
3. Используйте Redocly CLI (redocly bundle) — он корректно резолвит ссылки и выдаст единый файл для валидации.
