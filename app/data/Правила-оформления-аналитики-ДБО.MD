# Правила оформления аналитики ДБО

Документ фиксирует соглашения, по которым аналитики команд описывают требования.
Часть правил иногда нарушается аналитиками — это считается ошибкой, исправляется
вручную или агентом и относится к цене перехода на Doc as Code.

Общий принцип: требования описываются на **логическом уровне**, а не на физическом.
Наименования сущностей, атрибутов, функций и параметров пишутся на русском языке
(наименования, а не коды). Коды атрибутов могут присутствовать в таблицах для связи
логической и физической моделей и часто заполняются разработчиками.

---

## 1. Нотация именования элементов

### 1.1. Имя логического элемента — квадратные скобки

Название сущности или функции при использовании указывается в квадратных скобках:

```
[Запрос подтверждения]
[Функция отправки кода]
[КК_ВК] Заявка на выпуск карты
```

Поскольку по текущим правилам наименование элемента полностью совпадает с
наименованием страницы (документа) с его описанием, ссылка на имя фактически
является ссылкой на требуемую логическую структуру (wiki-механизм).

### 1.2. Обращение к атрибутам сущности — треугольные скобки

Обращение к атрибуту сущности:

```
[Название сущности].<атрибут сущности>
```

Треугольные скобки позволяют использовать в наименовании атрибута несколько слов:

```
[КК_ВК] Заявка на выпуск карты.<дата заявки>
```

Если атрибут является объектом, обращение по иерархии оформляется через символ `.`
(точка) и указание наименования вложенного атрибута в треугольных скобках.

### 1.3. Обращение к параметрам функции

Входные параметры:

```
[Название функции].<Входные параметры>.<параметр_1>
```

Исходящие (возвращаемые) параметры:

```
[Название функции].<Исходящие параметры>.<параметр_1>
```

**Важно для различения данных и функций.** Обращение к возвращаемому значению
функции следует писать именно как `[Функция].<Исходящие параметры>.<параметр>`,
а не как `[Сущность].<атрибут>`. В старых описаниях встречается обращение вида
`[Сущность].<атрибут>` к выходному параметру функции — это считается устаревшим
и вводящим в заблуждение.

На практике обращение к функции распознаётся по предшествующему слову-маркеру:
«вызывается», «вызов», «обращение» и т.п. Установка входных параметров обычно
выглядит как:

```
вызывается функция [префикс] Наименование функции с параметрами:
a1 = параметр_1; a2 = параметр_2
```

Асимметрия (для входных параметров маркер-слово, для выходных — явная конструкция
`<Исходящие параметры>`) существует из-за привычки экономить усилия. Ошибок в этой
части почти не возникает: разработчики оперативно определяют, о каком параметре речь.
При необходимости распознавание автоматизируется агентом.

### 1.4. Присваивание значения атрибуту сущности

Когда сущности модели данных присваивается значение:

```
[КК_ВК] Заявка на выпуск карты.<дата заявки> = текущие дата и время
```

## 2. Операция «найти/взять запись» — стрелка →

### 2.1. Назначение

Символ `→` (синоним `->`) означает **найти / взять запись**. Используется для
манипулирования сущностями модели данных.

Договорённость: для иных целей (например, показ переходов меню) следует
использовать стрелку другого вида, например `=>`, чтобы не пересекаться
с операцией поиска записи.

### 2.2. Полный синтаксис

```
операнд_A → Операнд_B
```

### 2.3. Операнды

- **операнд_A** — ключ (чаще всего скалярный, например строка), по которому
  происходит поиск записи сущности `Операнд_B`.
- **Операнд_B** — сущность / запись / объект, поиск которого производится по ключу
  `операнд_A`. Предполагается, что в `Операнд_B` есть атрибут `операнд_A`, и его
  значение позволяет однозначно идентифицировать конкретную запись.

### 2.4. Приоритет

Операция `→` имеет более высокий приоритет, чем:
- присваивание `=`
- логические операции `==`, `≤`, `≥`, …

Операции `.` (обращение к атрибуту) и `→` (поиск записи) имеют **одинаковый**
приоритет и применяются последовательно слева направо, поэтому скобки обычно
не ставятся.

### 2.5. Примеры

```
операнд_A → Операнд_B.<атрибут>
```
Взять значение атрибута `<атрибут>` сущности `Операнд_B`, найденной по ключу `операнд_A`.

```
XX.a1 → YY.b1
```
Значение атрибута `b1` сущности `YY`, найденной по ключу `XX.a1`.
С расставленными для читаемости скобками: `((XX.a1) → YY).b1` — так как `.` и `→`
имеют один приоритет, скобки не ставят, операции идут последовательно.

## 3. Описание условий — паттерн «Если … то … иначе …»

Условия описываются по шаблону:

```
Если <условие> то <действие> иначе <действие>
```

## 4. Типы требований

Каждый тип требований оформляется по своему шаблону (отдельная Confluence-страница).
Одна страница (документ) содержит требования только одного типа.

Известные типы:
- процессы (process)
- диаграммы состояний (states)
- модели данных (dataModel)
- функции (function)
- интеграционные методы (integration)
- экранные формы (screenListForm, screenItemForm)
- печатные формы (printForm)
- контроли / проверки (control)
- нотификации (notification)

Для конкретного типа может быть несколько страниц (например, по странице на каждую
экранную форму, по странице на каждую сущность модели данных).

### 4.1. Дочерние структуры

Если набор требований семантически относится к родительской структуре, описание
дочерних структур оформляется как дочерние страницы. Примеры: вкладки экранной
формы — на дочерних страницах к форме; вложенные сущности модели данных
(например, анкета держателя карты внутри заявки на выпуск карты) — на дочерних
страницах к родительской сущности.

## 5. Цветовая маркировка изменений (текущий процесс Confluence)

Все изменения/добавления на странице делаются цветом, соответствующим текущей
задаче Jira (например, GBO-12345 — сиреневый, GBO-567890 — зелёный). После
ввода в эксплуатацию (статус задачи «Установлено на ПРОМ») аналитик сбрасывает
цвет в чёрный.

Следствие: всё **чёрное** = действующий функционал в промышленном контуре
(утверждённые требования). Всё **цветное** = изменения в рамках незавершённой задачи.

При переходе на Doc as Code этот механизм заменяется полем `status` во frontmatter
(`approved` / `draft`) и историей Git.

---

## 6. Связь с механизмом межсервисных ссылок (Doc as Code)

Для различения «ссылки на данные» и «ссылки на действие» при резолвинге внешних
ссылок (см. отдельную проработку) синтаксис этого документа служит сигналом:

- `[Сущность].<атрибут>` и конструкции с `→` — обращение к **данным**
  (резолвится в публичную карточку сущности).
- `[Функция].<Исходящие параметры>.<…>` и маркеры «вызывается/вызов/обращение»
  — обращение к **действию** (резолвится в swagger-операцию).

Авторская форма внешней ссылки использует фигурные скобки `{{сервис:элемент}}`
(не квадратные — чтобы не пересекаться с нотацией имени `[Имя]` и markdown-ссылками).
