# Модель контроля экранной формы (FE-контроли, Зона 3)

> **Канонический источник.** Этот файл — единственный источник истины по модели
> контролей (состав строки, `rule`/`visibility`, режимы `R`/`V`, критерий
> видимости). В остальных reference-файлах — только краткие напоминания со
> ссылкой сюда; правь модель здесь.

Контроль — спецификация **централизованной проверки**. Он либо пройден, либо нет;
что делать с результатом, решает вызывающая сторона. Задача документа контролей —
не описать поведение формы, а задать проверки так, чтобы их можно было
отревьюить и реализовать.

## Что содержит строка контроля (и только это)

1. **Поле / атрибут** — объект проверки (если применимо).
2. **Название проверки** — её суть/цель. Обязательно (см. ниже, как формировать).
3. **Условие** — способ реализации, в форме **Если … то … иначе …**; исход каждой
   ветки — код/сообщение (для `rule`) либо «поле видимо»/«поле невидимо» (для
   `visibility`).
4. **Событие** — группа(ы) триггеров, в рамках которых проверка выполняется
   (отметки `V`), и/или режим `R` (реактивный).

## Что в контроль НЕ входит (частые ошибки)

В строке контроля — и особенно в колонке «Условие» — **запрещено**:

- **Логика получения значения поля.** «Пока не выбрана организация…», «значение
  = [Клиент Банка].<…>», «заполняется из справочника…» — это способ заполнения
  поля, живёт в форме, не в контроле.
- **Реакция на исход.** «Подсвечивается красным», «под полем отображается
  ошибка», «поле очищается», «поле блокируется», «открывается модальное окно» —
  это реакция. Контроль лишь проверяет условие и формирует код/сообщение при
  непрохождении; где и как обработать результат — решает запросившая сторона.
  Единые правила реакций сервиса ведутся отдельно (для КК —
  [[КК] Правила информирования пользователя Клиента об ошибках](../../../../conf-requirements/cc/[КК]-Корпоративные-Карты/[КК]-Библиотека-модулей-Сервиса/[КК]-Правила-информирования-пользователя-Клиента-об-ошибках.md)); чего не
  хватает — дописывается туда.

Правило-детектор: если фрагмент отвечает на вопрос «**как поле получает
значение**» или «**что произойдёт при провале**» — он не в контроле.

## Название проверки — как формировать

Краткая формулировка того, **что** проверяется и **зачем**, по которой ревьюер
соотнесёт проверку со способом реализации. Не пересказ условия и не `SF-NN`.

- Хорошо: «Проверка, что клиент является резидентом»; «Обязательность поля
  Организация»; «Соответствие валюты лимита валюте счёта»; «Видимость блока
  Реквизиты ГОЗ»; «Наличие недозаполненной заявки».
- Плохо: «SF-05»; «дневной ≤ месячный» скопированное из условия; «Проверка поля».

Форма — именная («Обязательность …», «Соответствие …», «Видимость …») или
глагольная («Проверка, что …»). Для `visibility` название начинается с
«Видимость …».

## Условие — всегда Если-то-иначе

Условие структурируется как **Если … то … иначе …**; исходы раскладываются по
веткам. Для видимости: «отображается» → ветка «то», «не отображается» → ветка
«иначе». Тексты условий — **дословно** из исходника (техническая нормализация
ссылок/`<…>`/пробелов допустима; усечение `[:N]` и «…» — нет).

**Форматирование** (references/formatting.md): «Если/то/иначе» — жирным; шаги
«**Шаг N**» — с новой строки; смысловые блоки разделяй `<br>`, не сливай в поток.

## Тип контроля: `rule` и `visibility`

- **`rule`** — проверка с широким спектром исходов: каждая уникальная строка ≈
  длинный код ошибки. Имеет «Сообщение об ошибке».
- **`visibility`** — тот же контроль, но с ровно **двумя** исходами: «поле видимо»
  / «поле невидимо». Сообщения об ошибке нет (`—`) — провалиться нечему; исход
  сообщается **прямо в Условии** (в ветках то/иначе).

Иных типов нет. `behavior` (реакции, модалки, недоступность, плашки) — **не
контроль**; см. «Что в контроль НЕ входит».

## Режимы срабатывания: `V` и `R`

- **`V` (событийный)** — проверка выполняется в рамках **группы триггеров**
  (отметка `V` на пересечении со столбцом группы). Барьер: не пройдена → вызывающий
  процесс дальше не идёт.
- **`R` (реактивный)** — проверка выполняется при **первичной отрисовке формы** и
  при **изменении любого поля из колонки «Поля»**. Своей событийной группы у чисто
  реактивного правила нет.
- **Сочетание `R` + `V`** — у валидаций-гарантий: ранняя реакция при вводе (`R`)
  плюс барьер по кнопке (`V`). Одна строка, один контроль, два режима срабатывания
  — та же логика сосуществования, что у пары «FE-ранняя-реакция / BE-гарантия»,
  только внутри самой формы. Причина: реактивная проверка не гарантия — значения
  могли прийти программно (импорт, автозаполнение), пользователь мог
  проигнорировать раннюю реакцию.
- **Чистая видимость — только `R`.** Она непрерывно поддерживается, а не
  «проверяется при сохранении»; `V` у неё нет, сообщения нет.
- **Чисто событийные (только `V`)** — когда реактивно гонять дорого или
  бессмысленно: проверка против внешнего состояния, которое в процессе ввода не
  меняется (валюта счёта), или тяжёлый вызов справочника/бэка (тариф).

**Как проставить режим (кратко):** `visibility` → только `R`. `rule` → `V` в своей
группе (по кнопке/действию); добавь `R`, только если ошибку нужно показать **сразу
при вводе** и все участники условия — поля этой формы. Проверка против внешнего
состояния или дорогой вызов — `V` без `R`.

`R` — **отдельная колонка перед группами**, не значение внутри групповой колонки:
у `R` нет «своей группы», смешение заставляло бы гадать, к какому событию `R`
относится.

## Группы триггеров

Событийные (кнопка/действие формы), объявляются **над таблицей** с расшифровкой
(что за событие). Нумерация/именование — **согласованно с существующим документом
контролей сервиса**; при миграции страницы из Confluence переиспользуй исходную
нумерацию групп, чтобы сшить с уже мигрированными контролями. Пример: «Группа №1 —
кнопка "Продолжить"», «Группа №7 — загрузка реестра».

## Формат таблицы

```
| ID | Ориг. | Название проверки | Тип | Поля | Условие | Сообщение об ошибке | R | Гр.1 | Гр.2 | … |
```

- `Ориг.` — исходный № поля/строки Confluence (прослеживаемость); опционально.
- Столбцов групп — столько, сколько событийных групп у формы; отметка `V` на
  пересечении.
- `visibility`-строки: «Сообщение об ошибке» = `—`; как правило `R` без `V`.
- Нотация «Поля» для `visibility`: **`Объект ← зависимости`** — что показывается
  (объект видимости) и от чего зависит (триггеры пересчёта). Объект может не быть
  триггером (видимость поля «Согласовано» зависит от «Лимит» и «Порог», а не от
  себя). Если стрелка неочевидна — можно развести на две колонки «Объект» /
  «Зависит от».

## Механический критерий: видимость в колонке «Видимость» vs в контролях

Видимость **по умолчанию живёт в колонке «Видимость» формы** — у поля, где её ищет
читатель (свойства поля читают у поля). Выносится в контроли строкой `visibility`
**только** при выполнении **любого** из трёх **счётных** условий:

1. **Ветвление** — условие содержит более одного «если» / вложенность.
2. **Разделяемость** — одно и то же **нетривиальное** условие видимости у
   нескольких полей/блоков (вынос устраняет дублирование — то же соображение,
   что для групп контролей).
3. **Внешнее состояние** — условие ссылается **не на поля этой формы**, а на
   справочник / атрибут сущности / полномочия.

Иначе — остаётся у поля. **Число зависимостей само по себе не критерий**: одно
условие с двумя полями-триггерами («показано, если лимит > порога») —
тривиальное, живёт у поля.

Две оговорки против ошибок счёта (важны для механического применения):

- **Тривиальная разделяемость не выносится.** Одно составное условие вида
  «**Если** \<поле формы\> == X, **то** отображается, **иначе** не отображается»
  (ровно одно «если», без вложенности, без ссылок на внешнее состояние) **остаётся
  в колонке «Видимость» каждого поля**, даже если оно повторяется у нескольких
  полей. Критерий 2 применяется, когда общее условие **само по себе** нетривиально
  (критерий 1 или 3) — тогда вынос устраняет дублирование сложного текста.
- «если A **и** B» — это **одно** «если» (составное условие), не ветвление.
  Ветвление — это вложенность или второе «если»: «если … иначе если …» либо «если»
  внутри ветки то/иначе.
- Критерий 3 смотрит, **на что ссылается само условие видимости**, а не откуда
  поле взяло значение. «Если Тип счёта == 2» → Тип счёта — поле формы (внутреннее,
  не выносим). «Если [Клиент Банка].<Статус> == Действующий» → прямая ссылка на
  атрибут сущности (внешнее, выносим).

Каждое условие проверяется механически, без оценочных суждений: посчитать «если»;
посчитать поля-владельцы условия; определить, все ли участники — поля формы.

**Оговорка про среду рендеринга.** Размерность экрана (Метод определения
размерности, XL/L/M/S/XS) — **не внешнее состояние**, а среда рендеринга; видимость
по размеру остаётся у поля в колонке «Видимость». (Практика сервиса: под XL–M и
S–XS часто делают отдельные ЭФ, ссылаясь на условия друг друга во избежание
дублирования.)

**Смешанное условие.** Если в одном условии есть и размер экрана, и поля формы, и
ссылка на атрибут сущности — оно **выносится** по критерию 3 из-за атрибута (размер
сам по себе не выносит, но и не отменяет вынос). Пример: «Если (размер XL/L/M) И
(«Кому выпустить карту» == «Себе»/«Сотруднику») И ([Сотрудник клиента].<Фамилия> и/или
<Имя> пустые), то показать» → в контроль (ссылка на атрибуты `Сотрудник клиента`).

**Машиночитаемость `R` при этом не теряется** — «место кодирует режим»: видимость
в колонке «Видимость» **по определению реактивна** (пересчитывается при изменении
упомянутых полей и при отрисовке), маркер `R` там не нужен. `R` как явная пометка
остаётся только у строк в контролях. У поля — реактивный инвариант; в контролях —
вынесенное правило с явными режимами.

## Пример (лимиты по карте)

Группы: **№1** — «Сохранить»; **№2** — «Применить ко всем картам счёта».

| ID | Название проверки | Тип | Поля | Условие | Сообщение об ошибке | R | Гр.1 | Гр.2 |
|----|-------------------|-----|------|---------|---------------------|:-:|:-:|:-:|
| SF-01 | Дневной лимит не выше месячного | rule | Дневной лимит, Месячный лимит | Если дневной лимит ≤ месячный лимит, то проверка пройдена, иначе ошибка | Дневной лимит не может превышать месячный | R | V | V |
| SF-02 | Соответствие валюты лимита валюте счёта | rule | Код валюты лимита ↔ Валюта счёта карты | Если валюта лимита совпадает с валютой счёта карты, то проверка пройдена, иначе ошибка | Валюта лимита должна совпадать с валютой счёта карты |  | V | V |
| SF-04 | Видимость блока «Реквизиты ГОЗ» | visibility | Блок «Реквизиты ГОЗ» ← Тип счёта | Если тип счёта == 2 (счёт ГОЗ), то блок отображается, иначе не отображается | — | R |  |  |

- **Только `V`** (SF-02): сверка с внешним состоянием (валюта счёта в процессе
  ввода не меняется) — реактивно гонять незачем.
- **Только `R`** (SF-04): видимость **блока** — условие разделяют поля блока
  (критерий 2), поэтому вынесена в контроль; поддерживается непрерывно, `V` и
  сообщения нет.
- **`R` + `V`** (SF-01): ранняя реакция при вводе + барьер по кнопкам.
