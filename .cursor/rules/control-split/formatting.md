# Форматирование таблицы контролей

## Треугольные скобки в именах атрибутов

Треугольные скобки `<…>` — не обязательный синтаксис имени, а инструмент
устранения двусмысленности. В тексте проверки **проверяемый** атрибут адресуется
словом «реквизит»/«атрибут», а **второй** участник проверки обязан называться явно
— его и берут в `<…>`. Когда путаницы нет, скобки опускают.

Практика:

- В колонке «Проверяемый атрибут» имя давай **без** скобок — там и так ясно, что
  это проверяемый атрибут (`Буквенный код валюты счёта`, не `<Буквенный код…>`).
- Внутри текста условия скобки у **второго** атрибута **сохраняй** — это значимое
  разрешение двусмысленности, не шум.
- Не навязывай скобки везде; убирай их там, где они избыточны.

## Имена атрибутов дочерних/связанных сущностей

- Имя атрибута дочерней/связанной сущности указывается **без** префикса в виде
  имени родительской сущности (просто `Серия документа`, не
  `Держатель карты.Серия документа`).
- Если в проверке требуется атрибут **родительской** сущности, его указывают явно
  в нотации: `<родительская_сущность>.<атрибут_N>`.

## Объединение ячеек

- Ячейки с **одинаковыми именами атрибутов** в соседних строках объединяй через
  `rowspan` (как в исходнике) — несколько проверок одного атрибута идут под одним
  объединённым именем.
- При описании в одной таблице проверок **разных сущностей** имя сущности
  занимает строку на всю ширину (через `colspan`), разделяя блоки.
- Если таблица описывает проверки **одной** сущности, имя сущности в таблице
  **не используется вовсе** — оно уже указано в названии файла.

## Колонки

Документ сущности (`data-model`):
`ID | Ориг. | Тип | Проверяемый атрибут | Название проверки | Условие (кратко) |
Сообщение об ошибке | Гр.0 | Гр.1 | … | Гр.N`

Документ формы (`screen-form`):
`ID | Ориг. | Поле (атрибут) | Название проверки | Сообщение об ошибке |
Гр.X | Гр.Y` (только группы-события этой формы/страницы).

- `Тип` — `field` (ограничение значения атрибута) или `rule` (межатрибутное /
  межсистемное правило).
- `Ориг.` — исходный № Confluence-страницы для прослеживаемости.
- Условие — **дословно из исходника**, без перефразирования и усечения (`[:N]`, «…»).
  Сохраняй пошаговую структуру (вложенные «если-то-иначе» через `<br>`/маркеры), не
  склеивай в одну строку. Сжатие «до сути» — отдельный ручной шаг аналитика, не часть
  разделения.


## Именование выходных файлов

**Базовый принцип.** Имя нового файла строится из **оригинального имени файла**
(исходного файла контролей — для BE; файла экранной формы — для FE) с
**минимальной правкой**: вставляется маркер `BE-`/`FE-Контроли`, остальное имя
сохраняется как есть. Соблюдай общую дисциплину именования: **без пробелов**
(разделитель — дефис), расширение `.md`, чтобы имя годилось для URL-ссылок без
энкодинга и читалось человеком. Не пересобирай имя из «логического» названия —
бери оригинальное имя файла и правь минимально.

> Замечание про выгрузку: при загрузке в среду квадратные скобки в имени могут
> быть заменены на `_` (например `_КК_ВК_-…`). Это артефакт выгрузки — в
> формируемых именах используй каноничный формат с `[ ]`, как в `doc_id` и
> ссылках внутри файлов (`[КК_ВК]-…`).

- **Файл контролей модели данных (BE):** в оригинальное имя исходного файла
  контролей добавляется `BE-` перед словом «Контроли».
  Пример: `[КК_ВК]-Контроли-заявки-на-выпуск-карты.md` →
  `[КК_ВК]-BE-Контроли-заявки-на-выпуск-карты.md`.
- **Файл контролей экранной формы (FE):** строится от оригинального имени файла
  **экранной формы** (берётся из ссылки в описании соответствующей группы
  триггеров), в начало добавляется `FE-Контроли`. Если в начале имени есть
  префикс в квадратных скобках, `FE-Контроли` ставится **после** него.
  Пример: `[КК_ВК]-ЭФ-Клиента-страница-Анкеты.md` →
  `[КК_ВК]-FE-Контроли-ЭФ-Клиента-страница-Анкеты.md`.

**Имя файла — главное; `title` и `doc_id` согласуй с ним.** Они строятся из той же
основы (маркер `BE`/`FE Контроли` + исходный заголовок) и отличаются только
оформлением: `title` — пробелы вместо дефисов и обычная пунктуация; `doc_id` —
`{{<service>: <текст title без внешних кавычек>}}`. Не задавай `title` отдельным
форматом («Проверки сущности — …»): рассинхрон имени файла, `title` и смарт-ссылок
ломает навигацию и редактирование. Шаблоны — references/templates.md.

Новые файлы помещаются в **тот же каталог**, где лежит обрабатываемый оригинальный
файл контролей (при работе в IDE — например, PyCharm — в ту же папку дерева
проекта). Это сохраняет относительные ссылки и порядок в дереве. Если каталог в
конкретной ситуации не важен — это не критично.

## Подразделы по сущностям в многосущностном BE-файле

Когда в одном data-model файле описаны несколько сущностей, в разделе «Контроли»
под каждую сущность создаётся **отдельный подраздел** с её именем:

```
## Контроли
### [КК_ВК] Заявка на выпуск карты и открытие счета
<таблица>
### [КК_ВК] Выпускаемая карта
<таблица>
### [КК_ВК] Держатель карты
<таблица>
```

Нумерация ID и номера групп — **сквозные** по всем подразделам. Подразделы
заменяют строки-разделители внутри одной таблицы (используй подразделы, а не
строку `colspan` с именем сущности, когда сущностей несколько).

## Имя экранной формы — из описания триггера

В FE-документе «Назначение» и врезка ссылаются на **экранную форму**, а не на
функции. Источник имени формы — описание соответствующей **группы триггеров** в
исходном файле (там форма уже названа со ссылкой, напр. «переход со страницы
[ЭФ Клиента: страница "Анкеты"](…)»). Бери имя и ссылку формы оттуда; не
подставляй ссылки на функции создания/редактирования — это разные сущности.

Если у страницы **нет группы-перехода** (терминальная страница, см.
multi-entity.md), бери имя и ссылку формы из её **строки-агрегата** «обязательные
поля страницы …» или из описания страницы в теле документа — там она тоже названа
со ссылкой. Не оставляй FE-документ без канонической ссылки на форму.
