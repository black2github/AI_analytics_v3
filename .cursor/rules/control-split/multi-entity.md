# Несколько сущностей и несколько форм

## Несколько связанных сущностей — один data-model файл

Иногда в исходном файле контролей описаны проверки не одной, а нескольких
**связанных** сущностей (иерархия вложенности: заявка, в ней выпускаемая карта, в
ней держатель карты, в нём документы и вложения). В этом случае:

- допустимо использовать **один** data-model файл на все связанные сущности;
- номера групп (триггеры) — **единые** по всему файлу. Это ключевое условие.

Логика: срабатывание триггера запускает проверки по всему документу, частью
которого являются связанные сущности. Идёт бизнес-процесс, срабатывает триггер, и
проверяются все сущности модели данных, помеченные этой группой. Иерархия
сущностей **не порождает** собственные группы — группы общие по всему бэковому
документу. Именно поэтому один файл на несколько сущностей корректен.

Оформление:

- каждая сущность описывается **плоской таблицей** по тем же правилам;
- **сквозной уникальный ID** на каждую проверку (продолжаем нумерацию `DM-NN`
  через все сущности);
- имя сущности — строкой-разделителем на всю ширину таблицы (`colspan`), чтобы
  при чтении было видно, к какой сущности относится блок;
- колонки групп — те же (0…N) для всех сущностей.

## Критерий одного файла: агрегат и x-controls (API First)

Решающий признак «один файл vs несколько» — не таблица исходника, а **агрегат
записи**. Несколько сущностей идут в ОДИН data-model файл (со сквозными `DM-NN` и
едиными группами), если они образуют **агрегат** (aggregate root с вложенными
под-ресурсами), у которых **нет собственного контракта записи** в API.

- **Признак из OpenAPI.** Под-сущности вложены в DTO корня (например,
  `issuedCards[]`, `documents[]` внутри `IssueCardRequestDto`), а операции над ними
  требуют `id` корня и пишут в него (импорт/обогащение реестром требует id заявки,
  а не создаёт держателей отдельным каналом). Это и есть один контракт.
- **Отдельные эндпоинты ≠ отдельные документы.** Несколько операций над одним
  агрегатом — это **разные триггеры**, отбирающие проверки **по группе**, а не повод
  дробить документ. Группы остаются сквозными по всему файлу.
- **x-controls при нескольких операциях.** Все операции-триггеры ссылаются на ОДИН
  `source` (этот BE-файл), различаясь только `group`. Раздел «Связь с OpenAPI»
  перечисляет **все** триггеры, а не один: создание/сохранение — группы статусов
  (DRAFT/NEW); импорт реестра — группа импорта; подписание/ФЛК — группы подписи;
  загрузка вложений — группа файлов.
- **Связывай с реальной спецификацией, не выдумывай.** `operationId` и путь каждой
  операции бери из **OpenAPI-спецификации проекта** — найди операцию по смыслу триггера
  (создание/сохранение, импорт реестра, подпись/ФЛК). Выдуманный правдоподобный
  `operationId` не сопоставится с операцией и делает x-controls бесполезным. Если
  подходящей операции в спецификации нет — поставь
  `operationId: TODO — не найдено в спецификации (<что искал>)` и пометь для аналитика,
  а не подставляй фиктивное имя. `source` — смарт-ссылка на сам BE-документ.

```yaml
# один source, разные group — по одной записи на операцию-триггер
post: { operationId: createOrUpdate, x-controls: { source: "{{…BE…}}", group: [0, 1] } }
---
post: { operationId: importRegistry,  x-controls: { source: "{{…BE…}}", group: 4 } }
---
post: { operationId: sign,            x-controls: { source: "{{…BE…}}", group: [2, 3, 9] } }
```

- **Исключение (когда дробить).** Если под-сущность имеет **самостоятельный контракт
  записи** — свой `create`/`update`-эндпоинт со своим телом запроса, не требующий
  id корня, — у неё отдельный документ контролей и собственный `source` в
  x-controls. Read-операции (получить/сравнить) контролей не несут и на выбор не
  влияют.

## Несколько экранных форм — отдельные файлы

То же по смыслу справедливо и для экранных форм, но встречается реже: изменение
полей одной формы обычно не меняет поля другой. Поэтому **число screen-form файлов
по умолчанию равно числу описанных экранных форм/страниц**.

Если форма состоит из нескольких страниц-вкладок с собственными переходами
(событиями-группами), делай **отдельный файл на каждую страницу**, перенося в него
проверки соответствующей группы-перехода. Каждый файл — со своими группами
триггеров (переходы именно этой страницы).

### Единица дробления FE — страница, а не группа-триггер

Дроби screen-form файлы **по страницам, а не по группам**. Одна страница может нести
**несколько** групп-триггеров (например, на странице есть и проверки при входе/
заполнении, и проверки при переходе с неё дальше — это разные группы-столбцы, но обе
принадлежат **одной** странице с одним набором полей). Все группы одной страницы идут
в **один** файл этой страницы — **несколькими столбцами групп рядом** (Гр.X, Гр.Y), а
не в разные файлы по столбцу на группу.

Признак «одна страница»: группы — это события **одного и того же** набора полей (входы,
изменения, переход именно с этой страницы). Тогда — один FE-файл с несколькими
столбцами групп. Разводи на разные файлы только когда группам соответствуют **разные
страницы** (разный набор полей). Это типичная точка расхождения: группы 5 и 6 одной
страницы правильно свести в **один** файл с двумя столбцами, а не раздробить на два
файла «по группе».

### Страница без группы-перехода (терминальная страница)

Последняя страница мастера часто **не имеет** «Продолжить»-перехода: её событие —
действие жизненного цикла («Подписать и отправить», «Отправить в банк»), которое
запускает **бэковые** группы (подпись/ФЛК), а не отдельную форменную группу-столбец.
Типичный признак: в исходнике у форменных контролей этой страницы (например, агрегат
«обязательные поля страницы») стоят только бэковые группы, а столбца-перехода для
неё нет.

Такую страницу **всё равно оформляй отдельным FE-файлом** — не пропускай и не
сливай с предыдущей (иначе её форменные проверки теряются, а разные модели расходятся:
кто-то файл создаёт, кто-то нет). Правила оформления:

- В разделе «Группы триггеров» опиши **реальное событие** — кнопку действия
  («Подписать и отправить»), а не выдуманный переход вкладки.
- Если у страницы нет ни одной форменной группы-столбца, **опусти столбцы групп** в
  таблице и укажи единый триггер словами (в шапке/врезке). Не подставляй чужие
  группы.
- Бэковые последствия действия (проверки атрибутов при подписи/ФЛК) живут в
  **BE-документе** под соответствующей группой; на форме остаётся только проверка
  полей страницы (обычно агрегат «обязательные поля страницы …»).

## Правило импорта реестра

Импорт реестра (excel/csv с массивом записей, например держателей карт) — это
**ускорение ручного ввода**, а не отдельный канал создания сущностей. Данные из
реестра ложатся на экран, правятся пользователем и идут стандартным путём к
созданию/сохранению сущностей. Поэтому:

- проверки **значений** (длина, формат, обязательность, справочник) растворяются
  в обычных файлах — формы или сущности — по объекту проверки;
- импорт при этом остаётся **самостоятельным триггером**: сохраняй его как
  отдельный столбец-группу в тех файлах, где контроль и так присутствует по
  объекту (охват проверок при импорте может не совпадать с охватом при ручном
  вводе);
- проверки **формата самого файла реестра** (маппинг колонок, расположение полей,
  расширение, размер, структура) — это не поле экрана и не атрибут сущности, а
  проверка структуры входного файла функции импорта. Относи их в **бэковый**
  (data-model) файл под группой импорта с пометкой объекта «структура файла
  реестра», либо — если их немного — оставь в описании самой функции импорта.

Отдельный файл контролей под импорт **не создавай**.
