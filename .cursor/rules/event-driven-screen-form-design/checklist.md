# Чеклист финальной сверки и инварианты предпроверки

Прогоняется перед сдачей результата. В forward-режиме отчёт по инвариантам
прикладывается к передаче на ревью (человек проверяет смысл, не структуру).
Определения принципов — SKILL.md; правила структуры и шагов — templates.md;
здесь — только проверки.

## Смысл / заземление (ВЫСШИЙ ПРИОРИТЕТ)

- [ ] (reverse) Каждый фрагмент исходника перенесён как полный смысловой
      эквивалент; неуверенность помечена `[НЕ РАСПОЗНАНО: <дословно>]`
      (принцип — SKILL.md).
- [ ] (forward) Каждый шаг события соответствует правилу заземления SKILL.md:
      есть основание-источник либо `уточняется`; открытые вопросы — вне
      документа, inline-TODO нет.
- [ ] Каждый вызов функции/метода — с инициализацией входных параметров
      (принцип — `@.cursor/rules/_shared/function-calls.md`); «голых» вызовов нет.

## Инварианты структуры (машинная предпроверка)

1. [ ] **Поле ↔ FR-N**: каждое поле §2 «Реестр полей» имеет подраздел FR-N и
       каждый FR-N — строку в §2. Допущение (templates.md §4): пара однотипных
       служебных полей может разделять один FR-подраздел — тогда обе строки
       реестра указывают на него.
2. [ ] Каждый блок из §2 упомянут в §3 «Видимость блоков».
3. [ ] **Событие ↔ подраздел**: каждое событие реестра §5 имеет детальный
       подраздел и наоборот. EV-1 «Открытие формы» присутствует.
4. [ ] **Достижимость и завершимость шагов**: каждый шаг, кроме первого в
       своём событии, упомянут хотя бы в одном переходе или диапазоне; каждая
       ветвь каждого шага оканчивается переходом на `EV-N.M`, «конец»,
       именованным исходом или показом MSG-N (+ исход); переходы — только
       явными ID, формулировок «следующий шаг» нет. Исключение (reverse):
       шаг, недостижимый уже в исходнике, — допустим при явной пометке дефекта
       исходника (templates.md §5).
5. [ ] **Стабильные ID шагов** соответствуют правилам templates.md §5:
       у каждого шага ID `EV-N.M` жирным, второй (markdown) нумерации нет;
       ID уникальны, следов перенумерации/переиспользования нет; **все ссылки
       на шаги резолвятся** в существующие ID (переходы, диапазоны, «Показ» в
       MSG, кнопки «Повторить …»); для каждого диапазона проверено, что
       физически лежащие между граничными ID шаги действительно входят в
       переиспользование (иначе — оговорка-исключение).
6. [ ] **Сообщение ↔ показ**: каждое MSG-N имеет непустой «Показ» (при
       неопределённом/противоречивом условии показа в исходнике —
       `уточняется` с причиной, templates.md §6); каждое сообщение, упомянутое
       в шагах/правилах, есть в реестре §6; тексты сообщений вне §6 не
       дублируются.
7. [ ] Каждый вызов внешнего метода: **в комплекте** — ссылка на карточку
       EXTINT или заглушку с пометкой; **вне комплекта** — исходная ссылка.
       Выдуманных методов и голых упоминаний нет (политика — SKILL.md).
8. [ ] Каждый упомянутый контроль/группа: в комплекте — резолвится в
       CTL-NNN / CTL-GRP-N; вне комплекта — исходная ссылка. Текст правил
       контролей не дублируется.
9. [ ] **Исходы**: каждый использованный именованный исход определён; каждый
       определённый — использован.
10. [ ] Каждый локальный ID, упомянутый в тексте (FR-N, EV-N, EV-N.M, MSG-N),
        существует в документе.
11. [ ] (в комплекте) Ссылки на артефакты (ENT/EXT/EXTINT/FUN/PRC/STM/CTL/SCR)
        ведут на существующие файлы/якоря.
12. [ ] **Элементы данных ↔ FR (двусторонняя сверка с реестром DE)**: каждый
        атрибут «Элемент данных:» в FR ссылается на существующий DE-NNN;
        каждая связь колонки «Представления» реестра имеет FR с этим DE;
        тип связи соответствует способу заполнения («вводится» — Вручную /
        Выбор из списка; «отображается» — Автоматически). У каждого поля,
        чьё значение сохраняется в сущность, атрибут «Элемент данных» есть;
        презентационные поля — без него (спорные случаи — в отчёт).
13. [ ] **Межд документные ссылки на локальные ID — только полным адресом**
        `<id документа>/<локальный ID>` (например `SCR-CL-03.1/FR-2`); голых
        FR-N/EV-N/MSG-N вне их документа нет.
14. [ ] **(только конвейер, фаза 1) Кандидаты в контроли**: маркеры `CAND-N` уникальны в
        документе и стоят **только у проверок полевой природы** (валидация
        при вводе / барьер группы — templates.md, «Фаза 1»); у проверок-шагов
        процедур маркеров нет (они остаются шагами: ссылка на карточку, если
        проверка описана в контролях, иначе — упоминание по имени без ссылки);
        не помечаются также формат / видимость / обязательность. Каждый маркер
        содержит название-суть; условие в маркере не дублируется; сверка всех
        встроенных проверок с общим файлом выполнена.
15. [ ] **(только конвейер, после фазы 3) Замыкание выполнено**: маркеров `CAND-N` и пометок
        «исх. № … будет замкнута» не осталось; каждому бывшему кандидату
        соответствует ровно одна карточка FE-файла (по колонке «Ориг.»);
        все ссылки формы на контроли ведут в FE-файл(ы) — ссылок на BE-файл
        и на общий файл контролей нет.
16. [ ] **(прямая привязка, вариант по умолчанию)**: маркеров `CAND-N` и
        пометок «будет замкнута» в документе **нет вовсе**; каждая ссылка
        `CTL-NNN`/`CTL-GRP-N` резолвится в реестр/карточку каталога контролей
        (якорь — из фактического заголовка); барьерные шаги ссылаются на
        группы `CTL-GRP-N`; обязательность §2 — на карточку обязательности
        вкладки; ненайденные проверки — inline дословно, кандидаты в новые
        CTL перечислены в отчёте/open-questions, не в документе; реестр и
        карточки CTL не изменены.

## Нотация и идентификаторы

- [ ] Текст соответствует авторитету — `@.cursor/rules/_shared/formatting.md`
      (жирные Если/то/иначе; `==`/`≠` в сравнениях,
      `=` в присваиваниях; `→` только «найти/взять запись»); нотационные
      ошибки эталона/исходника при переносе исправлены.
- [ ] Все **идентификаторы** — латиница (`FR`/`EV`/`MSG` и артефакты);
      кириллицы в ID нет. Проверка — шаблоны вида `XX-N` **вне кавычек**:
      бизнес-литералы в кавычках («ИТП-3», «[СОПРЮЛ]») идентификаторами не
      являются и под проверку не попадают.
- [ ] Иконки/ссылки/списки/токены `<Имя атрибута>` сохранены (reverse).

## Структура документа

- [ ] Гранулярность: документ описывает одну вкладку/страницу
      (templates.md, «Гранулярность»).
- [ ] Раздел «Описание ЭФ» присутствует, оформлен таблицей; строка
      «Дополнительные действия…» ссылается на EV-1.
- [ ] В реестре §2 — только поля: кнопок и сообщений среди строк нет.
- [ ] **«Тип» реестра — чистый UI-тип** из словаря type-dictionary (Текст /
      Список / Переключатель / Чекбокс / Заголовок / Иконка / …); способ
      заполнения («автоматически», «вручную», «выбор из списка») в колонку не
      примешан — он в FR-N.
- [ ] Обязательность в §2 делегирована контролю CTL (ссылка на группу), с
      оговоркой «проверяются только отображаемые поля».
- [ ] Модель SF-NN / R / V не используется; видимость — только §3 / FR-N.
- [ ] Кнопки описаны событиями (Видимость / Активность / При нажатии), не
      отдельным разделом «Действия».
- [ ] **Имена полей и названия событий-кнопок — без позиционных уточнений**
      («(у поля X)», «(в шапке)», «(слева внизу)»): расположение — в макетах.
      Поля-омонимы различены блоком (реестр и заголовок FR), не позицией.
- [ ] **Атрибуты FR — стандартные имена и канонический порядок**: Тип, Формат,
      Способ заполнения, Обязательность, Видимость, Логика установки значения
      (с под-ярлыками) — именно в этой последовательности; дополнительные
      атрибуты (из реестра templates.md §4) — только после стандартных;
      синонимов и разовых имён нет; разные атрибуты не слиты в одну строку.
- [ ] **Тип и Обязательность согласованы** между реестром §2 и FR-N каждого
      поля (авторитет — FR; расхождение — дефект).
- [ ] (reverse) Пометки прослеживаемости на месте: `*(исх. Шаг N)*` у каждого
      шага, `*(исх. Сообщение …)*` / `*(исходный номер отсутствует)*` у
      каждого сообщения.
- [ ] frontmatter: `id/type/status/service/version` (+ `source`/
      `confluence_page_id` и «История изменений» при reverse); имя файла — по
      конвенции комплекта либо, вне комплекта, вставка `EV-` **после сегмента
      роли `ЭФ-<Роль>`** (`[О2+NEW]-ЭФ-Клиента-EV-…`; нет сегмента роли —
      после скобочного префикса); для вкладки — суффикс
      `-вкладка-<название>` без номера.
